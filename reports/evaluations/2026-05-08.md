## 미해결 이슈

### 1. TTFT p99 오버헤드 (+286.9%) — Fail

- **원인**: baseline p99(6.66ms)는 일반 부하 시나리오이나, 선점형 스케줄러 도입 시
  KV 복원 경로를 거치는 요청이 p99를 끌어올림. 정상 부하에서도 비선점 요청 대비
  복원 오버헤드가 존재.
- **개선 방향**: (1) 선점 이전에 복원 비용을 예측해 복원 vs 재계산을 동적 선택하는
  비용 모델 도입; (2) p99 SLO를 정의한 요청에 대해 선점 면제 확장(SLA Tier-A 범위 확장);
  (3) eOptShrinkQCodec decode 지연을 줄이기 위한 비동기 프리패치 기법 적용.

### 2. 요청 처리 공정성 (burst p99 / normal p50 = 4.0×) — Fail

- **원인**: burst 시나리오에서 일부 요청이 최대 26ms 대기하나,
  normal 시나리오 p50(6.43ms)의 2배(12.86ms)를 초과함.
  fairness_max_wait=10 스텝이 절대 지연 시간(ms) 기준이 아닌 배치 스텝 수 기준이라
  burst 부하 시 실제 ms 단위 공정성을 보장하지 못함.
- **개선 방향**: (1) fairness_max_wait를 배치 스텝 수 대신 절대 시간(ms) 기준으로 전환;
  (2) burst 감지 시 선점 임계값(threshold_preempt)을 동적 조정해 burst 초기에 더 적극적
  선점으로 대기 시간 균일화; (3) group-first 정렬과의 연동 강화.

## 다음 사이클 제언

1. **TTFT p99 SLO 보장 메커니즘**: 복원-vs-재계산 비용 모델을 구현하고, 요청별 p99 SLO
   예산(deadline-aware scheduling)을 도입하면 공정성과 p99 동시 개선이 가능하다.
   EDF(Earliest Deadline First) 또는 예측 기반 선점 해제가 유효한 접근이다.

2. **선점 경로 압축 오버헤드 감소**: eOptShrinkQCodec의 decode를 비동기 프리패치
   (요청 재개 예정 시점 이전에 CPU→GPU 전송 시작)로 파이프라인화하면 복원 지연을
   정상 배치 실행과 오버랩할 수 있다.

3. **Activity B 강화**: StaticDynamicSegmentCache의 정적 세그먼트 자동 탐지(접근 빈도
   임계값 기반 자동 mark_static)를 추가하면 사용자 API 호출 없이 히트율이 추가 개선된다.

4. **메모리 절감과 처리량의 결합 효과 정량화**: 현재 51% 메모리 절감은 A+C 복합이나,
   활성 배치 크기(batch size) 증가를 통한 처리량 추가 이득을 별도 측정하여
   메모리-처리량 트레이드오프 곡선을 그리면 향후 사이클의 하이퍼파라미터 튜닝 근거가 된다.
