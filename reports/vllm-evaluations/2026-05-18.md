| 압축/해제 오버헤드 TTFT | N/A | 모델 환경 미측정 |
| CacheConfig compression_method 필드 | Pass | extend_cache_config_dp_attn_aware_compression() 정상 |

**Loop 2 수정 사항 — INT8 직접 저장 + per-row scale**:
- 이전(Loop 1): `(q.float() * scale).half()` 로 FP16 재패킹 → 실측 reduction = 0%.
- 현재(Loop 2): INT8 텐서 직접 저장 (`k_int8: torch.int8, v_int8: torch.int8`), scale을 `_store_scale()`에 사이드채널 저장.
- 메모리: FP16 32,768 bytes → INT8+scale 17,408 bytes = **46.9% 감소** (목표 ≥ 30% 충족).
- Per-row scale: `x_2d.abs().max(dim=-1, keepdim=True)[0] / 127.0` → shape `(n_rows, 1)`, outlier row로 인한 전체 정밀도 저하 방지.
- 정확도: per-row scale 적용 후에도 attention relative error 0.006 미만, cosine 0.9999 이상 유지.

**메모리 감소 계산 (실측)**:
```
원본 FP16:  (k.numel() + v.numel()) * 2 bytes
저장 INT8:  (k.numel() + v.numel()) * 1 byte   (INT8 = 1 byte/element)
scale 오버헤드: n_rows * 4 bytes (float32)
net reduction ≈ 1 - (n*d + n*4) / (n*d*2)
            ≈ 1 - (8192 + 512) / 16384 (k만, d=64)
            ≈ 46.9% (실측값과 일치)
```

### [참고] write_to_cache API — block_idx 파라미터 없음

평가 스모크 테스트에서 `write_to_cache(k, v, layer_idx=0, block_idx=0)` 호출 형태가 사용되었으나, 실제 구현 서명은 `write_to_cache(key, value, layer_idx=0)` 이다 (`block_idx` 미존재). 기능 평가는 올바른 서명으로 검증하였으며, 호출 측 코드가 있다면 `block_idx` 인자를 제거해야 한다. 현재 vllm_integration 내부 호출은 올바른 서명을 사용하므로 런타임 영향 없음.

## 다음 사이클 제언

1. **실 GPU 환경 처리량·TTFT 측정**: 현재 환경에서 모델 로딩이 불가하여 tokens/sec, TTFT p50/p99 실측이 이루어지지 않았다. 실 GPU 환경에서 SRFT+INT8 vs DPAttentionAware INT8 처리량 비교가 필요하다.
2. **vLLM 0.22+ 대비 API 안정성 검토**: `vllm.v1.core.kv_cache_manager.KVCacheManager` 인터페이스는 vLLM 마이너 업그레이드마다 변동 위험이 있다. `pip install --upgrade vllm` 후 호환성 재검증 권장.
3. **DPAttentionAwareVllmCodec.compression_hook() 일관성**: `attention_backend_patch.py`의 `DPAttentionAwareCompressionAttentionHook`는 INT8 직접 저장으로 수정되었으나, `compression_codec.py`의 `DPAttentionAwareVllmCodec.compression_hook()`는 별도 확인이 필요하다. 두 구현체 간 결과 일관성 검증을 권장한다.
