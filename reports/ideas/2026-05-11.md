## Activity A 아이디어

### 아이디어 A-1: WiCER-CEGAR 캐시 세그먼트 커버리지 정제 스케줄러 (CEGARCoverageRefineScheduler)
- **가설**: CEGAR(Counterexample-Guided Abstraction Refinement) 피드백 루프를 스케줄러에 적용하면, 캐시 세그먼트 커버리지 누락 지점을 자동으로 식별·보정하여 반복 배치마다 히트율이 단조 증가한다.
- **접근 방법**: (1) 배치 실행 후 히트 미스가 발생한 세그먼트를 "반례(counterexample)"로 수집한다. (2) 반례를 기반으로 해당 도메인 프리픽스를 "추상화 후보"로 표시하고 다음 배치에서 해당 세그먼트를 우선 워밍업 대상으로 삽입한다. (3) 히트율이 목표치(예: 80%)에 도달하면 루프를 종료하고 현재 캐시 상태를 "검증된 추상화"로 고정한다. 구현: `src/cache/` 위에 `CEGAR` 피드백 레이어 추가, 스케줄러가 미스 세그먼트 큐를 관리.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 노드 간 반례 집계 시 분산 카운터(AllReduce)를 사용해 글로벌 미스 세그먼트를 식별; 워밍업 삽입은 히트율이 낮은 노드에 우선 적용.
- **예상 효과**: 처리량 +10~15%, Scheduling overhead +2% 이내 (CEGAR 루프 수렴 후 오버헤드 소멸)
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 A-2: LatentRAG 잠재 적중 확률 사전 라우팅 (LatentHitPredictionRouter)
- **가설**: LatentRAG 방식으로 생성한 경량 잠재 서브쿼리 벡터를 요청 라우팅 단계에서 캐시 인덱스와 내적하면, 이산 토큰 프리픽스 비교 없이도 사전에 캐시 히트 가능성이 높은 노드/파티션으로 요청을 보낼 수 있다.
- **접근 방법**: (1) 각 요청 수신 시 소형 encoder(예: 2-layer MLP 또는 frozen dense retriever)로 잠재 서브쿼리 벡터 z ∈ R^d를 생성한다. (2) 캐시 인덱스에 저장된 각 파티션의 대표 벡터와 코사인 유사도를 계산해 top-k 파티션을 후보로 선정한다. (3) 후보 파티션의 현재 부하와 실제 토큰 히트율을 가중합산하여 최종 라우팅 결정을 내린다. 구현: `src/cache/segmented.py` + 새로운 `src/engine/latent_router.py`.
- **대상 환경**: 멀티 노드
- **멀티 노드 고려사항**: 잠재 벡터 전송 비용은 d=256 기준 1KB 미만으로 네트워크 오버헤드 무시 가능; 파티션 대표 벡터 테이블은 공유 메모리(RDMA) 또는 중앙 라우팅 서버에 관리.
- **예상 효과**: 처리량 +8~12%, Scheduling overhead +1.5% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: CEGAR 반복 Wiki KV 컴파일 캐시 (WiCERIterativeKVWikiCache)
- **가설**: CEGAR 패턴을 KV 캐시 구축에 적용하면, 도메인 지식을 KV 아티팩트로 컴파일→히트율 평가→미커버 갭 정제의 반복 루프를 통해 커버리지가 보장된 영구 KV 캐시를 구축할 수 있다.
- **접근 방법**: (1) **컴파일 단계**: 도메인 문서 코퍼스를 고정 청크로 분할하고 각 청크의 KV 행렬을 사전 계산하여 `SegmentedKVStore`에 저장한다. (2) **평가 단계**: 검증 쿼리 셋을 실행하고 히트되지 않은 청크(반례)를 수집한다. (3) **정제 단계**: 반례 청크를 더 세밀한 서브청크로 분할하거나 인접 청크와 병합하여 재컴파일한다. (4) 히트율 목표 달성 시 KV 아티팩트를 디스크에 직렬화하여 재사용한다. `src/cache/segmented.py`의 `SegmentedKVStore`를 기반으로 CEGAR 루프 관리자 클래스 추가.
- **예상 효과**: 비연속 히트율 +20~30% (Wiki/FAQ 도메인 기준), TTFT −40% (반복 재계산 제거)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-2: 잠재 공간 세그먼트 검색 캐시 (LatentSpaceSegmentSearchCache)
- **가설**: 이산 토큰 해시 대신 연속 잠재 공간에서 세그먼트를 검색하면, 표면 형태가 다른 의미적 동등 세그먼트도 히트시킬 수 있어 비연속 재사용률이 크게 향상된다.
- **접근 방법**: (1) 각 캐시 세그먼트를 저장할 때 해시 키 외에 frozen dense encoder로 생성한 잠재 벡터 e ∈ R^256도 함께 인덱싱한다(FAISS HNSW 인덱스). (2) 쿼리 시 정확 해시 히트 → 잠재 근사 히트 순서로 2단계 검색을 수행한다. 잠재 히트 시 코사인 유사도 > θ(예: 0.92)인 세그먼트의 KV를 재사용하고, 위치 보정을 적용한다. (3) 잠재 히트 세그먼트는 경량 어댑터(2-layer MLP, frozen) 없이 직접 삽입하거나 선택적으로 선형 보간(α·KV_cached + (1-α)·KV_recompute)한다. `src/cache/segmented.py`에 FAISS 인덱스 레이어 추가.
- **예상 효과**: 비연속 히트율 +15~25% (동의어/paraphrase 쿼리), TTFT −25%
- **구현 난이도**: medium
- **우선순위**: 2

## Activity C 아이디어

### 아이디어 C-1: 출력 인식 레이어별 KV 행렬곱 근사 퇴거 (LaProxOutputAwareLayerEviction)
- **가설**: KV 퇴거를 단순 어텐션 점수 기반이 아닌 출력 행렬곱의 근사 오차로 직접 측정하면, 레이어별 실제 출력에 미치는 영향이 작은 토큰을 더 정확히 식별해 5% KV 유지만으로 정확도를 보존할 수 있다.
- **접근 방법**: (1) 각 레이어 l에서 어텐션 출력 O_l = softmax(QK^T/√d)·V를 계산 후, 각 토큰 i의 기여를 제거했을 때의 출력 변화량 ΔO_l^i = ||O_l - O_l^{-i}||_F를 추정한다(전체 재계산 대신 1차 테일러 근사 사용). (2) 모든 헤드의 ΔO_l^i를 합산하고(헤드 간 의존성 포함), 레이어 전체 기준으로 상위 k% 중요도 토큰만 유지한다. (3) 레이어마다 독립적으로 k를 조정하는 레이어별 예산 스케줄을 사전 프로파일링으로 결정한다. `src/cache/segmented.py`에 `LaProxEvictionPolicy` 클래스 추가.
- **예상 효과**: 메모리 −85~90%, accuracy delta ±0.8%
- **accuracy-preserving 근거**: 출력 행렬곱 근사 오차를 직접 최소화하므로 어텐션 점수 기반 퇴거 대비 실제 출력 왜곡이 최소화된다. 1차 테일러 근사의 잔차(residual)는 O(||ΔKV||^2)로 충분히 작으며, 레이어별 독립 예산 조정은 하위 레이어 과도 압축을 방지해 perplexity 변화를 ±0.8% 이내로 유지한다.
- **구현 난이도**: high
- **우선순위**: 1

### 아이디어 C-2: E2E 차분 가능 학습 버짓+선택 코덱 (LKVEndToEndBudgetSelectorCodec)
- **가설**: 버짓 할당과 토큰 선택을 별도로 최적화하지 않고 E2E 차분 가능 프레임워크로 공동 학습하면, 두 결정이 서로를 보강하여 15% KV 유지 수준에서 near-lossless 압축이 달성된다.
- **접근 방법**: (1) **LKV-H (글로벌 버짓 할당)**: 모든 헤드의 KV를 플래튼하여 글로벌 버짓 예산을 놓고 소프트 경쟁(softmax gating)으로 헤드별 버짓을 학습한다. Straight-Through Estimator로 이산 버짓 결정을 역전파 가능하게 만든다. (2) **LKV-T (암묵적 토큰 중요도 도출)**: 어텐션 행렬을 명시적으로 구성하지 않고 Q·K 내적 스코어의 학습된 선형 변환으로 토큰 중요도를 유도하는 경량 selector 네트워크를 추가한다. (3) 두 모듈을 end-to-end로 학습: 재구성 손실 = 압축 KV로 계산한 출력과 풀 KV 출력의 MSE + 버짓 패널티. `src/cache/base.py`의 `CacheStore`를 상속하는 `LKVCodec` 클래스로 구현.
- **예상 효과**: 메모리 −85%, accuracy delta ±0.5%
- **accuracy-preserving 근거**: E2E 학습이 재구성 MSE를 직접 최소화하는 목적함수로 훈련되므로, 임의 휴리스틱 기반 퇴거 대비 출력 왜곡이 보장 수준에서 최소화된다. LKV 원논문에서 15% 유지 시 perplexity 변화 <0.5%가 실증되었으며, 본 구현도 동일한 목적함수를 사용하므로 동등한 수준의 accuracy 보존이 기대된다.
- **구현 난이도**: high
- **우선순위**: 2

### 아이디어 C-3: 역 물 채우기 율-왜곡 최적 헤드별 비트 할당 (RateQuantReverseWaterfillingCodec)
- **가설**: 정보 이론의 역 물 채우기 알고리즘으로 각 어텐션 헤드의 왜곡-비트 트레이드오프를 최적화하면, 동일 총 비트 예산에서 단순 균일 양자화 대비 perplexity가 현저히 낮아진다.
- **접근 방법**: (1) **캘리브레이션**: 소규모 캘리브레이션 셋(예: 512 샘플)에서 각 헤드 h의 KV 채널별 분산 σ²_{h,c}를 측정한다. (2) **역 물 채우기**: 총 비트 예산 R이 주어질 때, 각 헤드 h에 할당할 비트 수 r_h를 역 물 채우기 공식 r_h = max(0, (1/2)·log2(σ²_{h} / λ))으로 계산한다(λ는 전체 예산을 맞추는 라그랑지 승수). (3) **양자화 적용**: 산출된 r_h로 기존 양자화기(KIVI, QuaRot 등) 위에 최적 비트폭 레이어를 씌운다. 총 2~8비트 범위에서 헤드별 가변 비트폭 적용. `src/cache/` 에 `RateQuantCodec` 클래스로 구현, 기존 quantization 백엔드(INT4, FP8)와 플러그인 형태로 연동.
- **예상 효과**: 메모리 −50~60% (2.5비트 평균), accuracy delta ±0.3% (균일 4비트 대비 perplexity −70% 개선)
- **accuracy-preserving 근거**: 역 물 채우기는 주어진 비트 예산 하에서 총 재구성 왜곡의 이론적 하한을 달성하는 최적 해다. 헤드별 분산에 비례한 비트 할당이 정보 손실을 최소화하며, 원논문(RateQuant)에서 Qwen3-8B 2.5비트 시 perplexity가 49.3→14.9로 낮아진 결과가 이 메커니즘의 유효성을 직접 증명한다.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-4: SSM 기반 퇴거 KV 연속 메모리 진화 코덱 (RetentiveKVSSMLatentRetentionCodec)
- **가설**: 퇴거된 KV 토큰을 단순히 버리는 대신 SSM(State Space Model) 상태 전이로 압축 상태에 연속적으로 누적하면, 5× 압축에서도 오래된 컨텍스트 정보가 소실 없이 유지된다.
- **접근 방법**: (1) 각 레이어마다 SSM 히든 상태 h_t ∈ R^d_ssm을 유지한다. (2) 퇴거 대상 토큰 i의 KV를 버리기 전에 엔트로피 기반 중요도 w_i = −Σp·logp (p는 해당 토큰에 대한 어텐션 분포)를 계산한다. (3) SSM 상태 업데이트: h_{t+1} = A·h_t + w_i·B·KV_i (A, B는 학습 파라미터). (4) 디코딩 시 슬라이딩 윈도우 KV + SSM 상태를 혼합 어텐션으로 쿼리한다. 텍스트 전용 환경에 맞게 LightSSM(Mamba-2 경량화 버전) 기반으로 구현. `src/cache/`에 `RetentiveKVCodec` 클래스.
- **예상 효과**: 메모리 −80%, accuracy delta ±0.9%, 디코딩 속도 +50%
- **accuracy-preserving 근거**: 퇴거 토큰을 완전히 제거하지 않고 SSM 상태에 가중합산하여 보존하므로, 정보 손실이 "어텐션 근사 오차"로 전환된다. 엔트로피 가중치 w_i가 중요한 토큰은 SSM 상태에 더 강하게 반영되도록 보장하며, SSM의 선형 재귀 특성상 상태 용량 내에서 이론적으로 무손실 정보 누적이 가능하다. RetentiveKV 원논문에서 5× 압축 시 perplexity 변화 <1%가 보고되었다.
- **구현 난이도**: high
- **우선순위**: 2

### 아이디어 C-5: 프롬프트 유도 크로스 어텐션 KV 압축기 (PromptGuidedCrossAttentionKVCompressor)
- **가설**: LightKV의 텍스트-비주얼 크로스 모달 메시지 패싱 아이디어를 텍스트 전용 LLM에 적용하면, 쿼리 컨텍스트 토큰(프롬프트)이 KV 캐시 토큰 중 어떤 것을 보존해야 하는지 직접 "안내"하는 프롬프트 의존적 압축이 가능하다.
- **접근 방법**: (1) 프롬프트 토큰(Q-side)의 은닉 상태를 "가이드 신호"로 사용한다. (2) KV 캐시 내 각 토큰의 압축 마스크 m_i ∈ {0,1}을 프롬프트 가이드와 KV 토큰 간 크로스 어텐션 스코어 s_i = softmax(Q_prompt · K_i^T / √d)의 합산으로 결정한다: m_i = 1 if Σ_q s_{q,i} > τ. (3) m_i=0인 토큰만 저정밀도(INT2)로 압축하고, m_i=1인 중요 토큰은 FP8로 유지한다. (4) τ는 메모리 목표에 따라 동적으로 조정. `src/cache/segmented.py`에 `PromptGuidedCompressor` 추가.
- **예상 효과**: 메모리 −50%, accuracy delta ±0.7%
- **accuracy-preserving 근거**: 프롬프트 크로스 어텐션 스코어는 실제 모델이 어떤 KV 토큰에 주목하는지를 직접 반영하므로, 압축 결정이 모델 내부 우선순위와 일치한다. 프롬프트 의존적 마스킹이 정적 퇴거 대비 중요 토큰 오압축 확률을 낮추며, 중요 토큰은 FP8로 유지되므로 최소 정밀도가 보장된다.
- **구현 난이도**: medium
- **우선순위**: 3

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: WiCER Wiki KV + RateQuant 최적 비트 압축 파이프라인 (WiCER-RateQuantPipeline) (B+C)
- **조합 근거**: WiCERIterativeKVWikiCache(B-1)로 구축한 도메인 KV 아티팩트는 크기가 크고 장기 보존이 필요하다. 여기에 RateQuantReverseWaterfillingCodec(C-3)을 적용하면 헤드별 최적 비트 할당으로 아티팩트 크기를 최소화하면서 정확도를 보존할 수 있다. CEGAR 루프의 "정제" 단계에서 재컴파일된 청크에도 즉시 최적 양자화를 적용해 증분 메모리 증가를 억제한다.
- **예상 복합 효과**: 처리량 +35% (도메인 TTFT −40% × 처리량 향상), 메모리 −55%, 정확도 ±0.5%
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: 잠재 세그먼트 검색 + 출력 인식 퇴거 통합 (LatentSearch-LaProxFusion) (B+C)
- **조합 근거**: LatentSpaceSegmentSearchCache(B-2)로 의미적으로 유사한 세그먼트를 히트시킨 후, 재사용된 KV 세그먼트에서 LaProxOutputAwareLayerEviction(C-1)을 즉시 적용하면 재사용 KV의 메모리 점유를 최소화할 수 있다. 잠재 히트 세그먼트는 출력 행렬곱 근사 오차가 이미 낮은 토큰이 많으므로 LaProx 퇴거와 시너지가 발생한다.
- **예상 복합 효과**: 처리량 +25%, 메모리 −75%, 정확도 ±0.9%
- **구현 난이도**: high
- **우선순위**: 2

### 아이디어 Cross-3: CEGAR 스케줄러 + 동적 비트폭 감소 연동 (CEGARScheduler-RateQuantDynamic) (A+C)
- **조합 근거**: CEGARCoverageRefineScheduler(A-1)가 히트율이 높아지면 단위 시간당 캐시 재계산 빈도가 줄어 GPU 메모리 여유가 생긴다. 이 여유를 활용해 RateQuantReverseWaterfillingCodec(C-3)의 총 비트 예산을 동적으로 높이면, 히트율 향상 → 압축 품질 향상의 선순환이 발생한다. 반대로 메모리 압박 시 CEGAR가 비트 예산을 자동으로 낮추는 피드백 루프를 구성한다.
- **예상 복합 효과**: 처리량 +20%, 메모리 −45~65% (동적 범위), 정확도 ±0.4%
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 Cross-4: CEGAR Wiki KV + SSM 연속 메모리 + 잠재 라우팅 통합 프레임워크 (WiCER-RetentiveKV-LatentRouter) (A+B+C)
- **조합 근거**: (A) LatentHitPredictionRouter(A-2)로 요청을 최적 노드에 라우팅하고, (B) WiCERIterativeKVWikiCache(B-1)로 도메인 KV를 영구 아티팩트로 관리하며, (C) RetentiveKVSSMLatentRetentionCodec(C-4)으로 슬라이딩 윈도우 밖 KV를 SSM 상태에 보존한다. 세 기법은 각각 라우팅 정확도, 캐시 커버리지, 메모리 효율을 독립적으로 개선하면서 상호 의존 없이 스택 가능하다.
- **예상 복합 효과**: 처리량 +45%, 메모리 −80%, 정확도 ±1.0%
- **구현 난이도**: high
- **우선순위**: 2

## planner에게 전달할 요청사항
- **최우선 구현 C-3 (RateQuantReverseWaterfillingCodec)**: 캘리브레이션 단계(512 샘플)와 역 물 채우기 수식을 `src/cache/`에 독립 클래스로 구현. 기존 INT4/FP8 양자화 백엔드와 플러그인 인터페이스로 연동. accuracy-preserving 검증 방법: 캘리브레이션과 독립된 128 샘플로 perplexity 측정, 베이스라인 대비 ±1% 이내 확인 필수.
- **B-1 (WiCERIterativeKVWikiCache)**: CEGAR 루프의 종료 조건(히트율 목표 및 최대 반복 횟수)을 설정 파일(`configs/experiments/`)에서 제어 가능하게 설계.
- **Cross-1 파이프라인**: WiCER 아티팩트 저장 시 RateQuant 비트 메타데이터(헤드별 r_h)를 함께 직렬화하여 로드 시 재양자화 오버헤드 없이 즉시 사용 가능하게 구현.
- **이번 사이클 측정 지표**: KV Cache Memory Reduction (목표 −55%), Inference Throughput (목표 +35%), Compression Accuracy Delta (필수: ±1% 이내), Non-Contiguous Cache Hit Rate (목표 +20%).
- **Activity C 구현 시 공통 필수 사항**: 모든 C 아이디어는 perplexity 측정 코드를 `src/metrics/`에 포함하고, 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 함께 기록해야 한다.
