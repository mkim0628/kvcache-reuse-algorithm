## 변화 요약
- **Activity A PPD 패러다임 전환**: PPD(arXiv 2603.13358)의 "D 노드 로컬 append-prefill + 동적 P/D 선택"과 PrfaaS(arXiv 2604.15039)의 "이더넷 크로스-데이터센터 KV 전송 + 길이 기반 임계값 라우팅"이 이전 사이클의 모든 A 기법이 다루지 않은 두 새로운 설계 차원을 열었다. 특히 PPDAppendPrefillRouter는 비연속 세그먼트 히트 예측과 결합해 A+B 시너지의 가장 직접적인 구현 경로를 제공한다.
- **Activity B 로그 시간 검색 인덱스 등장**: LycheeCluster(arXiv 2603.08453)의 삼각부등식 기반 계층 인덱스가 이전 사이클까지 모든 B 기법의 미해결 병목이었던 O(N) 세그먼트 검색을 O(log N)으로 해결하는 새로운 자료구조를 제공한다. SemantiCache(arXiv 2603.14303)의 의미 경계 기반 청킹이 청크 내 의미 완결성을 보장하는 새로운 세그먼트 구성 패러다임을 제시한다.
- **Activity C 훈련 통합 압축 및 압축-디코딩 결합 최적화 신규 등장**: Stochastic KV Routing(arXiv 2604.22782)의 학습 중 내재화된 depth-wise KV 공유와 SpecKV(arXiv 2605.02888)의 압축-γ 결합 최적화가 이전 사이클의 모든 C 기법이 다루지 않은 두 새로운 압축 차원을 동시에 제시한다.

## 이번 사이클 초점
**A+B (PPD append-prefill 라우팅 + 로그 시간 계층 세그먼트 인덱스)** — 보조로 **C** StochasticDepthwiseKVShareCodec, SpecKVCompressionGammaController, **B+C** SemanticBoundarySegmentCache + GSC 클러스터 병합 압축

**선택 이유**:
1. 이전 두 사이클(2026-05-06, 2026-05-08)이 각각 B+C와 A+C에 집중했으므로, 이번 사이클에서는 A+B 통합에 초점을 맞춘다. PPDAppendPrefillRouter(A)와 TriangleInequalitySegmentIndex(B)의 조합이 "D 노드에서 비연속 세그먼트를 로그 시간에 검색하여 append-prefill 처리"라는 A+B 시너지의 가장 직접적인 구현 경로를 제공한다.
2. LycheeCluster의 O(log N) 검색이 이전 사이클까지의 모든 B 기법의 병목(O(N) 검색)을 근본적으로 해소하며, 이는 비연속 히트율 ≥30% 목표 달성에 가장 직접적으로 기여한다.
3. Activity C의 Stochastic KV Routing은 훈련 가능한 환경에서만 적용 가능하나, SpecKV의 압축-γ 결합 최적화는 기존 eOptShrinkQCodec(05-08 기구현)과 즉시 통합 가능한 실용적 경로를 제공한다.
4. Text2JSON 벤치마크(arXiv 2604.08426)가 Activity C 평가에 필수적으로 추가되어야 함을 강하게 시사한다 — "정확도 ±1% 이내" 제약의 실질적 검증 강화.

## Activity A 아이디어

### 아이디어 A-1: PPDAppendPrefillRouter — D 노드 append-prefill + 세그먼트 히트 예측 기반 P/D 동적 선택 라우터
- **가설**: 멀티턴 LLM 서빙에서 매 턴마다 이전 응답 KV를 P 노드로 재전송 후 재프리필하는 기존 P/D 분리 구조의 두 가지 비효율(Turn 2+ KV 재전송 병목, P→D KV 전송 대역폭 포화)을, D 노드에서 직접 append-prefill(새 입력 토큰만 처리 + 캐시된 KV 재사용)을 처리하는 동적 라우터를 설계하고 이전 사이클의 Activity B 세그먼트 히트 예측과 통합하면(PPD 원리 + LycheeCluster 로그 시간 인덱스), 이전 사이클의 모든 A 기법(PreemptiveKVOffloadScheduler, CrossRegionSegmentRouter, WAITThresholdBatcher 포함)이 P/D 노드 할당 자체를 요청 유형별로 동적 결정하지 않은 것과 달리 Turn 2+ TTFT 68% 단축이 가능하다.
- **접근 방법**:
  1. `PPDAppendPrefillRouter` 클래스 (`src/scheduler/ppd_append_prefill_router.py`): `BaseScheduler` 상속
  2. 요청 분류: Turn 1 (신규 세션) → P 노드 full prefill, Turn 2+ (멀티턴 계속) → D 노드 append-prefill 후보로 분류
  3. 세그먼트 히트 예측 통합: `TriangleInequalitySegmentIndex`(B-1)를 이용해 D 노드의 캐시에서 Turn 2+ 요청의 이전 KV 세그먼트 히트 가능성을 O(log N)으로 빠르게 추정
     - 히트 예상(히트 확률 > `threshold_append`, 기본 0.7) → D 노드 append-prefill 선택
     - 미스 예상 → P 노드 full prefill로 폴백
  4. append-prefill 실행 경로: D 노드가 캐시된 KV에 새 입력 토큰만 어텐션 → 재전송 없이 Turn 2+ TTFT 단축
  5. SLO 기반 가중치 조정: TTFT SLO 임박 시 append-prefill 임계값 하향 조정 (더 공격적으로 D 노드 처리)
  6. 이전 A 기법과의 차별점: PreemptiveKVOffloadScheduler가 선점 타이밍과 KV 오프로딩 결정에 집중하고, CrossRegionSegmentRouter가 지리적 라우팅 비용 공동 최적화에 집중한 반면, PPDAppendPrefillRouter는 "P/D 역할 자체를 요청 유형과 세그먼트 히트율에 따라 동적으로 결정"하는 완전히 새로운 라우팅 계층을 도입한다.
  7. WAITThresholdBatcher(05-08 기구현)와의 결합: WAIT 배치 임계값이 append-prefill 요청과 full prefill 요청을 구분해 배치 구성 최적화
- **대상 환경**: 멀티 노드 (Disaggregated Prefill P/D 분리 아키텍처)
- **멀티 노드 고려사항**: P 노드와 D 노드 간 KV 전송이 Turn 2+에서 제거되므로 P→D 대역폭 포화 문제 근본 해소. D 노드 KV 캐시 용량이 append-prefill 처리의 핵심 병목 — D 노드 메모리 예산 관리 우선. 이더넷 환경(PrfaaS, arXiv 2604.15039)에서도 P→D 전송이 줄어 크로스-데이터센터 적용 가능성 확장.
- **예상 효과**: Turn 2+ TTFT −68% (PPD arXiv 2603.13358 실증), KV 전송 대역폭 −50~70% (append-prefill로 재전송 제거), 처리량 +25~35% (TPOT 경쟁력 유지)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: EthernetKVTransferScheduler — 이더넷 기반 크로스-데이터센터 KV 전송 + 길이 임계값 라우팅
- **가설**: 기존 P/D 분리 설계가 저지연 RDMA 패브릭을 전제로 단일 데이터센터에 묶여 있는 제약을, 하이브리드 어텐션 모델(MLA 등)의 낮은 KV 크기를 활용해 장문 프리필만 전용 프리필 클러스터에 위탁하고 결과 KV를 일반 이더넷으로 전송하는 크로스-데이터센터 P/D 확장을 실현하면(PrfaaS 원리), 이전 사이클의 CrossRegionSegmentRouter(05-08)가 RDMA/InfiniBand 환경에서 세그먼트 히트율과 지연을 공동 최적화한 것과 달리 "RDMA 불필요한 이더넷 환경에서 길이 기반 선택적 외부 위탁"이라는 새로운 설계 공간을 탐구해 처리량 54% 향상이 가능하다.
- **접근 방법**:
  1. `EthernetKVTransferScheduler` 클래스 (`src/scheduler/ethernet_kv_transfer.py`): `BaseScheduler` 상속
  2. 길이 임계값 라우팅: `prompt_length > threshold_cross_dc` (기본 8K 토큰)이면 전용 프리필 클러스터로 외부 위탁; 이하이면 로컬 PD 처리
  3. KV 크기 감소 사전 조건 확인: MLA/하이브리드 어텐션 모델에서 KV 크기가 충분히 작은지 확인 (`kv_size_per_token < threshold_transfer`, 기본 2KB/토큰). 조건 불만족 시 로컬 PD로 폴백.
  4. 대역폭 인식 동적 임계값: 이더넷 링크 상태(현재 대역폭 사용률)를 실시간 모니터링 → 대역폭 여유 시 `threshold_cross_dc` 하향(더 적극적 외부 위탁), 포화 시 상향
  5. 하이브리드 프리픽스 캐시 풀: 로컬 PD 프리픽스 캐시 + 원격 프리필 클러스터 캐시의 두 계층으로 프리픽스 히트 관리
  6. Activity C(eOptShrinkQCodec, 05-08 기구현) 통합: KV 전송 전 압축 적용 → 이더넷 전송 KV 크기 추가 감소 (PrfaaS + eOptShrinkQ 이중 효과)
  7. 이전 A 기법과의 차별점: CrossRegionSegmentRouter(05-08)가 지리적 다중 리전에서 세그먼트 히트 가치 + 네트워크 지연 + 큐 상태 3차원 공동 최적화를 수행한 반면, EthernetKVTransferScheduler는 단순 길이 기반 임계값 라우팅으로 RDMA 불필요한 이더넷 환경에서의 실현 가능성을 우선 검증한다. 환경 가정(RDMA vs 이더넷)과 라우팅 전략(3차원 공동 최적화 vs 길이 임계값)이 모두 다르다.
- **대상 환경**: 멀티 노드 (크로스-데이터센터 / 이더넷 연결 클러스터)
- **멀티 노드 고려사항**: 이더넷 환경에서 KV 전송 지연이 RDMA 대비 높으므로 길이 임계값 설정이 핵심 변수. eOptShrinkQCodec(05-08 기구현)으로 전송 KV를 ~14% 크기로 줄이면 이더넷 환경에서도 실용적 지연 달성 가능. 네트워크 파티션 또는 고지연 상황에서의 로컬 폴백 정책 필수.
- **예상 효과**: 처리량 +54% (PrfaaS arXiv 2604.15039 실증, 1T 하이브리드 모델 기준), P90 TTFT −64% (장문 프리필 외부 위탁), KV 전송 크기 −86% (eOptShrinkQ 2.2비트 결합 시)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-3: NVMeDirectKVOffloadScheduler — NVMe-Direct 이중 경로 KV 오프로딩 스케줄러 (에지 특화)
- **가설**: 에지 기기에서 KV 캐시 오프로딩 시 기존 파일 기반 경로가 커널 페이지 캐시 스래싱과 높은 소프트웨어 오버헤드를 유발하는 문제를, 런타임 메모리 가용성에 따라 KV 텐서를 (1) 페이지-캐시 경로 또는 (2) NVMe-Direct 경로(파일시스템 우회, 연속 LBA 직접 매핑)에 동적 할당하는 이중 경로 스케줄러를 적용하면(Dual-Blade 원리), 이전 사이클의 ContiguousChunkIOScheduler(05-06)가 I/O 블록 정렬과 레이어 간 비동기 프리패칭에 집중한 것과 달리 "파일시스템 오버헤드 자체를 제거하는 NVMe 직접 접근"이라는 새로운 시스템 계층 최적화로 프리필 지연 33% 이상 단축이 가능하다.
- **접근 방법**:
  1. `NVMeDirectKVOffloadScheduler` 클래스 (`src/scheduler/nvme_direct_kv_offload.py`): `BaseScheduler` 상속
  2. 이중 경로 동적 할당: `available_host_memory / total_kv_size` 비율 기반 경로 선택
     - 비율 > `threshold_page_cache`(기본 0.5): 페이지-캐시 경로(표준 파일 I/O)
     - 비율 ≤ 0.5: NVMe-Direct 경로(KV 텐서를 연속 LBA 영역에 직접 매핑)
  3. KV 텐서 LBA 매핑: `map_kv_to_lba(kv_tensor, segment_id)` → NVMe 장치의 연속 LBA 블록 할당. `segment_id`를 LBA 오프셋 인덱스로 관리.
  4. 적응형 파이프라인 병렬성: NVMe I/O와 GPU DMA를 오버랩하는 비동기 파이프라인. GPU가 레이어 L 처리 중 NVMe에서 레이어 L+1 KV 프리패치.
  5. ContiguousChunkIOScheduler(05-06 기구현)와의 결합: ContiguousChunk 단위 I/O 정렬 원칙을 NVMe-Direct LBA 매핑에 적용 → 정렬된 연속 LBA 접근으로 NVMe 최대 처리량 달성
  6. 이전 A 기법과의 차별점: ContiguousChunkIOScheduler(05-06)가 청크 경계-I/O 블록 경계 정렬 + 레이어 간 비동기 프리패칭이라는 알고리즘 수준 최적화를 수행한 반면, NVMeDirectKVOffloadScheduler는 파일시스템 계층 자체를 우회하는 NVMe-Direct 경로라는 더 낮은 시스템 계층 최적화를 통해 추가 지연 감소를 달성한다.
- **대상 환경**: 단일 노드 (에지 기기 / 소비자급 GPU)
- **멀티 노드 고려사항**: 에지 특화 설계로 단일 노드가 주 환경. 클라우드 GPU 서버에서는 NVMe-Direct 경로가 CPU DRAM 오프로딩(PreemptiveKVOffloadScheduler, 05-08)보다 느릴 수 있으므로 환경 감지 후 경로 자동 선택.
- **예상 효과**: 프리필 지연 −33.1% (Dual-Blade arXiv 2604.26557 실증), 디코드 지연 −42.4%, SSD 이용률 +2.2×, 에지 기기 KV 캐시 유효 용량 +3~5× (NVMe 활용)
- **구현 난이도**: medium
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: TriangleInequalitySegmentIndex — 삼각부등식 기반 재귀 계층 KV 세그먼트 인덱스 (O(log N) 검색)
- **가설**: 비연속 KV 세그먼트 검색이 현재 O(N) 선형 스캔(SemanticSegmentCache/DHD 포함)에 의존해 세그먼트 수 증가 시 검색이 병목이 되는 문제를, 삼각부등식(`d(A,C) ≤ d(A,B) + d(B,C)`)을 이용해 현재 쿼리와 거리가 먼 서브트리를 단번에 가지치기하는 재귀 계층 인덱스를 구성하면(LycheeCluster 원리), 이전 사이클의 모든 Activity B 기법(SemanticSegmentCache/DHD, StaticDynamicSegmentCache, DiffAwareSegmentStore 등)이 O(N) 검색에 머문 것과 달리 이론적으로 한정된 O(log N) 가지치기 탐색으로 검색 효율을 3.6× 이상 향상시킬 수 있다.
- **접근 방법**:
  1. `TriangleInequalitySegmentIndex` 클래스 (`src/cache/triangle_index.py`): `CacheStore` 인터페이스 상속 및 인덱스 레이어로 기존 캐시 위에 추가
  2. 계층 인덱스 구성: 세그먼트 임베딩(평균 Key 벡터)을 클러스터 중심으로 재귀 분할
     - 루트: 전체 세그먼트 집합의 중심 벡터
     - 각 레벨에서 가장 먼 두 세그먼트를 pivot으로 선택 → 트리 분기
     - 각 노드에 "최대 반지름(max_radius)" 저장: `max_radius(node) = max_{s in subtree} d(center, s)`
  3. 삼각부등식 가지치기: 쿼리 q와 노드 중심 c의 거리 d(q, c)를 계산
     - `d(q, c) - max_radius(node) > current_best_distance` → 해당 서브트리 전체 가지치기 (삼각부등식 보장)
  4. 검색 알고리즘: 우선순위 큐 기반 best-first 탐색. 가지치기로 탐색 노드 수를 O(N) → O(log N) 수준으로 감소.
  5. `StaticDynamicSegmentCache`(05-08 기구현)와의 통합: 정적 세그먼트를 계층 인덱스에 등록, 동적 세그먼트는 인덱스 업데이트 빈도 조정 (동적 세그먼트 변경 시 로컬 재균형)
  6. 경계-인식 청킹 통합: 세그먼트 경계를 의미 경계(SemanticBoundarySegmentCache, B-2)와 일치시키면 임베딩 품질 향상 → 삼각부등식 가지치기 효율 증가
  7. 이전 B 기법과의 차별점: SemanticSegmentCache/DHD가 brute-force 코사인 유사도 검색(O(N))을 수행하고 StaticDynamicSegmentCache가 검색 효율보다 무효화 최소화에 집중한 반면, TriangleInequalitySegmentIndex는 O(N) 검색 자체를 O(log N)으로 줄이는 새로운 인덱스 자료구조를 도입한다.
- **예상 효과**: 비연속 히트율 +20~30%p (로그 시간 검색으로 대규모 캐시에서도 검색 가능), TTFT −40~60% (검색 오버헤드 제거), 검색 속도 3.6× 향상 (LycheeCluster arXiv 2603.08453 Quest/ClusterKV 대비 실증)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 B-2: SemanticBoundarySegmentCache — 의미 경계 기반 청킹 + GSC 클러스터 병합 + 비례 어텐션 재조정
- **가설**: 기존 KV 세그먼트 분할이 고정 토큰 수(예: 64 토큰)로 청킹해 의미 단편화가 발생하는 문제를, 자연적 의미 경계(구분자: 문장 끝, 절 분리 등)로 세그먼트를 분할하고 세그먼트 내 토큰을 시드-기반 탐욕 클러스터링(Greedy Seed-Based Clustering)으로 의미 코어에 병합하며 비례 어텐션(Proportional Attention)으로 병합된 토큰의 감소된 어텐션 기여를 재조정하면(SemantiCache 원리), 이전 사이클의 모든 Activity B 기법이 고정 크기 청킹에 머문 것과 달리 세그먼트 내 의미 완결성이 보장되어 세그먼트 재사용 정확도가 향상된다.
- **접근 방법**:
  1. `SemanticBoundarySegmentCache` 클래스 (`src/cache/semantic_boundary_cache.py`): `CacheStore` 인터페이스 상속
  2. 의미 경계 탐지: 입력 텍스트에서 문장 종결자(`.`, `!`, `?`), 단락 구분(`\n\n`), 코드 블록 경계(` ``` `) 등을 의미 경계로 탐지 → 세그먼트 청크 경계로 사용
  3. GSC 클러스터링: 각 세그먼트 내에서 시드 토큰(어텐션 스코어 상위 토큰)을 중심으로 유사 토큰을 탐욕적으로 클러스터링 → 클러스터를 의미 코어 하나로 병합
     - `merge_cluster(cluster_tokens) → core_kv`: 클러스터 내 Key/Value의 가중 평균 (가중치 = 어텐션 스코어)
  4. 비례 어텐션 재조정: 병합된 의미 코어의 어텐션 기여를 원래 클러스터 내 토큰 수에 비례하여 스케일링
     - `attention_weight_core = Σ_{t in cluster} attention_weight(t)` (합산, 평균 아님)
  5. 세그먼트 저장: 의미 코어 KV만 캐시 저장 → 원본 KV 대비 세그먼트 크기 대폭 감소 (병합 비율에 따라 −30~60%)
  6. `TriangleInequalitySegmentIndex`(B-1)와의 통합: 의미 코어 임베딩으로 계층 인덱스 구성 → 의미 경계 정렬 청크의 고품질 임베딩으로 삼각부등식 가지치기 효율 향상
  7. 이전 B 기법과의 차별점: SemanticSegmentCache/DHD가 세그먼트 간 유사도로 히트 결정하고, StaticDynamicSegmentCache가 갱신 패턴으로 세그먼트를 분류하는 반면, SemanticBoundarySegmentCache는 세그먼트 내부 구조 자체를 의미 코어로 재구성하는 완전히 다른 접근이다. QueryCentricRecomputeCache(05-06)가 재계산 예산 배분에 집중한 반면, 이 기법은 세그먼트 표현 압축에 집중한다.
- **예상 효과**: 비연속 히트율 +15~25%p (의미 완결 청크로 재사용 정확도 향상), TPOT 최대 2.61× 가속 (SemantiCache arXiv 2603.14303 실증), 세그먼트 크기 −30~60% (GSC 병합), accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-3: CrossChunkDependencyHybridReuse — 교차-청크 어텐션 의존성 보완 전략 조합 (선택적 재계산 + 병렬 처리)
- **가설**: 비연속 세그먼트를 조합할 때 교차-세그먼트(교차-청크) 어텐션 의존성이 출력 품질을 저하시키는 근본 문제를, CLC 실험 연구(arXiv 2603.20218)의 핵심 발견 — "기존 CLC 방법들이 정확도 제약(재계산 필요) 또는 적용 범위 제약(청크 독립성 가정)으로 나뉘며 상호 보완적" — 에 기반해 "선택적 교차-청크 재계산(ProphetKV 기반, 이미 기구현) + 병렬 독립 처리(InfoFlowKV 기반, 이미 기구현)"을 동적으로 조합하는 메타-전략을 설계하면, 이전 사이클에서 두 전략을 독립적으로 사용한 것과 달리 "어느 세그먼트 경계에서 재계산이 필요한지"를 분류한 후 적절한 전략을 선택해 적용하는 새로운 혼합 재사용 파이프라인이 가능하다.
- **접근 방법**:
  1. `CrossChunkDependencyHybridReuse` 클래스 (`src/cache/cross_chunk_hybrid_reuse.py`): `CacheStore` 인터페이스 상속
  2. 의존성 분류기: 인접 세그먼트 간 교차-어텐션 의존성 강도 추정
     - 의존성 강도 = 두 세그먼트 임베딩 간 코사인 유사도 × 위치 거리 역수
     - 높은 의존성(`dep_score > threshold_recompute`, 기본 0.6): 선택적 재계산 전략 적용 (QueryCentricRecomputeCache 호출)
     - 낮은 의존성: 병렬 독립 처리 전략 적용 (InfoFlowChunkReorderCache 호출)
  3. 재계산 예산 통합 관리: 전체 `recompute_budget_ratio` 파라미터를 높은 의존성 세그먼트 경계에만 집중 배분 → 예산 효율 최대화
  4. 분류 오버헤드: 세그먼트 수 N에 대해 O(N) 유사도 계산. TriangleInequalitySegmentIndex(B-1)의 계층 인덱스를 활용해 의존성 분류도 O(log N)으로 가속 가능
  5. CLC 실험 연구 통찰 적용: 재계산-우선 전략(ProphetKV 계열)과 청크 재순서화(InfoFlowKV 계열)가 상보 관계이므로 의존성 강도에 따라 동적 혼합이 이론적으로 최적
  6. 이전 B 기법과의 차별점: QueryCentricRecomputeCache(05-06)와 InfoFlowChunkReorderCache(05-06)가 각각 독립적으로 사용된 반면, CrossChunkDependencyHybridReuse는 의존성 분류를 통해 두 전략을 동적으로 혼합하는 메타-전략이다. 이 메타-전략 자체가 새로운 알고리즘 계층이다.
- **예상 효과**: 비연속 히트율 +10~15%p 추가 (재계산-독립처리 혼합으로 각 단독 대비 교차-청크 오류 감소), accuracy delta ±0.5% 이내 (CLC 실험 연구 실증 기반), TTFT 오버헤드 +3% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity C 아이디어

### 아이디어 C-1: StochasticDepthwiseKVShareCodec — 확률적 교차-레이어 어텐션 학습 기반 depth-wise KV 공유 압축
- **가설**: KV 캐시 최적화가 주로 시간 축(어느 토큰을 유지할 것인가)에 집중한 반면, 학습 중 각 레이어가 확률적으로 자신의 KV 또는 선행 레이어의 KV를 선택하는 무작위 교차-레이어 어텐션을 적용하면, 배포 시 하드웨어 제약에 따라 depth-wise KV 공유 전략을 유연하게 선택할 수 있는 모델을 만들 수 있다. 이전 사이클의 모든 Activity C 기법(eOptShrinkQCodec, TriAttentionCodec, ManifoldKVWindowedEviction 등)이 사후(post-hoc) 압축에 집중한 것과 달리, "학습 중 내재화된 depth-wise KV 공유"라는 훈련 통합 압축 방향을 처음으로 도입한다(Stochastic KV Routing, arXiv 2604.22782 기반).
- **접근 방법**:
  1. `StochasticDepthwiseKVShareCodec` 클래스 (`src/cache/stochastic_depth_kv_share.py`): `CompressionCodec` 인터페이스 구현
  2. 파인튜닝 단계 (적용 가능 환경 전제):
     - 학습 중 각 레이어 l의 어텐션이 확률 `p_share`로 레이어 l-1의 KV를 공유하고, `1 - p_share`로 자신의 KV 사용
     - `p_share_schedule(epoch)`: 학습 초기에는 낮게 시작, 후반에 점진적 증가 → 모델이 다양한 공유 수준에 적응
  3. 배포 시 공유 전략 선택: 파인튜닝된 모델에서 하드웨어 메모리 예산에 따라 공유 레이어 수 결정
     - 메모리 예산 50%: 짝수 레이어만 자체 KV, 홀수 레이어는 선행 레이어 KV 공유
     - 메모리 예산 30%: 3레이어마다 1개만 자체 KV, 나머지 공유
  4. KV-CoRE 발견과의 결합: Key는 더 적극적으로 공유(낮은 레이어 KV 재사용), Value는 선택적으로 공유(Key 비대칭 공유, 이전 사이클 eOptShrinkQCodec의 K/V 비대칭 원칙 계승)
  5. 파인튜닝 비용 최소화: DMS(이전 사이클 기구현 참고)와 유사한 1K~2K 스텝 경량 파인튜닝. `StochasticKVRoutingTrainer` 유틸리티로 기존 모델에 quick-adapt
  6. 이전 C 기법과의 차별점: eOptShrinkQCodec(05-08)이 BBP 위상전이로 자동 랭크를 선택하는 수학적 최적화에 집중한 반면, StochasticDepthwiseKVShareCodec은 "레이어 간 KV를 나눠 쓰는 것이 가능한 모델"을 파인튜닝으로 만드는 완전히 다른 압축 패러다임이다. 사후 압축 vs 훈련 통합 압축이라는 근본적 차이.
- **예상 효과**: 메모리 −40~60% (공유 레이어 수에 따라 조정 가능), 처리량 +15~25% (메모리 절감에 따른 배치 크기 증가), accuracy delta ±0.5% 이내 (Stochastic KV Routing arXiv 2604.22782 실증: 데이터-제약 환경에서 성능 유지 또는 향상)
- **accuracy-preserving 근거**: Stochastic KV Routing(arXiv 2604.22782, Apple ML Research)가 대규모 모델의 데이터-제약 환경에서 성능을 유지하거나 향상하면서 KV 메모리 풋프린트를 크게 감소시킴을 실증. 학습 중 확률적 공유가 모델을 다양한 공유 전략에 강건하게 만들므로 배포 시 성능 저하 최소. Key 우선 공유(Value 보존 강화)가 attention 출력의 정밀도를 유지. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 파인튜닝 전후 accuracy 비교.
- **구현 난이도**: high (파인튜닝 필요)
- **우선순위**: 3

### 아이디어 C-2: SpecKVCompressionGammaController — 압축 수준 기반 최적 추측 길이 자동 선택 (압축-추측 결합 최적화)
- **가설**: 기존 추측적 디코딩에서 추측 길이(γ)를 고정(보통 γ=4)하는 것이 KV 압축 수준(FP16/INT8/NF4)에 따라 최적이 아니며, 압축 수준이 높아질수록 γ를 줄여야 정확도 저하 없이 가속을 유지할 수 있다는 실증에 기반해, draft 모델 신호(최소 신뢰도, 최대 엔트로피)를 활용해 압축 수준별 최적 γ를 경량 MLP로 자동 선택하면(SpecKV 원리), 이전 사이클의 모든 Activity C 기법이 KV 압축과 추측적 디코딩을 독립적으로 처리한 것과 달리 "압축-γ 결합 최적화"라는 새로운 차원으로 고정 γ=4 대비 56% 가속이 가능하다.
- **접근 방법**:
  1. `SpecKVCompressionGammaController` 클래스 (`src/cache/speckv_gamma_controller.py`)
  2. 경량 MLP γ 컨트롤러: 입력 = {현재 압축 수준 태그(0=FP16/1=INT8/2=NF4), min_draft_confidence, max_draft_entropy} → 출력 = 최적 γ ∈ {1, 2, 3, 4, 5, 6}
  3. 프로파일링 데이터 수집: 4개 태스크 카테고리(코드/수학/대화/RAG) × 4개 γ × 3개 압축 수준 조합에서 step-level 레코드 수집 → MLP 학습 데이터 (SpecKV arXiv 2605.02888 설계 재현)
  4. 기존 eOptShrinkQCodec(05-08 기구현)과 통합: eOptShrinkQCodec의 압축 수준 출력을 `SpecKVCompressionGammaController`의 입력으로 주입 → 2.2비트 압축 시 최적 γ 자동 선택
  5. 온라인 적응: 실제 검증(verification) 통과율이 기대값에서 벗어나면 γ 컨트롤러 출력을 온라인 보정 (EMA 기반 온라인 갱신)
  6. 오버헤드: MLP 추론 비용 0.34ms (SpecKV arXiv 2605.02888 실증). 추가 계산 오버헤드 최소.
  7. 이전 C 기법과의 차별점: 이전 사이클의 모든 C 기법이 KV 메모리 압축 자체에 집중하고 추측적 디코딩과의 상호 작용을 무시한 반면, SpecKVCompressionGammaController는 "압축이 높을수록 γ를 낮춰 정확도 보상"이라는 결합 최적화 로직을 명시적으로 구현한다.
- **예상 효과**: 고정 γ=4 대비 +56% 가속 (SpecKV arXiv 2605.02888 실증), eOptShrinkQCodec + SpecKVCompressionGammaController 결합 시 2.2비트 압축에서도 추측 디코딩 속도 보존, accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: SpecKV(arXiv 2605.02888)가 FP16/INT8/NF4 압축 수준별 최적 γ를 실증적으로 도출. 압축 수준이 높아질수록 γ를 낮춰 검증 통과율을 유지함으로써 정확도 저하를 방지. draft 신호(min confidence, max entropy)가 현재 배치의 최적 γ를 실시간으로 추정해 정확도-속도 트레이드오프를 동적으로 최적화. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 압축 수준별 γ-정확도 곡선 측정.
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-3: ContextIntensiveAccuracyGuard — 컨텍스트-집약적 태스크 정확도 보호 계층 (Text2JSON 기반 평가 강화)
- **가설**: 기존 KV 오프로딩/퇴거 방법들이 일반 벤치마크(LongBench, RULER)에서는 정확도를 유지하지만 대량 구조화 지식 추출(Text2JSON 등 컨텍스트-집약적 태스크)에서 유의미한 성능 저하를 겪는 문제를(KV Cache Offloading for Context-Intensive Tasks, arXiv 2604.08426 실증), 컨텍스트 내 정보 밀도를 실시간 추정하고 밀도 높은 컨텍스트에서는 압축 비율을 자동으로 하향 조정하는 적응형 정확도 보호 계층을 설계하면, 이전 사이클의 모든 C 기법이 일반 벤치마크 기반 정확도 검증에 머문 것과 달리 "컨텍스트-집약적 태스크에서의 정확도 ±1% 이내"를 실질적으로 보장할 수 있다.
- **접근 방법**:
  1. `ContextIntensiveAccuracyGuard` 클래스 (`src/cache/context_intensive_guard.py`)
  2. 컨텍스트 정보 밀도 추정: 입력 컨텍스트의 첫 128 토큰을 샘플링해 고유 엔티티 밀도(entity density) 추정
     - 밀도 측정: 고유 명사 비율, 숫자/기호 비율, 평균 토큰 엔트로피 가중합
     - `context_density_score = w1 × entity_ratio + w2 × numeric_ratio + w3 × token_entropy`
  3. 압축 비율 자동 조정: `context_density_score > threshold_high_density`이면 현재 압축 코덱(eOptShrinkQCodec 등)의 압축 비율을 보수적 수준으로 자동 제한
     - 고밀도(≥0.7): 최대 4비트 압축 (하한 제한)
     - 중간 밀도(0.4~0.7): 2.2~4비트 (정상 eOptShrinkQ 동작)
     - 저밀도(≤0.4): 2.2비트 이하 공격적 압축 허용
  4. Text2JSON 벤치마크 통합: 평가 파이프라인에 Text2JSON 태스크 추가 (`experiments/run_text2json_accuracy.py`). arXiv 2604.08426의 벤치마크 구성을 재현해 KV 압축 방법들의 컨텍스트-집약 정확도 측정.
  5. 기존 C 기법과의 통합: `ContextIntensiveAccuracyGuard`를 eOptShrinkQCodec, ManifoldKVWindowedEviction 등의 입력 게이트로 사용 → 컨텍스트 밀도에 따라 압축 파라미터 자동 주입
  6. 이전 C 기법과의 차별점: 이전 사이클의 GVoteSegmentBudgetAllocator(05-08)가 미래 쿼리 투표로 세그먼트별 예산을 결정하는 반면, ContextIntensiveAccuracyGuard는 입력 컨텍스트의 정보 밀도를 기준으로 압축 한도 자체를 보호하는 완전히 다른 보호 메커니즘이다.
- **예상 효과**: 컨텍스트-집약 태스크 정확도 보존 (기존 방법의 유의미한 저하 제거), accuracy delta ±0.5% 이내 (일반 + 집약 태스크 통합), 고밀도 컨텍스트에서 메모리 효율 −40~60% (보수적 압축이지만 목표 −30% 초과 달성)
- **accuracy-preserving 근거**: KV Cache Offloading for Context-Intensive Tasks(arXiv 2604.08426)가 컨텍스트-집약 태스크에서 기존 방법들의 정확도 저하를 체계적으로 문서화. 컨텍스트 정보 밀도 기반 적응형 압축 한도가 "정보가 많은 컨텍스트는 덜 압축"이라는 원칙으로 정확도 보호. 고밀도 컨텍스트에서 4비트 하한 제한이 "정확도 ±1% 이내" 제약의 실질적 보장. 검증: LongBench(일반) + Text2JSON(집약) + RULER + WikiText-2 perplexity ±1% 통합 검증.
- **구현 난이도**: low
- **우선순위**: 1

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: A+B — PPDAppendPrefillRouter + TriangleInequalitySegmentIndex (히트-인식 PPD 라우터)
- **조합 근거**: PPDAppendPrefillRouter(A-1)가 Turn 2+ 요청을 D 노드 append-prefill 또는 P 노드 full prefill로 라우팅할 때, "세그먼트 히트 예상 여부"가 핵심 라우팅 신호다. TriangleInequalitySegmentIndex(B-1)가 O(log N) 속도로 D 노드 캐시의 세그먼트 히트 가능성을 빠르게 추정하면, "히트 예상 시 → D 노드 append, 미스 예상 시 → P 노드 full prefill"하는 캐시-인식 PPD 라우터가 실현된다. 트렌드 리포트의 A+B 시너지 포인트를 직접 구현한다.
- **구체 설계**:
  1. `HitAwarePPDRouter` 모듈 (`src/scheduler/hit_aware_ppd_router.py`)
  2. Turn 2+ 요청 수신 시: `TriangleInequalitySegmentIndex.estimate_hit_probability(request_segments)` → O(log N) 히트 확률 추정
  3. 히트 확률 > `threshold_append`(기본 0.7): D 노드 append-prefill 선택
  4. 히트 확률 ≤ 0.7: P 노드 full prefill 선택 (KV 재전송 필요 → 비용 높지만 정확도 보장)
  5. 임계값 적응: 실제 D 노드 히트율 피드백으로 `threshold_append` 온라인 조정 (히트율 낮아지면 임계값 상향)
  6. SemanticBoundarySegmentCache(B-2)와 통합: 의미 경계 청크의 고품질 임베딩으로 히트 확률 추정 정밀도 향상
- **예상 복합 효과**: 처리량 +35~50%, Turn 2+ TTFT −68% (PPD 효과) + 추가 히트율 향상 (로그 시간 검색으로 더 많은 D 노드 히트 발견), 비연속 히트율 +25~35%p
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: B+C — SemanticBoundarySegmentCache + ContextIntensiveAccuracyGuard (의미 경계 압축 + 집약 정확도 보호)
- **조합 근거**: SemanticBoundarySegmentCache(B-2)의 GSC 클러스터 병합이 세그먼트를 의미 코어로 압축하고, ContextIntensiveAccuracyGuard(C-3)가 컨텍스트 정보 밀도에 따라 압축 한도를 보호한다. 두 기법을 결합하면 의미 코어 병합으로 세그먼트 크기가 줄면서도 고밀도 컨텍스트에서는 의미 코어 병합 비율을 자동으로 제한해 정확도를 보호한다. 트렌드 리포트의 B+C 시너지("의미 경계를 기준으로 depth-wise 공유 범위 제한")를 실용적 형태로 구현한다.
- **구체 설계**:
  1. `SemanticCoreCompressionPipeline` 모듈 (`src/cache/semantic_core_compression.py`)
  2. 컨텍스트 밀도 평가: `ContextIntensiveAccuracyGuard.assess(context)` → 밀도 스코어 반환
  3. GSC 병합 비율 제어: 밀도 스코어에 따라 `SemanticBoundarySegmentCache`의 클러스터 병합 비율 동적 설정
     - 고밀도: 병합 비율 ≤ 30% (의미 코어 보존 강화)
     - 저밀도: 병합 비율 ≤ 70% (더 공격적 압축)
  4. 비례 어텐션 스케일링 유지: 병합 비율과 무관하게 비례 어텐션 재조정 항상 적용
  5. eOptShrinkQCodec(05-08 기구현) 후처리: 의미 코어 병합 후 남은 KV에 eOptShrinkQ 압축 추가 적용 (B+C 이중 압축)
- **예상 복합 효과**: 처리량 +25~35%, 메모리 −50~70% (의미 코어 병합 + eOptShrinkQ 이중 압축), 비연속 히트율 +20~30%p, 컨텍스트-집약 정확도 ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: A+B+C — PPDAppendPrefillRouter + TriangleInequalitySegmentIndex + SpecKVCompressionGammaController (멀티턴 에이전틱 완전 최적화 파이프라인)
- **조합 근거**: 트렌드 리포트의 A+B+C 통합 시너지 포인트 — PPD 라우팅(A) → LycheeCluster 세그먼트 검색(B) → Stochastic KV Routing 압축(C) → 추측 디코딩 최적화 — 를 이번 사이클의 핵심 아이디어로 구현한다. PPDAppendPrefillRouter(A-1)가 Turn 2+ 요청을 D 노드로 라우팅하고, TriangleInequalitySegmentIndex(B-1)가 D 노드에서 관련 세그먼트를 O(log N)으로 검색하며, SpecKVCompressionGammaController(C-2)가 현재 압축 수준에서 최적 γ로 추측적 디코딩을 가속한다.
- **구체 설계**:
  1. Turn 2+ 요청 수신 → PPDAppendPrefillRouter가 히트 예상 여부 확인
  2. 히트 예상 시: TriangleInequalitySegmentIndex가 O(log N)으로 관련 세그먼트 검색 → D 노드 append-prefill 실행
  3. append-prefill 결과 KV: eOptShrinkQCodec(05-08 기구현)으로 2.2비트 압축 → ContextIntensiveAccuracyGuard가 압축 한도 보호
  4. 디코딩 단계: SpecKVCompressionGammaController가 압축 수준 2.2비트에 최적화된 γ를 자동 선택 → 추측적 디코딩 가속
  5. 세그먼트 캐시 업데이트: 새로 생성된 KV를 SemanticBoundarySegmentCache에 의미 경계로 분할 후 TriangleInequalitySegmentIndex에 등록
- **예상 복합 효과**: 처리량 +50~65%, 메모리 −60~75%, 비연속 히트율 +35~45%p, Turn 2+ TTFT −68% (PPD), 추측 디코딩 +56% (SpecKV), accuracy delta ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 2

## 이번 사이클 최우선 구현 타겟

**Cross-1(A+B): PPDAppendPrefillRouter(A-1) + TriangleInequalitySegmentIndex(B-1) + HitAwarePPDRouter 통합**, 보조로 SemanticBoundarySegmentCache(B-2), SpecKVCompressionGammaController(C-2), ContextIntensiveAccuracyGuard(C-3) 구현

구현 순서:
1. **B-1** `TriangleInequalitySegmentIndex`: 삼각부등식 재귀 계층 인덱스 구성 + best-first 탐색 + 서브트리 가지치기. `CacheStore` 인터페이스 통합. StaticDynamicSegmentCache(05-08 기구현) 위에 인덱스 레이어로 추가. **검색 속도 O(N) vs O(log N) 비교 측정 필수**.
2. **A-1** `PPDAppendPrefillRouter`: Turn 1/2+ 분류 + D/P 노드 동적 선택 + TriangleInequalitySegmentIndex 히트 확률 연동. `BaseScheduler` 상속. TTFT p50/p99 측정 포함.
3. **Cross-1** `HitAwarePPDRouter`: A-1 + B-1 통합. 히트 확률 임계값 온라인 적응 포함.
4. **B-2** `SemanticBoundarySegmentCache`: 의미 경계 탐지 + GSC 클러스터링 + 비례 어텐션. `CacheStore` 구현. TriangleInequalitySegmentIndex와 통합.
5. **C-2** `SpecKVCompressionGammaController`: 경량 MLP γ 컨트롤러 + eOptShrinkQCodec(기구현) 통합. draft 신호 파이프라인. **accuracy-preserving 검증 필수**: 압축 수준별 γ-정확도 곡선 + WikiText-2 perplexity ±1%.
6. **C-3** `ContextIntensiveAccuracyGuard`: 컨텍스트 밀도 추정 + 압축 비율 자동 조정 게이트. **Text2JSON 벤치마크 통합 필수**.

이유:
- TriangleInequalitySegmentIndex는 이전 사이클까지의 모든 B 기법의 공통 병목(O(N) 검색)을 O(log N)으로 해결하는 가장 근본적인 인프라 개선이다. 비연속 히트율 ≥30% 목표 달성에 가장 직접 기여한다.
- PPDAppendPrefillRouter는 Turn 2+ TTFT 68% 단축이라는 EuroSys/arXiv 최고 수준 실증 데이터를 가진 신규 설계 차원을 도입하며, Activity A의 P/D 라우팅 패러다임을 전환한다.
- SpecKVCompressionGammaController는 기구현된 eOptShrinkQCodec과 즉시 통합 가능한 낮은 구현 비용으로 추측적 디코딩 속도를 56% 추가 향상시킨다.
- ContextIntensiveAccuracyGuard와 Text2JSON 벤치마크 통합은 "정확도 ±1% 이내" 제약의 실질적 검증을 강화하는 필수 평가 인프라다.

## planner에게 전달할 요청사항
- **Activity B-1(TriangleInequalitySegmentIndex) 구현 시**: `src/cache/triangle_index.py` 신규 생성. pivot 선택: 각 분기에서 가장 먼 두 세그먼트 임베딩을 재귀적으로 선택. `max_radius` 저장으로 삼각부등식 가지치기 활성화. **검색 속도 O(N) vs O(log N) 벤치마크 필수**: `experiments/run_index_speed_benchmark.py` 작성, N=100/1K/10K 세그먼트 수에서 검색 시간 측정. StaticDynamicSegmentCache(05-08 기구현)의 정적 세그먼트를 인덱스 초기 노드로 사용. 인덱스 업데이트(새 세그먼트 삽입/삭제) 비용 측정 포함.
- **Activity A-1(PPDAppendPrefillRouter) 구현 시**: `src/scheduler/ppd_append_prefill_router.py` 신규 생성. Turn 분류는 `session_id` + `turn_count` 메타데이터로 결정. D/P 노드 선택 로직은 `TriangleInequalitySegmentIndex.estimate_hit_probability()` 결과를 기반으로. `threshold_append` 파라미터(기본 0.7) YAML 외부화. TTFT p50/p99 측정 스크립트(`experiments/run_ppd_ttft.py`) 작성. 멀티턴 시나리오(5턴 대화) 시뮬레이션 포함.
- **Cross-1(HitAwarePPDRouter) 구현 시**: `src/scheduler/hit_aware_ppd_router.py` 신규 생성. A-1과 B-1을 단일 라우팅 결정으로 결합. 히트 확률 임계값 온라인 적응 로직(`threshold_append`를 실제 히트율 피드백으로 EMA 갱신). Turn 1/2+ 히트율 분리 측정.
- **Activity B-2(SemanticBoundarySegmentCache) 구현 시**: `src/cache/semantic_boundary_cache.py` 신규 생성. 의미 경계 탐지: 구분자 기반 규칙(`.!?\n\n` + 코드 블록). GSC 클러스터링: `min_cluster_size=3`, `max_merge_ratio=0.7` 파라미터 YAML 외부화. 비례 어텐션: `attention_weight_core = sum(attention_weights in cluster)`. **accuracy-preserving 검증 필수**: 병합 비율별 WikiText-2 perplexity ±1% + TPOT 측정.
- **Activity C-2(SpecKVCompressionGammaController) 구현 시**: `src/cache/speckv_gamma_controller.py` 신규 생성. 경량 MLP: 입력 차원 3 (압축 수준 one-hot + min_confidence + max_entropy), 출력 6 (γ 선택). 프로파일링 데이터: 최소 512개 스텝-레벨 레코드로 MLP 학습. eOptShrinkQCodec(기구현)의 압축 수준 출력을 자동으로 MLP 입력으로 주입하는 인터페이스 구현. **accuracy-preserving 검증 필수**: 압축 수준별 검증 통과율 + WikiText-2 perplexity ±1%.
- **Activity C-3(ContextIntensiveAccuracyGuard) 구현 시**: `src/cache/context_intensive_guard.py` 신규 생성. 밀도 추정은 첫 128 토큰 샘플링 + 엔티티 비율/수치 비율/엔트로피 가중합. 가중치 파라미터(`w1=0.3, w2=0.3, w3=0.4`) YAML 외부화. **Text2JSON 벤치마크 통합 필수**: `experiments/run_text2json_accuracy.py` 작성, arXiv 2604.08426 벤치마크 재현. 압축 코덱(eOptShrinkQCodec, ManifoldKVWindowedEviction)과의 게이트 인터페이스 구현.
