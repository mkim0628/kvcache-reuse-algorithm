## 이번 사이클 초점
**B+C** (Activity B 비연속 세그먼트 재사용 + Activity C FibQuant 랜덤-접근 VQ 압축) — 보조로 **A** CacheSolidarity 보안-인식 스케줄러, **A+C** Fluxion 오프로딩-인식 스파스 복원

**선택 이유**: Activity B에 이번 주 신규 논문이 없는 상황에서, FibQuant의 랜덤-접근 VQ가 비연속 세그먼트의 독립 압축·복원에 이상적으로 부합하는 B+C 통합이 가장 임팩트 높다. CacheSolidarity는 멀티테넌트 프로덕션 환경에서 Activity B 비연속 세그먼트 크로스-요청 공유의 실용 배포 경로를 여는 새로운 A 기법으로, 중기 구현 타겟이다. Fluxion의 출력-인식 CPU-GPU 스파스 복원은 기구현된 TemporalTieredKVManager와 결합할 때 A+C 시너지가 성립한다.

## Activity A 아이디어

### 아이디어 A-1: CacheSolidarity보안인식세그먼트라우터 (SecurityAwareSegmentIsolationScheduler)
- **가설**: 멀티테넌트 LLM 서빙에서 비연속 세그먼트 크로스-요청 공유가 타이밍 사이드채널(APC side channel)을 통해 사용자 프라이버시를 침해할 수 있으므로, 세그먼트별 크로스-사용자 공유 패턴을 경량 모니터링으로 추적하여 "공격 유발 가능 의심 세그먼트"만 선택적으로 격리하고 나머지 비연속 세그먼트는 크로스-요청 재사용을 그대로 허용하면, 보안 격리 방어 기법 대비 캐시 재사용률을 70% 보존하면서 멀티테넌트 환경에서 Activity B의 비연속 세그먼트 공유를 실용 배포할 수 있다.
- **접근 방법**:
  1. `SecurityAwareSegmentIsolationScheduler` 클래스 (`src/scheduler/security_aware_segment_scheduler.py`): `BaseScheduler` 상속
  2. 세그먼트 공유 패턴 모니터링: 각 세그먼트 ID에 대해 "크로스-사용자 접근 빈도"를 슬라이딩 윈도우(기본 1,000 요청)로 추적. 단일 사용자가 동일 세그먼트를 반복 접근하는 패턴이 아닌 "다수 사용자가 동일 세그먼트를 빠르게 교대 접근"하는 패턴을 탐지.
  3. 의심 세그먼트 플래깅: 크로스-사용자 접근 속도가 임계값(`side_channel_rate_threshold`, 기본 10 접근/초) 초과 시 해당 세그먼트에 보안 플래그 설정. 플래그된 세그먼트는 사용자별 독립 캐싱(격리), 미플래그 세그먼트는 크로스-요청 공유 허용.
  4. 자동 플래그 해제: 의심 접근 패턴이 `cooldown_window`(기본 60초) 동안 사라지면 플래그 자동 해제 → 격리 → 공유 복원.
  5. 기존 `TriangleInequalitySegmentIndex`(05-09 기구현)와 통합: 플래그된 세그먼트는 인덱스에서 사용자별 파티션으로 분리 저장. 미플래그 세그먼트는 전역 공유 파티션에서 O(log N) 검색.
  6. 이전 A 기법과의 차별점: 이전 14개 사이클의 모든 A 기법이 캐시 히트율 최대화만 추구하고 보안 제약을 설계 차원에서 다루지 않은 반면, 이 기법은 CacheSolidarity(2603.10726)의 원리를 Activity B 세그먼트 레벨로 재구성해 "보안 vs. 효율" 트레이드오프를 명시적으로 제어하는 첫 스케줄러다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 보안 플래그 정보를 노드 간 gossip으로 동기화(플래그된 세그먼트 ID + 플래그 타임스탬프만 전파, KV 데이터는 전파 없음). 노드별 독립 모니터링 후 클러스터-레벨 집계로 글로벌 의심 세그먼트를 식별하는 2단계 집계 구조.
- **예상 효과**: 처리량 +15~25% (보안 격리 방어 대비 캐시 재사용률 70% 보존으로 TTFT 단축), 비연속 히트율 미플래그 세그먼트에서 기존 대비 유지, 멀티테넌트 APC 사이드채널 위험 제거, Scheduling overhead TTFT +3% 이내
- **구현 난이도**: medium
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: FibQuantRandomAccessSegmentCache (FibQuantVQSegmentCompressedCache)
- **가설**: 비연속 KV 세그먼트를 저장할 때 FibQuant(2605.11478)의 구형-베타 소스 매칭 방사상-각도 코드북으로 각 세그먼트를 독립적으로 압축하면, 임의 세그먼트를 재사용 시 해당 세그먼트만 선택적으로 복원(랜덤-접근)하는 것이 가능하므로, 비연속 세그먼트 캐시의 메모리 효율을 4~10× 향상시키면서 비연속 히트율과 정확도를 유지할 수 있다.
- **접근 방법**:
  1. `FibQuantVQSegmentCache` 클래스 (`src/cache/fibquant_vq_segment_cache.py`): `CacheStore` 인터페이스 상속
  2. 세그먼트 압축 저장: 각 세그먼트 KV를 FibQuant 방식으로 독립 인코딩. 구현 세부:
     - 구형 정규화 후 반경 분포를 베타 분포로 모델링 → 베타-분위수 반경 양자화 격자 계산.
     - 피보나치/Roberts-Kronecker 격자로 고차원 구면에 균일 방향 집합 구성 → 방향 양자화.
     - 소규모 보정 데이터(≥200 세그먼트)로 코드북 1회 다중-재시작 Lloyd-Max 정제 후 재사용.
     - 저장 형식: `(radial_code, direction_code, codebook_id)` per block (block_size=64 또는 128).
  3. 랜덤-접근 복원: 재사용 요청 시 필요한 세그먼트 ID에 해당하는 코드만 on-demand 복원. 다른 세그먼트 복원 불필요 → 비연속 재사용의 핵심 장점 유지.
  4. 기존 `SegmentedHashCache`(`src/cache/segmented.py`)의 세그먼트 키 인터페이스를 상속해 기존 content-hash 기반 인덱싱과 호환.
  5. 기존 `RSimVQCodec`(05-10 기구현)과의 차별점: RSimVQCodec은 pre-RoPE 공간 k-means 코드북(잔차 VQ, 코드북 크기 M=64~256)을 사용하는 반면, FibQuantVQSegmentCache는 구형-베타 분포를 수학적으로 최적 활용하는 방사상-각도 분리 코드북으로 10× 이상 고압축 영역에서 성능이 차별화된다.
  6. 압축률 선택: `compression_target` 파라미터(5×/10×/20×)로 코드북 해상도 조절. 비연속 재사용 세그먼트는 4× 이상 압축에서 랜덤-접근 복원 오버헤드가 어텐션 계산 시간 내에 숨겨지도록 설계.
- **예상 효과**: 비연속 세그먼트 캐시 메모리 −75~90% (4~10× 압축), 동일 메모리 예산에서 보존 가능한 세그먼트 수 4~10× 증가 → 비연속 히트율 +15~30%p, accuracy delta 5× 압축 시 ±0.1% / 10× 시 ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 1

## Activity C 아이디어

### 아이디어 C-1: FibQuantSphericalBetaVQCodec
- **가설**: 기존 스칼라 양자화와 k-means VQ(RSimVQCodec 포함)가 KV 벡터의 구형 정규화 후 형성되는 구형-베타 분포 기하 구조를 무시해 고압축 영역(10×+)에서 품질이 급격히 저하되는 문제를, FibQuant(2605.11478)의 방사상-각도 분리 코드북 + 베타-분위수 반경 + 피보나치 방향 집합으로 대체하면, 10× 이상 압축에서 유일하게 정확도를 보존하는 범용 랜덤-접근 VQ 코덱을 구성할 수 있다.
- **접근 방법**:
  1. `FibQuantVQCodec` 클래스 (`src/cache/fibquant_vq_codec.py`): `CompressionCodec` 인터페이스 구현
  2. 코드북 학습 (오프라인 1회):
     - 보정 데이터셋에서 각 레이어·헤드의 KV 벡터를 block 단위(block_size=64)로 구형 정규화.
     - 반경 분포를 베타 분포로 피팅 → 베타-분위수 격자로 반경 코드북 생성.
     - 고차원 구면에 피보나치 수열 기반 N_dir개 균일 방향 집합 생성 (N_dir = 2^bits_direction).
     - 다중-재시작 Lloyd-Max로 코드북 정제 (재시작 횟수 R=10, 보정 반복 K=5).
  3. 인코딩: 각 KV 블록을 구형 정규화 → (반경 코드, 방향 코드) 2-tuple로 인코딩. 저장 크기: `ceil(log2(N_radii) + log2(N_dir)) bits/block`.
  4. 디코딩: (반경 코드, 방향 코드) → 반경값 × 방향 벡터 → 원본 KV 블록 재구성 (on-demand).
  5. 기존 RSimVQCodec(05-10 기구현)과의 결합: RateQuantReverseWaterfillingCodec(05-11 기구현)의 헤드별 비트 할당 출력을 FibQuantVQCodec의 코드북 해상도 입력으로 사용하는 "RateQuant → FibQuant" 파이프라인으로 이론 최적 비트할당 + 구형-베타 VQ 결합.
  6. 정밀도 파라미터: `bits_total` (5~/비트/차원 기준), 방향/반경 비트 분할은 보정에서 자동 결정.
- **예상 효과**: 메모리 −80~95% (4~34× 압축), accuracy delta 4× 시 ±0.1%, 10× 시 ±0.5%, 34× 시 ±2.0%에서 정확도 보존 (FibQuant 2605.11478 실증 기반)
- **accuracy-preserving 근거**: FibQuant(2605.11478, POSTECH)가 GPT-2 small에서 5× 압축 코사인 유사도 0.99, 34× 시 0.95를 실증. TinyLlama-1.1B 4× 압축 시 fp16 대비 perplexity 차이 0.10 이내. 구형-베타 소스 매칭이 기하 구조 손실을 수학적으로 최소화하며, 피보나치 방향 집합이 고차원 구면 균일 커버리지를 보장해 방향 양자화 오류를 이론적으로 최소화한다. RateQuant 헤드별 최적 비트 할당과 결합하면 총 비트 예산 대비 왜곡이 정보이론적 하한에 근접한다.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: ABSparseAdaptiveHeadBlockSizeCodec
- **가설**: 기존 블록-스파스 어텐션이 모든 어텐션 헤드에 균일한 블록 크기를 적용해 헤드별 중요도 패턴의 이질성을 무시하는 문제를, AB-Sparse(2605.12110)의 핵심 발견 — 헤드마다 블록 크기 민감도가 현저히 다르지만 다양한 입력에 걸쳐 안정적이다 — 을 이용해 소규모 보정 데이터로 각 헤드의 최적 블록 크기를 1회 사전 결정하면, 기존 블록-스파스 어텐션 대비 5.43% 정확도 향상과 처리량 보존을 동시에 달성할 수 있다.
- **접근 방법**:
  1. `ABSparseHeadBlockProfiler` 클래스 (`src/cache/ab_sparse_head_block.py`): `CompressionCodec` 인터페이스 구현
  2. 헤드별 블록 크기 보정 (배포 전 1회):
     - 보정 데이터셋(다양한 도메인, ≥50 샘플)에서 각 어텐션 헤드 h에 대해 블록 크기 후보 집합 `B = {16, 32, 64, 128}`로 근사 어텐션과 완전 어텐션의 출력 코사인 유사도를 측정.
     - 헤드 h의 최적 블록 크기 `b_h* = argmax_{b in B} CosSim(Attn_full, Attn_sparse(b))`. 결과를 `configs/ab_sparse_head_config_layer_{l}.json`에 저장.
  3. 중심 양자화 (centroid quantization): 블록 중심 벡터를 랭킹에만 사용하므로 FP8로 저장해 메모리 절감 (AB-Sparse 원리).
  4. 가변 블록 크기 추론: 헤드별로 `b_h*`를 로드해 스파스 어텐션 실행. `paged_kv_compatible=True` 옵션으로 vLLM paged KV 캐시와 호환.
  5. 기존 `TriAttentionCodec`(05-06 기구현)의 pre-RoPE 중요도 추정과의 결합: TriAttention이 중요 토큰을 선별하고, AB-Sparse가 해당 중요 토큰 블록의 최적 크기로 스파스 어텐션을 실행하는 "선별 + 최적 블록 스파스" 이중 파이프라인.
  6. 이전 C 기법과의 차별점: MISAMoEIndexerAccelerator(05-12 기구현)가 인덱서 헤드를 MoE 라우터로 선택적 활성화하는 반면, ABSparseHeadBlockProfiler는 어텐션 헤드별 블록 크기를 보정으로 결정하는 것으로 최적화 목표(계산량 vs. 정확도)와 메커니즘이 다르다.
- **예상 효과**: 메모리 효율적 스파스 어텐션 (처리량 오버헤드 없음), 기존 균일 블록-스파스 대비 정확도 +5.43%p (AB-Sparse 2605.12110 실증), accuracy delta ±1% 이내 (이전 스파스 방법 대비 큰 향상), TTFT 무변화
- **accuracy-preserving 근거**: AB-Sparse(2605.12110, SJTU/Ant Group)가 기존 블록-스파스 어텐션 기준선 대비 최대 5.43% 정확도 향상 + 처리량 오버헤드 없음을 실증. 헤드별 블록 크기 민감도가 다양한 입력에 걸쳐 안정적임을 발견해 보정 기반 1회 프로파일링의 타당성 확립. paged KV 캐시 호환 커스텀 커널로 vLLM 이식 직접 지원. 중심 양자화가 랭킹 정밀도에 무영향 (실증됨). 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 헤드별 블록 크기 민감도 프로파일 측정.
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-3: FluxionOutputAwareCPUKVSparseRestoreCodec
- **가설**: CPU-상주 KV 캐시를 어텐션 계산 시 전량 GPU로 복원하는 대신, Fluxion(2605.07719)의 출력-인식 KV 예산 할당 원리 — 어텐션 점수뿐만 아니라 헤드별 출력 편차와 상대적 기여도를 함께 고려해 KV 예산을 배분 — 을 적용하면, 필요한 KV만 선택적으로 CPU→GPU 전송해 처리량을 1.5~3.7× 향상시키면서 정확도를 보존할 수 있다.
- **접근 방법**:
  1. `FluxionSparseRestoreCodec` 클래스 (`src/cache/fluxion_sparse_restore.py`): 기존 `TemporalTieredKVManager`(05-10 기구현)의 DRAM 계층 복원 경로에 통합
  2. 출력-인식 KV 예산 할당:
     - 각 헤드 h에 대해 출력 편차(output variance) `σ²_h`와 상대적 기여도(relative contribution) `w_h`를 보정에서 사전 계산.
     - 쿼리 시 예산 `B_h = B_total × (σ²_h × w_h) / Σ_j(σ²_j × w_j)`. 각 헤드에 출력 기여도 비례한 KV 예산 배분.
  3. 헤드별·세분성-인식 희소성 구성: 헤드 h에 대해 최적 (예산 B_h, 블록 크기 g_h) 쌍을 보정에서 결정 (`configs/fluxion_head_config_layer_{l}.json`).
  4. 크로스-디바이스 조율 실행: CPU가 선택된 KV 블록을 GPU로 비동기 전송하는 동안 GPU는 이미 전송된 KV로 부분 어텐션 계산 (I/O-연산 오버랩). `torch.cuda.Stream` 이중 스트림 활용.
  5. `TemporalTieredKVManager`(05-10 기구현) DRAM 계층 → `FluxionSparseRestoreCodec` 스파스 복원 통합 경로: 기존 블록-단위 스트리밍 어텐션을 Fluxion 출력-인식 예산으로 교체.
  6. 이전 C 기법과의 차별점: `TemporalTieredKVManager`가 전체 DRAM KV를 블록 단위로 스트리밍 복원하는 반면, FluxionSparseRestoreCodec은 출력 기여도 기반으로 필요한 KV만 선택적으로 복원한다. LaProxOutputAwareLayerEviction(05-11 기구현)이 KV를 영구 퇴거하는 반면, Fluxion은 CPU에 임시 보존 후 스파스 접근하는 가역적(reversible) 압축이다.
- **예상 효과**: CPU→GPU 전송 KV 양 −60~80% (스파스 선택), 처리량 +1.5~3.7× (Fluxion 2605.07719 실증 기반), GPU 유휴율 70.65% → 45.78% 감소 (Fluxion 실증), accuracy delta −0.26 평균 품질 저하에 불과
- **accuracy-preserving 근거**: Fluxion(2605.07719)이 2개 모델, 3개 벤치마크, 40개 태스크에서 최강 고정-스파스 기준선(KV 예산 0.05) 대비 1.5~3.7× 속도향상 달성, 평균 품질 저하 -0.26에 불과함을 실증. 출력-인식 KV 예산 할당이 어텐션 점수 단독 대비 실제 출력에 미치는 영향을 정확히 추정해 중요 KV 누락을 최소화. 헤드별 희소성 구성이 각 헤드의 출력 민감도에 맞게 보정되어 전체 어텐션 출력의 왜곡이 한계 내로 유지. 검증: LongBench + 40개 태스크 + WikiText-2 perplexity ±1% + CPU 스파스 복원 전후 perplexity 비교.
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 Cross-1: FibQuantVQSegmentCacheWithAdapShot (B+C)
- **조합 근거**: FibQuantVQSegmentCache(B-1)가 각 비연속 세그먼트를 FibQuant로 독립 압축 저장하고, 기구현된 RoPEReencodingNonContiguousCache(05-12)의 position-decoupled KV 저장 원리를 결합하면, "위치-독립 저장 + FibQuant 고압축"의 이중 효과로 더 많은 비연속 세그먼트를 더 작은 메모리에 보존하면서 임의 위치에서 무손실 재사용이 가능하다. 트렌드 리포트의 B+C 시너지("FibQuant 랜덤-접근 VQ + 비연속 세그먼트 독립 압축")를 직접 구현한다.
- **구체 설계**:
  1. `FibQuantPositionFreeSegmentCache` 모듈 (`src/cache/fibquant_position_free_segment.py`)
  2. 세그먼트 저장: KV를 pre-RoPE 상태(position-decoupled)로 수집 → FibQuantVQCodec으로 방사상-각도 인코딩 → `(radial_code, direction_code, pre_rope_flag=True)` 저장.
  3. 재사용 시: FibQuantVQCodec.decode() → pre-RoPE KV 복원 → 목표 위치의 RoPE 행렬 in-place 적용 → 어텐션 계산.
  4. 인덱스: 기존 content-hash(위치 무관)로 세그먼트 검색. TriangleInequalitySegmentIndex(05-09 기구현)로 O(log N) 검색.
  5. 메모리 예산 관리: FibQuant 10× 압축으로 동일 메모리에 10× 더 많은 세그먼트 보존 → 비연속 히트율 대폭 향상.
  6. RateQuantReverseWaterfillingCodec(05-11 기구현)의 헤드별 비트 할당 출력을 FibQuantVQCodec의 코드북 해상도 입력으로 연결하는 선택적 결합.
- **예상 복합 효과**: 처리량 +20~35%, 메모리 −80~90% (FibQuant 10× + position-decoupled 저장), 비연속 히트율 +20~35%p (동일 메모리에 10× 더 많은 세그먼트), accuracy delta ±0.5% 이내 (FibQuant 4× 시 ±0.1% 실증 기반)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: CacheSolidarity보안인식FluxionスパースRestoreScheduler (A+C)
- **조합 근거**: SecurityAwareSegmentIsolationScheduler(A-1)가 멀티테넌트 환경에서 미플래그(보안 안전) 세그먼트와 플래그(의심) 세그먼트를 구분하고, FluxionSparseRestoreCodec(C-3)을 미플래그 공유 세그먼트에만 적용하면, 격리가 필요한 세그먼트는 전량 복원(정확도 보장), 공유 가능 세그먼트는 Fluxion 스파스 복원(효율 최적화)하는 "보안-인식 이중 복원 정책"이 실현된다. 트렌드 리포트의 A+C 시너지 포인트를 직접 구현한다.
- **구체 설계**:
  1. `SecurityAwareFluxionRestoreScheduler` 모듈 (`src/scheduler/security_fluxion_restore.py`)
  2. 세그먼트 복원 시 플래그 확인: `is_flagged(segment_id)` 조회.
     - 미플래그(공유 가능): `FluxionSparseRestoreCodec.restore(segment_id, budget_ratio=0.05~0.2)` → 스파스 복원 (처리량 최적)
     - 플래그(격리 필요): 사용자 로컬 캐시에서 전량 복원 (정확도 보장, 사이드채널 방지)
  3. 출력-인식 예산 동적 조정: 플래그율이 낮을수록(보안 안전한 환경) Fluxion 예산을 높여 스파스 복원 효율 증가. 플래그율이 높을수록(위협 환경) 전량 복원 비율 증가.
  4. TemporalTieredKVManager(05-10 기구현) DRAM 계층에서의 미플래그 세그먼트에 FluxionSparseRestoreCodec 우선 적용.
- **예상 복합 효과**: 처리량 +1.5~3× (Fluxion 미플래그 세그먼트 스파스 복원), 보안 격리 달성 (플래그 세그먼트 전량 복원), 메모리 효율 유지, accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: CacheSolidarity보안인식비연속세그먼트공유정책 (A+B)
- **조합 근거**: SecurityAwareSegmentIsolationScheduler(A-1)의 보안 플래그 메커니즘이 Activity B 비연속 세그먼트 크로스-요청 공유 정책의 게이트로 작동한다. 미플래그 세그먼트는 TriangleInequalitySegmentIndex(05-09 기구현)를 통해 크로스-사용자 공유 허용, 플래그 세그먼트는 사용자별 독립 캐싱으로 완전 격리하는 "보안-인식 비연속 KV 공유 정책"이 멀티테넌트 환경에서 Activity B의 실용 배포 경로를 형성한다. 트렌드 리포트의 A+B 시너지 포인트를 직접 구현한다.
- **구체 설계**:

  2. 세그먼트 히트 판단 시 플래그 확인: `is_flagged(segment_id)` → True면 현재 요청 사용자의 로컬 캐시에서만 검색. False면 전역 세그먼트 인덱스에서 크로스-사용자 검색.

  4. 플래그 → 미플래그 전환 시: 격리 캐시의 세그먼트를 FibQuant 압축 후 전역 공유 캐시로 이동 → 크로스-사용자 공유 시작.
- **예상 복합 효과**: 처리량 +15~25% (미플래그 세그먼트 크로스-사용자 공유로 TTFT 단축), 비연속 히트율 +10~20%p (전역 공유 풀 확장), 멀티테넌트 APC 사이드채널 위험 제거, accuracy delta ±0.3% 이내 (격리 세그먼트 전량 보존)
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 Cross-4: 멀티테넌트비연속세그먼트완전통합파이프라인 (A+B+C)

- **구체 설계**:

  2. 미플래그 세그먼트: FibQuantVQSegmentCache에서 랜덤-접근 검색 → FluxionSparseRestoreCodec으로 필요 KV만 CPU→GPU 스파스 전송 → ABSparseHeadBlockProfiler의 헤드별 최적 블록 크기로 스파스 어텐션 계산

- **예상 복합 효과**: 처리량 +30~50%, 메모리 −75~90% (FibQuant 10× + Fluxion 스파스 복원), 비연속 히트율 +20~35%p, 멀티테넌트 APC 사이드채널 완전 차단, accuracy delta ±0.5% 이내

## 이번 사이클 최우선 구현 타겟

**Cross-1 (FibQuantPositionFreeSegmentCache, B+C)** 과 **C-1 (FibQuantVQCodec)** 을 최우선 구현 타겟으로 권장한다.

이유:
1. C-1(FibQuantVQCodec)은 구형-베타 소스 수학적 최적화 기반으로 10× 이상 고압축 영역에서 기구현된 RSimVQCodec 대비 차별화된 성능을 달성하며, 보정 데이터 1회로 즉시 배포 가능하다. Activity C 메모리 목표 −30%를 10× 이상 초과하는 가장 강력한 신규 C 기법이다.
2. B-1(FibQuantVQSegmentCache)은 C-1 FibQuantVQCodec을 세그먼트 단위로 적용한 자연스러운 확장이며, 랜덤-접근 복원이 비연속 재사용과 완벽히 부합한다. 동일 메모리에 10× 더 많은 세그먼트를 보존해 비연속 히트율 목표(≥30%)에 직접 기여한다.
3. Cross-1은 B-1과 C-1의 통합 + position-decoupled 저장(기구현 원리 재활용)으로 구현 재사용률이 높으며, B+C 복합 효과(메모리 −80~90% + 비연속 히트율 +20~35%p)를 단일 사이클에서 검증 가능하다.
4. C-2(ABSparse)는 vLLM paged KV 캐시 호환 구현이 6단계 vLLM 이식에 즉시 활용 가능하므로 이후 사이클의 vLLM 이식 후보로 우선 지정.
5. A-1(CacheSolidarity)과 C-3(Fluxion)은 구현 난이도 medium이지만 기구현 컴포넌트(TriangleInequalitySegmentIndex, TemporalTieredKVManager)와의 통합이 필요하므로 Cross-1 완성 후 순차 구현 권장.

## planner에게 전달할 요청사항
- **C-1(FibQuantVQCodec) 구현 시**: `src/cache/fibquant_vq_codec.py` 신규 생성. 보정 단계: `torch.linalg.norm` 기반 구형 정규화 → 베타 분포 피팅(`scipy.stats.beta.fit`) → 베타-분위수 반경 격자 → 피보나치 방향 집합(d=128 기준 N_dir=512). Lloyd-Max 정제: `sklearn.cluster.KMeans` 또는 직접 구현(재시작 R=10, 반복 K=5). 저장: `(radial_code_bits, direction_code_bits)` YAML 외부화. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 압축률(4×/10×/20×) 별 코사인 유사도 곡선. 기구현 RSimVQCodec과 압축률-정확도 비교 포함.
- **B-1(FibQuantVQSegmentCache) 구현 시**: `src/cache/fibquant_vq_segment_cache.py` 신규 생성. `CacheStore` 인터페이스 상속. 세그먼트별 독립 인코딩 인터페이스: `encode_segment(segment_kv, segment_id)` / `decode_segment(segment_id)`. content-hash 키(위치 무관) 유지. `compression_target` 파라미터(5×/10×/20×) YAML 외부화. **측정 필수**: 동일 메모리 예산에서 FP16 세그먼트 캐시 vs FibQuant 세그먼트 캐시의 보존 세그먼트 수 + 비연속 히트율 비교.
- **Cross-1(FibQuantPositionFreeSegmentCache) 구현 시**: `src/cache/fibquant_position_free_segment.py` 신규 생성. B-1 FibQuantVQSegmentCache + pre-RoPE 저장(`store_pre_rope()`) + RoPE 재적용(`apply_rope_at_position()`) 통합. TriangleInequalitySegmentIndex(05-09 기구현)와 인터페이스 연결. **accuracy-preserving 검증 필수**: pre-RoPE + FibQuant 압축 후 복원 KV의 어텐션 출력 코사인 유사도 측정 + WikiText-2 perplexity ±1%.
- **C-2(ABSparseHeadBlockProfiler) 구현 시**: `src/cache/ab_sparse_head_block.py` 신규 생성. 보정 스크립트: `experiments/run_ab_sparse_calibration.py`. 블록 크기 후보 `B = [16, 32, 64, 128]`. 결과 `configs/ab_sparse_head_config_layer_{l}.json` 저장. paged KV 캐시 호환: `paged_kv_compatible=True` 옵션. **vLLM 이식 고려**: paged KV 페이지 매핑 메커니즘 주석으로 명시. **accuracy-preserving 검증 필수**: 헤드별 블록 크기 ON/OFF 비교 + WikiText-2 perplexity ±1% + LongBench.
- **C-3(FluxionSparseRestoreCodec) 구현 시**: `src/cache/fluxion_sparse_restore.py` 신규 생성. 보정: 헤드별 출력 편차 `σ²_h` + 상대적 기여도 `w_h` 측정 (`configs/fluxion_head_config_layer_{l}.json` 저장). 크로스-디바이스 조율: `torch.cuda.Stream` 이중 스트림. TemporalTieredKVManager(05-10 기구현) DRAM 계층 복원 경로에 drop-in 교체. **accuracy-preserving 검증 필수**: CPU 스파스 복원 전후 perplexity ±1% + LongBench 40개 태스크.
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (목표 +20% 이상)
  - Non-Contiguous Cache Hit Rate (목표 30% 이상 — FibQuant 10× 압축으로 동일 메모리에 10× 더 많은 세그먼트 보존)
  - KV Cache Memory Reduction (목표 −30% 이상, C-1 FibQuantVQCodec 구현 시 −80~90% 목표)
  - Compression Accuracy Delta (필수: ±1% 이내 — 4×/10×/20× 압축률 별 측정)
  - TTFT p50 (목표 Scheduling overhead +5% 이내)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 아이디어는 `src/metrics/` 내 perplexity 측정 코드를 포함하고, 압축 전후 perplexity를 `results/<exp-name>/metrics.json`에 함께 기록해야 한다. C-1(FibQuantVQCodec)은 압축률(4×/10×/20×) 별 코사인 유사도 + perplexity 곡선을 필수 측정.
