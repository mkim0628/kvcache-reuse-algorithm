## Activity A 아이디어

### 아이디어 A-1: RadixGuidedFeatherHomogeneousBatchScheduler (Radix-Feather 동질 배치 스케줄러)
- **가설**: 배치 크기와 프리픽스 동질성 사이에 근본 트레이드오프가 존재한다는 Feather(2605.06046)의 발견을, 기구현된 `src/cache/radix.py`의 Radix 트리에서 실시간으로 추출하는 공유 프리픽스 동질성 신호와 결합하면, RL로 학습된 배치 중단 정책이 현재 Radix 트리 상태에 기반해 동질성-크기 최적 균형점을 동적으로 결정하는 "Radix-guided Feather 스케줄러"를 구성할 수 있으며, 정적 프리픽스 우선 배치(기존 기법) 대비 처리량을 15~25% 향상시킬 수 있다.
- **접근 방법**:
  1. `RadixFeatherBatchScheduler` 클래스 (`src/scheduler/radix_feather_batch.py`): `BaseScheduler` 상속
  2. 배치 중단 신호 추출: 현재 대기 요청 큐에서 각 요청의 최장 공유 프리픽스 길이를 `src/cache/radix.py`의 Radix 트리 순회로 O(prefix_len) 시간에 추출한다. 동질성 지표 계산: `homogeneity_score = (공유 프리픽스 총 토큰 수) / (현재 배치 총 입력 토큰 수)`.
  3. Chunked Hash Tree 구성: 공유 프리픽스 체인을 chunk 단위(기본 256 토큰)로 해시해 빠른 동질성 집계에 사용. Feather 원 논문의 C++ Chunked Hash Tree를 Python으로 단순화 구현 (`src/cache/chunked_hash_tree.py`).
  4. RL 기반 배치 중단 정책 학습:
     - 상태 공간: `(batch_size, homogeneity_score, avg_shared_prefix_len, queue_depth, current_throughput_ema)`
     - 행동 공간: `{STOP_BATCH, ADD_REQUEST}` (이진 결정)
     - 보상 신호: (배치 완료 후 측정된 토큰/초) − (이전 배치 평균 토큰/초)
     - 알고리즘: PPO-Clip (간단 2층 MLP 정책 네트워크, 경량화)
     - 훈련: 시뮬레이션 트레이스(배포 전 1회 학습, 약 5K 배치 에피소드)
  5. 기존 정적 prefix-aware batching(최장 공유 프리픽스 우선)과의 비교 베이스라인 통합: `mode` 파라미터로 정적/RL 선택 가능.
  6. 이전 A 기법과의 차별점: 이전 15개 사이클의 모든 A 기법(PPDAppendPrefillRouter, ContiguousChunkIO, WAITThresholdBatcher, CacheFlow3DParallel, StabilityBounded, PBKV, CacheSolidarity 등)은 모두 정적 휴리스틱 또는 이론 최적 레이턴시 공식에 기반했다. RadixFeatherBatchScheduler는 배치 구성 자체를 RL로 학습하는 최초 접근이다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 Radix 트리의 공유 프리픽스 정보를 노드 간 분산 집계 시 가벼운 메타데이터(공유 프리픽스 해시 + 길이)만 전파. RL 정책 파라미터는 중앙 컨트롤러에서 주기적으로 각 노드에 동기화(매 1K 배치).
- **예상 효과**: 처리량 +15~25% (동질 소배치의 TBT 단축 효과), Scheduling overhead TTFT +4% 이내 (RL 정책 추론이 2층 MLP로 < 0.1ms), 비연속 세그먼트 히트율 +8~15%p (동질 배치로 세그먼트 공유 증가)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-2: MooncakeRelayClusterKVRoutingScheduler (클러스터-인식 크로스-에이전트 KV 라우팅 스케줄러)
- **가설**: vLLM x Mooncake Store(2026-05-06)의 클러스터-와이드 분산 KV 풀과 RelayCaching(2603.13289)의 크로스-에이전트 KV 재사용을 결합하면, 이전 에이전트가 Mooncake Store에 저장한 디코딩 KV를 다음 에이전트가 RelayCaching 레이어 범위 프로파일러로 직접 조회·재사용하는 "분산-릴레이 크로스-에이전트 KV 라우팅"이 가능하며, 에이전틱 워크로드에서 TTFT를 4~10× 단축하고 처리량을 3~5× 향상시킬 수 있다.
- **접근 방법**:
  1. `MooncakeRelayKVRoutingScheduler` 클래스 (`src/scheduler/mooncake_relay_routing.py`): `BaseScheduler` 상속
  2. KV 메타데이터 레지스트리: 에이전트가 생성한 디코딩 KV를 분산 KV 풀에 저장할 때 `{agent_id, task_context_hash, layer_deviation_profile}` 메타데이터를 레지스트리에 등록. task_context_hash는 에이전트 입력 컨텍스트의 SimHash(64비트) 기반 내용 유사도 키.
  3. RelayCaching 레이어 범위 프로파일러 통합: 다음 에이전트가 요청 시작 시 프로파일러가 후보 KV(이전 에이전트 저장)와 현재 컨텍스트의 레이어별 KV 일관성을 추정. U자형 편차 프로파일을 기반으로 재사용 가능 레이어 범위 [l_start, l_end] 결정.
  4. 라우팅 결정: (a) 분산 풀에 유사 컨텍스트 KV 존재 시 → RDMA로 KV 로드 + RelayCaching 선택적 레이어 갱신 → 프리필 80%+ 스킵. (b) 분산 풀 미스 시 → 일반 프리필 + KV 풀 등록.
  5. 비연속 세그먼트 인덱스(`src/cache/radix.py`)와 통합: 분산 풀 KV를 세그먼트 단위로 분할해 Radix 트리에 인덱싱, 에이전트 간 세그먼트 수준 재사용 가능.
  6. 이전 A 기법과의 차별점: PBKV(05-13)가 단일 인스턴스 내 예측 기반 KV 보존에 집중했다면, 이 기법은 클러스터 분산 KV 풀 + RelayCaching 에이전트 간 KV 직접 조회를 결합하는 클러스터 수준 에이전틱 라우팅이다.
- **대상 환경**: 멀티 노드 (클러스터-와이드 분산 KV 풀 필수)
- **멀티 노드 고려사항**: RDMA를 통한 KV 블록 전송 (GPU HBM 직접 → 직접). KV 풀 마스터가 `{agent_id, context_hash, node_id, layer_range}` 인덱스를 관리. 네트워크 토폴로지(InfiniBand vs NVLink) 인식: 동일 랙 내 KV 우선 참조로 전송 지연 최소화.
- **예상 효과**: 처리량 +3~5× (에이전틱 워크로드 기준, Mooncake Store 실증 기반), TTFT −80~95% (RelayCaching 4.7× + Mooncake RDMA 결합), Scheduling overhead TTFT +3% 이내
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 B-1: RelayUShapeLayerSelectiveSegmentCache (RelayCaching 기반 레이어-선택적 세그먼트 재사용)
- **가설**: RelayCaching(2603.13289)이 발견한 "KV 편차가 U자형으로 국소화된다" — 첫 레이어와 마지막 레이어에서 편차 크고 중간 레이어에서 작다 — 는 이론적 관찰을, 비연속 KV 세그먼트 재사용에 적용하면, 세그먼트가 byte-identical하지 않을 때도 "편차 작은 중간 레이어 세그먼트를 재사용하고 편차 큰 경계 레이어만 재계산"하는 레이어-선택적 그레이디드 세그먼트 재사용이 가능하며, 기존 all-or-nothing 세그먼트 재사용 대비 비연속 히트율을 25~40% 향상시킬 수 있다.
- **접근 방법**:
  1. `RelayUShapeLayerSelectiveSegmentCache` 클래스 (`src/cache/relay_ulayer_segment.py`): `CacheStore` 인터페이스 상속
  2. 레이어 범위 프로파일러 (오프라인 1회):
     - 보정 데이터(100~200 세그먼트 쌍)에서 "유사하지만 비동일" 세그먼트 쌍의 레이어별 KV 코사인 유사도를 측정.
     - 임계값 τ_layer(기본 0.95) 이상인 레이어 집합 `R_reuse` 결정. U자형이므로 중간 레이어 70~80%가 R_reuse에 속할 것으로 예상.
     - 레이어 범위 `[l_start, l_end]`를 `configs/relay_ulayer_profile.yaml`에 저장.
  3. 세그먼트 저장 형식 확장: 기존 content-hash 기반 키에 `layer_reuse_mask` (비트마스크, 각 레이어별 재사용 가능 여부) 추가 저장.
  4. 재사용 시 레이어별 선택:
     - `layer_reuse_mask`에서 R_reuse 레이어의 KV를 캐시에서 로드.
     - 나머지 레이어(경계 레이어)는 on-demand 재계산 (입력 토큰 주어짐).
     - 재계산량: 전체의 20~30% (경계 레이어만). 순 절감: 70~80% 레이어 재사용.
  5. 기존 `src/cache/segmented.py`의 `SegmentedHashCache`를 베이스로, 레이어 마스크 저장 및 부분 로드 인터페이스(`load_layers(segment_id, layer_mask)`) 추가.
  6. 이전 B 기법과의 차별점: 기존 모든 B 기법(RoPEReencodingNonContiguousCache, KVFoldAccumulativeRadixCache, HalfspaceRangeSearchKVIndex 등)은 세그먼트 전체를 재사용하거나 전체를 재계산하는 이진 결정 구조였다. RelayUShapeLayerSelectiveSegmentCache는 레이어별 선택적 부분 재사용이라는 연속 결정 구조를 최초로 도입한다.
- **예상 효과**: 비연속 히트율 +25~40% (비동일 세그먼트에서도 중간 레이어 70~80% 재사용), TTFT −40~55% (70~80% 레이어 재계산 제거), 메모리 추가 오버헤드 최소 (layer_reuse_mask = 레이어 수 / 8 바이트 추가)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 B-2: ICaRusLogicalEncoderSharedMultiModelRadixCache (ICaRus 크로스-모델 KV 공유 Radix 캐시)
- **가설**: ICaRus(2603.13281)의 핵심 발견 — 논리적 인코더(KV 생성)와 논리적 디코더(토큰 예측)를 분리해 동결된 인코더 부분의 KV를 이종 모델들이 공유할 수 있다 — 을 `src/cache/radix.py`의 Radix 트리 인덱스에 모델 ID 차원을 추가해 구현하면, 동일 세그먼트를 공유하는 N개 전문화 모델이 KV 캐시를 1/N으로 절감하는 "멀티-모델 Radix 캐시"를 구성할 수 있다.
- **접근 방법**:
  1. `MultiModelSharedRadixCache` 클래스 (`src/cache/multi_model_radix.py`): `CacheStore` 인터페이스 상속
  2. Radix 트리 인덱스 확장: 기존 `(token_ids) → KV` 매핑을 `(token_ids, encoder_model_id) → shared_KV` + `(token_ids, task_adapter_id) → adapter_delta_KV`로 확장.
     - `encoder_model_id`: 동결된 논리적 인코더(공유 KV 생성 레이어 집합) 식별자. 동일 베이스 모델에서 파생된 어댑터들은 동일 encoder_model_id.
     - `task_adapter_id`: 각 태스크-특화 어댑터(논리적 디코더) 식별자.
     - 공유 KV는 encoder 레이어 집합(전체 레이어의 60~80%)에서 생성된 KV를 저장. 어댑터별 delta_KV는 나머지 레이어에서 발생하는 편차만 저장.
  3. 공유 비율 결정 (오프라인 프로파일링): 베이스 모델과 각 어댑터 모델 사이의 레이어별 KV 코사인 유사도를 측정해 τ_share(기본 0.97) 이상 레이어를 공유 레이어로 분류. `configs/icarus_shared_layers.yaml`에 저장.
  4. 어댑터 저장: LoRA 스타일 어댑터가 공유 KV에 delta를 추가하는 방식으로 태스크별 편차 저장. 편차 크기가 공유 KV의 5~15%이면 delta_KV 압축 저장(FP8).
  5. 이전 B 기법과의 차별점: 이전 모든 B 기법은 단일 모델 내 비연속 세그먼트 재사용에 집중했다. ICaRusLogicalEncoderSharedMultiModelRadixCache는 이종 다중 모델 간 KV 공유라는 새로운 차원을 최초로 Activity B에 도입한다. 결과적으로 N개 모델이 참여할 때 KV 메모리가 1/N으로 절감된다.
- **예상 효과**: 비연속 세그먼트 메모리 −(N-1)/N × 60~80% (N 모델 공유 시), 처리량 +2~4× (다중 모델 에이전틱 워크로드), TTFT −60~75% (공유 KV 재사용으로 프리필 절감), accuracy delta ±0.5% (각 어댑터가 태스크별 delta_KV로 보정)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-3: QKVShareCacheCardNonContiguousExchangeProtocol (CacheCard 자기-완결 비연속 세그먼트 교환 표준)
- **가설**: QKVShare(2605.03884)의 CacheCard — 양자화 메타데이터를 캐시 페이로드에 패키징해 수신 에이전트가 의존성 없이 즉시 주입 가능한 자기-완결 표현 — 를 Activity B의 비연속 세그먼트를 에이전트/노드 간 교환 가능한 표준 형식으로 일반화하면, 분산 환경에서 비연속 세그먼트를 외부 의존성 없이 전달·재사용하는 "비연속 세그먼트 CacheCard 표준"이 가능하다.
- **접근 방법**:
  1. `NonContiguousCacheCard` 데이터클래스 (`src/cache/cache_card.py`):
     ```python
     @dataclass
     class NonContiguousCacheCard:
         segment_id: str           # content-hash 기반 세그먼트 ID
         kv_data: bytes            # 양자화된 KV 바이트 (INT8/FP8)
         quant_metadata: dict      # {scale, zero_point, dtype} per layer
         position_range: Tuple[int, int]  # 원본 위치 범위
         pre_rope: bool            # RoPE 적용 전 저장 여부
         layer_reuse_mask: bytes   # 레이어별 재사용 가능 비트마스크 (B-1 호환)
         model_encoder_id: str     # 공유 인코더 모델 ID (B-2 호환)
     ```
  2. CacheCard 직렬화/역직렬화: `serialize_to_bytes()` / `from_bytes()` — 표준 Protocol Buffers 또는 MessagePack 사용, 의존성 없이 자기-완결.
  3. HuggingFace 호환 캐시 주입: `inject_to_model_cache(model, cache_card)` — 기존 모델 코드 변경 없이 CacheCard의 KV 데이터를 모델 KV 캐시에 직접 삽입.
  4. 토큰-레벨 혼합 정밀도 할당: QKVShare 원 논문의 토큰 중요도 기반 차등 비트폭 할당을 CacheCard 생성 시 적용. 중요 토큰 INT8 보존, 덜 중요한 토큰 INT4 저장.
  5. 기존 `src/cache/segmented.py`와 통합: `SegmentedHashCache.export_as_cache_card(segment_id)` / `import_from_cache_card(cache_card)` 인터페이스 추가.
  6. 이전 B 기법과의 차별점: 기존 모든 B 기법은 단일 시스템 내부의 세그먼트 관리에 초점을 맞춘 반면, QKVShareCacheCardProtocol은 에이전트/노드/기기 경계를 넘어 비연속 세그먼트를 교환 가능한 표준화된 자기-완결 단위로 패키징하는 인터페이스 표준을 도입한다.
- **예상 효과**: 비연속 히트율 에지/분산 환경에서 +20~35% (에이전트 간 재-프리필 없이 세그먼트 전달), TTFT −60~80% (에지 온-디바이스 기준, QKVShare 실증 기반), 세그먼트 전달 크기 −50~75% (혼합 정밀도 양자화)
- **구현 난이도**: medium
- **우선순위**: 3

## Activity C 아이디어

### 아이디어 C-1: LookaheadKVFutureAwareLoRAEvictionCodec (드래프트-프리 미래-인식 LoRA 퇴거 코덱)
- **가설**: KV 캐시 퇴거 정확도를 높이기 위해 LookaheadKV(2603.10899, ICLR 2026)의 학습 가능한 룩어헤드 토큰 + 경량 LoRA 어댑터가 드래프트 생성 없이 미래 응답의 어텐션 패턴을 직접 예측하면, 기존 현재-인식 퇴거(H2O, SnapKV, LaProx) 대비 동등한 런타임 오버헤드로 더 정확한 퇴거 결정을 내려 메모리 −70~85%에서 accuracy delta ±1% 이내를 달성할 수 있다.
- **접근 방법**:
  1. `LookaheadKVEvictionCodec` 클래스 (`src/cache/lookahead_kv_eviction.py`): `CompressionCodec` 인터페이스 구현
  2. 룩어헤드 모듈 구성:
     - 각 어텐션 헤드당 학습 가능한 룩어헤드 토큰 집합 `L_h ∈ R^{n_la × d_k}` (n_la = 3~5, 파라미터 수 매우 적음).
     - 각 레이어마다 경량 LoRA 어댑터 (rank=8~16): 룩어헤드 쿼리가 실제 모델 응답의 어텐션 패턴과 정렬되도록 학습.
     - 훈련: 소규모 보정 데이터(500~1000 샘플)에서 룩어헤드 토큰 + LoRA 어댑터를 5~10 에포크 파인튜닝. 기존 모델 가중치 동결. 훈련 비용 < 1 GPU-hour (LookaheadKV 원 논문 기반).
  3. 퇴거 파이프라인 (프리필 단계):
     - 입력 시퀀스 끝에 룩어헤드 토큰 추가 → 어텐션 계산 실행.
     - 룩어헤드 토큰의 각 입력 토큰에 대한 어텐션 점수가 미래 응답이 해당 토큰을 얼마나 참조할지를 예측.
     - 퇴거 점수 = `max_over_heads(lookahead_attention_score_i)`. 하위 k%를 퇴거.
     - 퇴거 비율 `eviction_ratio`: YAML 파라미터화 (기본 0.7, 즉 70% 퇴거).
  4. 기존 현재-인식 퇴거와의 앙상블 옵션: `blend_ratio` 파라미터로 LookaheadKV 점수 + LaProx 출력-인식 점수 혼합.
  5. vLLM 이식 친화성: 기존 모델 가중치 변경 없이 LoRA 어댑터만 추가하므로 vLLM의 `lora_modules` 인터페이스 활용 가능.
  6. 이전 C 기법과의 차별점: LaProxOutputAwareLayerEviction(05-11)이 현재 어텐션의 출력 행렬곱 근사 오차로 중요도를 추정하고, LKVEndToEndBudgetSelectorCodec(05-11)이 E2E 학습으로 버짓-선택을 공동 최적화하는 반면, LookaheadKVEvictionCodec은 미래 응답의 어텐션 패턴을 직접 예측하는 완전히 다른 정보 소스를 사용한다. "현재 패턴 → 미래 패턴 예측"으로의 패러다임 전환이다.
- **예상 효과**: 메모리 −70~85% (퇴거 비율 0.7~0.85), accuracy delta ±0.5% (4× 압축 기준, LookaheadKV ICLR 2026 실증 기반), 퇴거 비용 기존 draft-based 방법 대비 14.5× 절감
- **accuracy-preserving 근거**: LookaheadKV(2603.10899, ICLR 2026, Samsung AI)가 다양한 태스크에서 동등 오버헤드 기준 기존 heuristic(H2O, SnapKV) 대비 더 높은 퇴거 품질을 실증했다. 룩어헤드 토큰이 미래 응답이 참조할 토큰을 직접 예측하므로 중요 토큰 누락을 현재-인식 방법보다 근본적으로 줄인다. LoRA 어댑터의 낮은 랭크(8~16)가 과적합을 방지해 보정 데이터 외 분포에서도 일반화된다. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 퇴거 비율(0.5/0.7/0.85)별 정확도 곡선.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: BatchLevelSharedLookaheadKVEviction (배치-레벨 공유 룩어헤드 퇴거)
- **가설**: RadixFeatherBatchScheduler(A-1)의 동질 배치 내 요청들이 유사한 프리픽스를 공유하므로 LookaheadKV 룩어헤드 토큰이 예측하는 미래 어텐션 패턴도 유사하다. 배치 내 N개 요청의 룩어헤드 어텐션 점수를 평균해 공유 퇴거 마스크를 생성하면, 퇴거 오버헤드를 N개 → 1회로 줄이면서 집단 미래-인식 신호로 개별 예측보다 노이즈가 감소한다.
- **접근 방법**:
  1. `BatchLevelSharedLookaheadEviction` 클래스 (`src/cache/batch_lookahead_eviction.py`): `LookaheadKVEvictionCodec`(C-1) 상속
  2. 배치 공유 퇴거 마스크:
     - 동질 배치 내 각 요청의 룩어헤드 어텐션 점수를 토큰 위치 기준으로 정렬.
     - 공유 프리픽스 부분: 배치 내 모든 요청의 점수를 평균 → 공유 퇴거 마스크.
     - 비공유 접미사 부분: 요청별 개별 룩어헤드 점수 사용.
  3. RadixFeatherBatchScheduler(A-1) 통합: 배치가 "동질 모드"로 구성된 경우에만 배치-레벨 공유 마스크 활성화. 이종 배치는 개별 LookaheadKV 퇴거로 fallback.
  4. 점수 집계 오버헤드: 배치 크기 B 기준 O(B × n_la × d_k) 추가 연산. n_la=5, d_k=128, B=8 → 5120 FP16 연산으로 무시 가능.
  5. 이전 C 기법과의 차별점: 기존 모든 C 기법은 요청별 독립 퇴거 결정을 내렸다. BatchLevelSharedLookaheadEviction은 배치를 단위로 집단 미래-인식 퇴거를 수행하는 최초 배치-레벨 퇴거 기법이다. A+C 시너지가 직접 구현된다.
- **예상 효과**: 퇴거 비용 추가 −N× (배치 크기 N 기준), 집단 평균으로 퇴거 마스크 노이즈 −30~50%, accuracy delta ±0.3% (집단 예측 정확도 향상)
- **accuracy-preserving 근거**: 동질 배치 내 요청들이 유사한 미래 어텐션 패턴을 가지므로 집단 평균 점수가 개별 점수보다 분산이 낮아 중요 토큰 보존 확률이 높아진다. 수학적으로 E[평균 점수 오차] ≤ E[개별 점수 오차] / √B (독립 가정 하 중심극한정리). 검증: 동질 배치(B=4/8) 대비 개별 퇴거 accuracy delta 비교.
- **구현 난이도**: low
- **우선순위**: 2

### 아이디어 Cross-1: LookaheadKVRelayAgentSegmentPipeline (B+C) — 중요도-가중 크로스-에이전트 KV 재사용
- **조합 근거**: RelayUShapeLayerSelectiveSegmentCache(B-1)의 레이어 선택적 재사용 결정과 LookaheadKVEvictionCodec(C-1)의 미래-인식 중요도 점수를 앙상블하면, "이전 에이전트 KV 중 미래 응답 어텐션 패턴이 높다고 예측된 토큰의 KV만 선택적으로 재사용하고 나머지는 재계산"하는 이중-필터 크로스-에이전트 KV 재사용 전략이 가능하다. RelayCaching의 편차 기반 필터(레이어 선택) + LookaheadKV의 미래-인식 필터(토큰 선택)를 결합하는 새로운 B+C 조합이다.

  3. 메모리 효율: 재사용 결정된 KV만 캐시에 보존. 전체 이전 에이전트 KV 대비 20~30%만 유지.

- **예상 복합 효과**: 처리량 +25~40%, TTFT −60~75% (크로스-에이전트 재사용 + 미래-인식 선택), 메모리 −70~80% (20~30% KV만 보존), accuracy delta ±0.4% (이중 필터로 중요 토큰 보존)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: FeatherICaRusMultiModelHomogeneousBatchScheduler (A+B) — ICaRus-인식 Feather 스케줄러
- **조합 근거**: RadixFeatherBatchScheduler(A-1)의 프리픽스 동질 배치 구성이 ICaRusLogicalEncoderSharedMultiModelRadixCache(B-2)의 크로스-모델 KV 공유와 결합되면, "동일 공유 KV를 참조하는 모델들(ICaRus)을 Feather가 동질 배치로 묶어 스케줄링"하는 A+B 시너지가 성립한다. 동질 배치의 모든 요청이 공유 인코더 KV를 참조하므로 배치 내 KV 재사용률이 최대화된다.
- **구체 설계**:
  1. `ICaRusAwareFeatherBatchScheduler` 모듈 (`src/scheduler/icarus_feather_scheduler.py`)
  2. 배치 동질성 신호 확장: 기존 동질성(공유 토큰 프리픽스 기반)에 "공유 encoder_model_id 일치 여부"를 추가. 동일 encoder_model_id 요청들은 동질 가중치 1.5× 적용.

  4. MultiModelSharedRadixCache(B-2)의 공유 KV 히트율을 RL 보상 보정 신호로 활용: 배치 후 공유 KV 히트율이 높으면 양의 보상 추가.
  5. 이전 크로스 아이디어와의 차별점: 05-13의 Cross-1(PBKVKVFoldAgentChunkPreCaching)이 예측 기반 청크 선점(A+B)이었다면, 이 기법은 RL 배치 동질성 + 크로스-모델 KV 공유의 A+B 결합이다.
- **예상 복합 효과**: 처리량 +30~50% (Feather 동질 배치 + ICaRus N×KV 공유), 메모리 −(N-1)/N × 60~80% (N 모델 공유), 비연속 히트율 +20~35%p, accuracy delta ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: DistributedLookaheadRelayAgenticFullPipeline (A+B+C) — 에이전틱 멀티모델 완전 통합 파이프라인
- **조합 근거**: 트렌드 리포트의 A+B+C 통합 시너지를 오늘 사이클의 신규 기법으로 직접 구현한다. RadixFeatherBatchScheduler(A-1)가 동질 배치를 구성하고 → ICaRusLogicalEncoderSharedMultiModelRadixCache(B-2)가 크로스-모델 KV 공유 제공 → RelayUShapeLayerSelectiveSegmentCache(B-1)가 에이전트 간 레이어-선택적 재사용 수행 → LookaheadKVEvictionCodec(C-1)이 미래-인식 퇴거로 불필요 KV 제거 → MooncakeRelayKVRoutingScheduler(A-2)가 클러스터 분산 KV 풀에 보존.
- **구체 설계**:
  1. `AgenticMultiModelLookaheadPipeline` 모듈 (`src/engine/agentic_lookahead_pipeline.py`)
  2. 처리 순서:
     - 배치 구성: RadixFeatherBatchScheduler로 동질 배치 (encoder_id 일치 요청 우선)
     - 공유 KV 조회: MultiModelSharedRadixCache에서 공유 인코더 KV 로드
     - 에이전트 KV 릴레이: RelayUShapeLayerSelectiveSegmentCache로 이전 에이전트 KV 레이어 선택 재사용
     - 미래-인식 퇴거: BatchLevelSharedLookaheadEviction으로 배치-레벨 공유 퇴거 마스크 생성 + 불필요 KV 퇴거
     - 분산 KV 보존: 보존된 KV를 분산 풀(Mooncake Store 스타일)에 등록 (다음 에이전트 참조용)
  3. 모니터링: 각 단계의 KV 재사용률, 퇴거율, TTFT, 메모리를 `results/<exp-name>/metrics.json`에 단계별 기록.
- **예상 복합 효과**: 처리량 +50~80%, 메모리 −75~90% (ICaRus 공유 + LookaheadKV 퇴거), 비연속 히트율 +35~50%p, TTFT −70~90%, accuracy delta ±0.6% 이내
- **구현 난이도**: high
- **우선순위**: 4

## 이번 사이클 최우선 구현 타겟

**C-1 (LookaheadKVFutureAwareLoRAEvictionCodec)** 과 **B-1 (RelayUShapeLayerSelectiveSegmentCache)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. C-1(LookaheadKVEvictionCodec)은 ICLR 2026 채택 논문 기반으로 accuracy-preserving 근거가 가장 강력하며, 기존 모델 가중치 변경 없이 LoRA 어댑터 추가만으로 배포 가능하다. 퇴거 비용 14.5× 절감은 Activity C의 처리량 목표 +20%와 메모리 목표 −30%를 단독으로 달성 가능한 수준이다.
2. B-1(RelayUShapeLayerSelectiveSegmentCache)은 훈련-프리(보정 데이터만 필요)로 즉시 `src/cache/`에 통합 가능하며, 비동일 세그먼트에서도 중간 레이어 70~80%를 재사용하는 새 패러다임으로 비연속 히트율 목표(30% 이상)에 직접 기여한다.
3. Cross-1(LookaheadRelaySegmentCache)은 B-1과 C-1의 자연스러운 조합으로 구현 재사용률이 높으며, 단일 사이클에서 B+C 복합 효과를 검증할 수 있다.
4. A-1(RadixFeatherBatchScheduler)은 구현 난이도 medium이고 기존 radix.py를 활용하지만 RL 학습 루프 구성이 필요하므로 B-1/C-1 완성 후 순차 구현을 권장한다.
5. B-2(ICaRusMultiModelRadixCache)는 다중 모델 어댑터 설정이 필요해 단일 사이클에서 완전 검증이 어렵다. 이후 사이클로 미루되 Cross-2와 연계 구현 권장.

## planner에게 전달할 요청사항
- **C-1(LookaheadKVEvictionCodec) 구현 시**: `src/cache/lookahead_kv_eviction.py` 신규 생성. 룩어헤드 모듈: `n_la=5` 토큰, LoRA rank=8~16. 훈련 스크립트: `experiments/train_lookahead_lora.py` (보정 데이터 500~1000 샘플, ≤1 GPU-hour). 퇴거 비율 `eviction_ratio` YAML 외부화 (기본 0.7). **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 퇴거 비율(0.5/0.7/0.85)별 정확도 곡선. 기구현 LaProxOutputAwareLayerEviction(05-11)과 동일 설정에서 정확도 비교 포함. Samsung Research 공개 GitHub(SamsungLabs/LookaheadKV) 코드 참조.
- **B-1(RelayUShapeLayerSelectiveSegmentCache) 구현 시**: `src/cache/relay_ulayer_segment.py` 신규 생성. `CacheStore` 인터페이스 상속. 레이어 범위 프로파일러: `experiments/run_relay_layer_calibration.py`. 보정 데이터: 100~200 "유사하지만 비동일" 세그먼트 쌍. `layer_reuse_mask` 비트마스크 저장 방식 명시. 기존 `src/cache/segmented.py`에 `load_layers(segment_id, layer_mask)` 인터페이스 추가. **accuracy-preserving 검증 필수**: 레이어-선택 재사용 vs 전량 재사용 perplexity 비교 + 비연속 히트율 측정 (비동일 세그먼트 포함).
- **Cross-1(LookaheadRelaySegmentCache) 구현 시**: `src/cache/lookahead_relay_segment.py` 신규 생성. B-1 RelayUShapeLayerSelectiveSegmentCache + C-1 LookaheadKVEvictionCodec 순차 파이프라인. 토큰 선택 임계값 `τ_token=0.3` YAML 외부화. **accuracy-preserving 검증 필수**: 이중 필터(레이어 + 토큰) 적용 후 perplexity ±1% + 비연속 히트율 + 재사용 KV 비율 측정.
- **A-1(RadixFeatherBatchScheduler) 구현 시**: `src/scheduler/radix_feather_batch.py` 신규 생성. `BaseScheduler` 상속. `src/cache/chunked_hash_tree.py` 신규 생성 (공유 프리픽스 체인 추적). RL 정책: PPO-Clip, 2층 MLP (hidden=64). 시뮬레이션 트레이스 생성: `experiments/gen_batch_sim_traces.py`. **측정 필수**: 정적 prefix-aware 배치 vs RL Feather 배치 처리량 비교 + 배치 동질성 분포 측정.
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (목표 +20% 이상)
  - Non-Contiguous Cache Hit Rate (목표 30% 이상 — B-1 레이어-선택적 재사용으로 비동일 세그먼트 히트 포함)
  - KV Cache Memory Reduction (목표 −30% 이상 — C-1 LookaheadKV 퇴거로 −70~85% 목표)
  - Compression Accuracy Delta (필수: ±1% 이내 — 퇴거 비율 0.5/0.7/0.85 별 측정)
  - TTFT p50 (목표 Scheduling overhead +5% 이내)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 아이디어는 `src/metrics/` 내 perplexity 측정 코드를 포함하고, 압축/퇴거 전후 perplexity를 `results/<exp-name>/metrics.json`에 함께 기록해야 한다. C-1(LookaheadKVEvictionCodec)은 퇴거 비율별 perplexity + 코사인 유사도 곡선을 필수 측정. 기존 C 기법(LaProx, LKV)과의 동일 설정 정확도 비교 포함.
