## 변화 요약
- **이론적 스케줄링 보장 등장**: GSA가 상수 경쟁비(61.92)를 수학적으로 보장하는 non-clairvoyant 스케줄러를 제시해, 이전 사이클의 모든 Activity A 아이디어(경험적 히트율·부하 기반)와 본질적으로 다른 설계 공간을 열었다.
- **B+C 단일 구조 융합**: Self-Indexing KVCache(1-bit 부호 VQ)가 비연속 세그먼트 검색과 KV 압축을 하나의 1-bit 표현으로 통합한다. 이전 사이클의 B+C는 압축 코덱과 세그먼트 인덱스를 별도 모듈로 유지했다.
- **훈련-무료 비연속 재사용 등장**: RelayCaching이 학습 없이 디코딩 KV를 프리필에 재주입함으로써, 이전 사이클의 KV Packet adapter(MLP 학습 필요) 대비 다른 구현 경로를 제공한다.

## 이번 사이클 초점
**B+C** — Non-Contiguous KV Cache Reuse + KV Cache Compression (1-bit 융합 + 훈련-무료 재주입)

**선택 이유**:
1. 이전 두 사이클에서 Activity A(멀티노드, 경쟁비 스케줄러)를 중점 개발했으므로, 이번 사이클에서 B와 C의 신규 기법(Self-Indexing 1-bit 융합, RelayCaching 훈련-무료 재주입, CapKV 정보이론 퇴거)을 구현하면 Cross A+B+C 통합의 C/B 레이어가 강화된다.
2. Self-Indexing KVCache(B+C 융합)와 RelayCaching(새 B 경로)이 AAAI 2026·ICLR 2026 수준으로 검증되어 구현 리스크가 낮다.
3. CapKV의 레버리지 스코어 기반 퇴거가 accuracy-preserving 근거를 정보이론적으로 제공하므로 ±1% perplexity 제약 충족에 가장 탄탄한 이론적 기반을 갖춘다.

## Activity A 아이디어

### 아이디어 A-1: GSA 경쟁비 보장 스케줄러 (Competitive Non-Clairvoyant)
- **가설**: 응답 길이를 사전에 알 수 없는 환경에서 기하급수적 단계(geometric phase) 구조와 엇갈린 파이프라인(staggered pipeline)으로 KV 메모리 예산 위반 없이 TTFT를 최소화하면, 이전 사이클의 경험적 스케줄러(FlowKV 스타일) 대비 최악의 경우 보장이 강화된다.
- **접근 방법**:
  1. `CompetitiveScheduler` 클래스 (`src/scheduler/competitive_scheduler.py`): GSA(Geometric Staircase Algorithm) 구현
  2. 기하급수적 단계: 메모리 예산 M에 대해 k번째 단계에서 허용 동시 요청 수 = ⌊M / (2^k × avg_kv_size)⌋
  3. 엇갈린 파이프라인: 각 단계에서 요청 완료 전에 새 단계 시작, 두 단계 중복 실행으로 GPU 활용률 유지
  4. 현재 `CacheAwareScheduler` (기존 구현)와 동일 인터페이스(`BaseScheduler`) 상속, drop-in 교체 가능
  5. 경쟁비 측정 벤치마크: 시뮬레이션 최적 오프라인 알고리즘 대비 실제 TTFT 비율 측정
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 메모리 예산 M을 노드별 로컬 HBM 용량 + CXL/RDMA 원격 용량의 합으로 확장하면 6-계층 메모리 위계(Predictive Multi-Tier, 2604.26968)와 통합 가능
- **예상 효과**: 처리량 +15~20%, Scheduling overhead TTFT p50 +3~4% 이내 (경쟁비 보장 범위 내)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-2: 3D 병렬 복원 인식 배치 스케줄러 (CacheFlow 스타일)
- **가설**: KV 캐시 복원(restoration)을 토큰-레이어-GPU 3차원 병렬로 실행하고, "배치-인식 two-pointer" 방식으로 재계산과 I/O 전송을 오버랩하면, 비연속 세그먼트 복원 병목을 제거해 Activity B의 히트 시 TTFT 오버헤드를 10~62% 줄일 수 있다.
- **접근 방법**:
  1. `RestoreAwareBatchScheduler` (`src/scheduler/restore_aware_scheduler.py`): 요청별 복원 비용(재계산 vs I/O) 추정
  2. two-pointer: 고복원비용 요청과 저복원비용 요청을 교차 배치하여 I/O-compute 오버랩 극대화
  3. 레이어 차원: 하위 레이어부터 복원 시작 → 상위 레이어 복원과 하위 레이어 어텐션 계산 파이프라인
  4. `CompressedSegmentCache`의 `restore()` 호출 경로에 병렬 스케줄 힌트 주입
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티노드 환경에서 원격 KV 복원(RDMA 전송)을 3D 병렬 파이프라인의 "I/O 차원"으로 추상화하면 노드 간 복원 병목도 해소 가능
- **예상 효과**: 처리량 +10~15%, 비연속 히트 시 TTFT p50 −15~30%
- **구현 난이도**: high
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: RelayCaching 훈련-무료 디코딩→프리필 KV 재주입
- **가설**: 이전 에이전트·턴의 디코딩 단계에서 생성된 KV 캐시를 훈련 없이 다음 에이전트·턴의 프리필에 재주입할 때, U자형 레이어 프로파일러로 "편차 집중 레이어(deviation-concentrated layers)"를 식별하고 해당 레이어의 핵심 토큰만 선택적으로 재계산하면, 80% 이상의 KV를 재사용하면서 어텐션 품질을 FP16 수준으로 유지할 수 있다.
- **접근 방법**:
  1. `RelayCacheStore` 클래스 (`src/cache/relay_cache.py`): 디코딩 KV 저장 + 프리필 재주입 인터페이스
  2. U자형 레이어 프로파일러: 소수 샘플로 레이어별 KV divergence 프로파일링 → 상위·하위 레이어(deviation 집중)를 "수정 레이어(correction layers)"로 표시
  3. 편차 기반 토큰 선택기(deviation-based token selector): correction layer에서 KL divergence > threshold인 토큰만 재계산 (전체의 ~20%)
  4. 영향 기반 토큰 선택기(influence-based selector): 이후 레이어 어텐션에 영향이 큰 토큰 우선 재계산
  5. 기존 `CacheStore` 인터페이스 상속, `put_decode_kv()` / `get_for_prefill()` 메서드 추가
  6. 멀티턴·멀티에이전트 시나리오: 디코딩 KV를 세션 ID별로 분리 저장, 동일 세션 다음 요청 시 자동 재주입
- **예상 효과**: 비연속 히트율 +15~20%p (디코딩 KV 재사용 경로 추가), TTFT −25~40% (대규모 재계산 제거)
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 핵심 타겟)

### 아이디어 B-2: 1-bit 부호 VQ 기반 세그먼트 유사도 검색 색인
- **가설**: 기존 `SegmentedHashCache`의 정확 해시 매칭(exact match) 대신, Self-Indexing KVCache의 1-bit 부호 벡터 양자화(sign VQ)로 Key를 압축하여 "근사 최근접 세그먼트 검색(approximate segment retrieval)"을 지원하면, 정확 일치가 없어도 유사 세그먼트를 히트시켜 비연속 히트율을 추가로 높일 수 있다.
- **접근 방법**:
  1. `SignVQSegmentIndex` 클래스 (`src/cache/sign_vq_index.py`): 프리필 단일 패스에서 코드북 구성
  2. Key 벡터를 sign(K) → 1-bit 코드로 변환 (d_head 비트의 이진 벡터)
  3. 코사인 유사도 ≈ 해밍 거리의 역함수로 근사 → bitwise XOR + popcount 연산으로 O(d/64) 검색
  4. top-k 유사 세그먼트 후보를 `CompressedSegmentCache`에서 로드 → 어텐션 sparsity 마스크로 활용
  5. 기존 정확 해시 히트와 근사 유사 히트를 분리 추적하여 hitrate 측정 시 구분
- **예상 효과**: 비연속 히트율 추가 +5~8%p (정확 해시 미스지만 유사 세그먼트 존재 시), 검색 오버헤드 O(N × d/64) — N=세그먼트 수
- **구현 난이도**: medium
- **우선순위**: 2

## Activity C 아이디어

### 아이디어 C-1: CapKV 레버리지 스코어 기반 정보 용량 최대화 퇴거
- **가설**: KV 캐시 퇴거를 Information Bottleneck 원리로 재해석해, 통계적 레버리지 스코어(log-determinant 근사)로 보유 KV 부분집합의 "유효 정보 용량"을 최대화하는 퇴거 정책을 적용하면, 이전 사이클의 attention score 기반 ARKV tri-state 대비 공격적 압축 비율에서도 정확도 저하 곡선이 완만(graceful degradation)해진다.
- **접근 방법**:
  1. `CapKVEvictionPolicy` 클래스 (`src/cache/capkv_policy.py`): 레버리지 스코어 계산 + 퇴거 결정
  2. 레버리지 스코어 근사: s_i = K_i^T (K^T K + λI)^{-1} K_i — 랭크-k 근사로 O(Nkd) 계산
  3. 로그-행렬식 목적함수: 보유 집합 S의 log det(K_S^T K_S + λI)를 탐욕적 최대화
  4. 예산 제약: 총 KV 토큰 수 B에서 레버리지 스코어 누적 기여가 최대인 B개 토큰 선택
  5. `CompressedSegmentCache`의 `evict()` 메서드를 `CapKVEvictionPolicy`로 교체
  6. ARKV tri-state의 retain/compress/evict 3-상태와 병용 가능: retain 후보를 레버리지 스코어로 재정렬
- **예상 효과**: 메모리 −35~45%, accuracy delta ±0.5% 이내 (공격적 압축 비율에서도 ARKV 대비 −30% 추가 정확도 보존)
- **accuracy-preserving 근거**: CapKV 논문이 선형-가우시안 어텐션 모델 하에서 닫힌 형태 상호 정보 목적함수로 정확도 보존을 수학적으로 보장. 레버리지 스코어는 데이터 분포에서 "정보적으로 다양한" 토큰을 선택하므로 어텐션 rank deficiency를 방지. 검증: WikiText-2 perplexity ±1% + LongBench 서브태스크 3개 정확도 비교.
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 핵심 타겟)

### 아이디어 C-2: VQKV 벡터 양자화 + ReCalKV K/V 비대칭 저랭크 계층적 압축
- **가설**: Key는 헤드-와이즈 유사도 인식 재정렬(HSR) 후 그룹화된 SVD 저랭크 근사(ReCalKV 스타일)를 적용하고, Value는 VQKV 스타일 벡터 양자화(SimVQ 기반 코드북)를 적용하면, 각각의 수학적 특성에 최적화된 비대칭 압축이 가능하며 단일 방법 대비 더 높은 압축률에서 정확도가 유지된다.
- **접근 방법**:
  1. `AsymmetricKVCompressor` 클래스 (`src/cache/asymmetric_compressor.py`)
  2. Key 경로: `HSRGroupedSVD` — 헤드 간 코사인 유사도 행렬로 유사 헤드 클러스터링 → 클러스터별 SVD (rank=r, r ≈ d_head × 0.5)
  3. Value 경로: `SimVQCodebook` — 프리필 시 K-means로 코드북 구성(코드북 크기 256), 추론 시 최근접 코드 인덱스로 저장
  4. 레이어별 비율 조정: 초기 N/3 레이어는 Key FP16 + Value INT8, 나머지는 K HSR-SVD + V VQ
  5. `CompressionCodec` 인터페이스 구현으로 기존 `CompressedSegmentCache`와 호환
- **예상 효과**: 메모리 −60~70%, accuracy delta ±0.8% 이내 (LLaMA3.1-8B LongBench 기준)
- **accuracy-preserving 근거**: ReCalKV가 50% 압축에서 정확도 2% 상대 저하(±1% 절대 이내)를 실증. VQKV가 82.8% 압축에서 98.6% 성능 유지. K/V 특성 분리: Key는 구조적 유사성(헤드 클러스터링), Value는 내용 정보(코드북 VQ) — 각 특성에 맞는 압축 적용으로 오류 누적 최소화. 검증: WikiText-2 perplexity ±1% + LongBench 정확도 ±1%.
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 C-3: LookaheadKV 경량 미래 중요도 예측 통합
- **가설**: 경량 파라미터-효율 모듈(lightweight PEM)로 미래 응답의 대리 표현을 예측해 KV 토큰 중요도를 프리필 시점에 사전 추정하면, 기존 attention score 기반 퇴거(ARKV 등) 대비 드래프트 생성 오버헤드 없이 퇴거 결정 품질이 향상된다.
- **접근 방법**:
  1. `LookaheadImportancePredictor` (`src/cache/lookahead_predictor.py`): Samsung Labs 공개 구현체 기반
  2. PEM 구조: 각 레이어의 현재 hidden state → 1-layer MLP → 미래 어텐션 패턴 예측 벡터 (d_pred=32)
  3. 중요도 스코어 = 현재 attention score × lookahead 예측 스코어 (가중 평균 α=0.3)
  4. `CapKVEvictionPolicy` (C-1)의 레버리지 스코어 입력으로 lookahead 스코어를 추가 특성으로 주입하는 앙상블 구성 가능
  5. TTFT 오버헤드: PEM 추론은 기존 경험적 방법 수준으로 최소화 (LookaheadKV 논문 실증)
- **예상 효과**: 퇴거 비용 −10~15× (드래프트 생성 제거), accuracy delta ±0.4% 이내, TTFT 오버헤드 +1~2%
- **accuracy-preserving 근거**: ICLR 2026 채택. 명시적 드래프트 생성 없이 미래 중요도를 예측하여 현재 토큰의 미래 어텐션 기여를 반영한 퇴거 — 어텐션 패턴의 미래 관련성을 고려하므로 단순 누적 attention score 기반 대비 중요 토큰 보존율이 높다. WikiText-2 perplexity + RULER long-context 벤치마크로 검증.
- **구현 난이도**: medium
- **우선순위**: 2

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: 1-bit Sign VQ 색인 + CapKV 레버리지 퇴거 융합 (B+C)
- **조합 근거**: Self-Indexing KVCache의 핵심 인사이트인 "압축된 Key가 동시에 유사도 검색 색인으로 작동"을 CapKV의 정보이론적 퇴거와 결합한다. 1-bit sign VQ는 압축(C)과 검색 색인(B)을 단일 표현으로 통합하고, CapKV의 레버리지 스코어는 어떤 세그먼트를 "고정밀도로 캐시할지 / 1-bit 색인만 보존할지 / 완전 퇴거할지"를 결정하는 3-계층 정책을 제공한다.
- **구체 설계**:
  1. 레버리지 스코어 상위 20%: 원본 FP16 KV 캐시 보존 + 1-bit sign 색인 생성
  2. 레버리지 스코어 중간 60%: 1-bit sign VQ 색인만 보존 (근사 검색 가능, 실제 KV 재계산 필요)
  3. 레버리지 스코어 하위 20%: 완전 퇴거 (sign 색인도 제거)
  4. 히트 시 처리: FP16 히트 → 즉시 사용, sign-only 히트 → 경량 재계산 후 사용, 미스 → 전체 재계산
  5. 메모리 절감: FP16(20%) + 1-bit(60%) + 없음(20%) = 전체 토큰 대비 약 77% 절감
- **예상 복합 효과**: 처리량 +20%, 메모리 −70~75%, 정확도 ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 전체 타겟)

### 아이디어 Cross-2: GSA 스케줄러 + CapKV 정보 용량 공동 최적화 (A+C)
- **조합 근거**: Competitive Non-Clairvoyant 스케줄러(GSA)의 메모리 예산 모델에 CapKV의 정보 용량 목적함수를 통합한다. GSA의 "단계별 허용 동시 요청 수"를 "레버리지 스코어 합산 기반 유효 KV 용량"으로 재정의하면, 메모리 예산 위반 없이 정확도가 보존되는 이론적으로 보장된 스케줄러-압축기 공동 최적화가 가능하다.
- **구체 설계**:
  1. 각 요청의 유효 KV 용량 = sum(leverage_scores) × d_kv / (압축 비율)
  2. GSA의 단계 k에서 허용 동시 요청 수 = ⌊M_info / avg_effective_capacity⌋
  3. CapKVEvictionPolicy가 각 단계 전환 시 메모리 압축을 수행해 예산 준수 보장
  4. 경쟁비 계산: 이론적 보장(±정보 용량 근사 오차)을 기존 GSA 분석에 추가
- **예상 복합 효과**: 처리량 +15~20%, 메모리 −40~50%, 정확도 ±0.6% 이내, 경쟁비 보장 유지
- **구현 난이도**: high
- **우선순위**: 4

### 아이디어 Cross-3: RelayCaching 훈련-무료 재주입 + 6-계층 메모리 계층 라우팅 (A+B)
- **조합 근거**: RelayCaching(B-1)의 디코딩 KV를 Predictive Multi-Tier Memory Management(2604.26968)의 6-계층 메모리 계층(HBM → CPU DRAM → CXL → NVMe → RDMA → 병렬 파일시스템)에 배치한다. 편차 집중 레이어의 수정 대상 KV는 HBM에, 비수정 재사용 KV는 DRAM/CXL에 계층적으로 배치하면 복원 지연과 메모리 압력을 동시에 최소화할 수 있다.
- **구체 설계**:
  1. `TieredRelayCacheRouter` (`src/cache/tiered_relay_router.py`): 레이어 프로파일 기반 계층 배치 결정
  2. correction_layers(편차 집중): KV → HBM 우선 배치, 용량 초과 시 DRAM으로 오프로드
  3. passthrough_layers(편차 낮음): KV → DRAM/CXL 직접 배치, HBM 미사용
  4. 복원 시: HBM 히트(즉시), DRAM 히트(~1ms), CXL 히트(~5ms) — 재계산보다 저렴한 계층까지만 복원
  5. 멀티노드: RDMA 계층을 통해 다른 노드의 passthrough KV를 원격 복원
- **예상 복합 효과**: 처리량 +18~22%, 비연속 히트율 +12~18%p, TTFT −20~35%
- **구현 난이도**: high
- **우선순위**: 3

## 이번 사이클 최우선 구현 타겟

**B-1(RelayCaching) + C-1(CapKV) + Cross-1(1-bit Sign VQ + CapKV 3-계층 융합)**

구현 순서:
1. **C-1** `CapKVEvictionPolicy`: 레버리지 스코어 계산 + 3-계층 분류(retain/sign-index/evict), `CompressedSegmentCache.evict()` 교체
2. **B-2** `SignVQSegmentIndex`: 1-bit sign VQ 코드 생성, 해밍 거리 근사 검색 추가
3. **B-1** `RelayCacheStore`: U자형 프로파일러 + 편차 기반 토큰 선택기, 훈련-무료 재주입 경로
4. **Cross-1** 통합: CapKV 레버리지 스코어로 Sign-only / FP16 / 퇴거 계층 결정, B-1의 재주입 경로와 연결

이유:
- Cross-1은 이전 사이클의 모든 B+C 구현(ARKV tri-state + HadamardInt4 + KV Packet adapter)과 알고리즘·자료구조가 다르면서, 메모리 목표 −30%를 2배 이상 초과 달성(-70~75%)한다.
- B-1(RelayCaching)은 훈련 없이 구현 가능해 이전 사이클의 KV Packet adapter(MLP 학습 필요) 대비 구현 리스크가 낮다.
- CapKV(C-1)는 ICLR 수준 정보이론 기반으로 accuracy-preserving 제약을 수학적으로 충족한다.
