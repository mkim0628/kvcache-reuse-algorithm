## 변화 요약
1. **AMPD(2602.14516)의 KV 지연 읽기(pull-on-demand) 원칙이 비연속 세그먼트 스케줄링에 새 설계 차원을 열었다**: 이전 18개 사이클의 모든 A 기법은 스케줄러가 KV를 사전에 전송(push)하거나 오프로딩하는 방식이었으나, AMPD는 스케줄 확정 순간에만 KV를 읽는 "pull-on-demand" 방식으로 불필요한 전송을 원천 차단한다. 이 원칙을 비연속 세그먼트 재사용 스케줄러에 적용하는 "세그먼트 지연-페치 스케줄러(Segment Lazy-Fetch Scheduler)"는 이전 어떤 A 기법에도 없던 새 메커니즘이다.
2. **PrefillShare(2602.12029)의 "1-to-N KV 배포 라우팅" 패턴이 Activity A에 신규 설계 방향을 제공한다**: PrefillShare는 단일 프리필 노드가 KV를 한 번만 생성하고 N개 디코딩 노드에 배포하는 "1-to-N 팬아웃 KV 라우팅"을 도입한다. 이전 A 기법들(MooncakeRelayKVRoutingScheduler, PBKVAgentSegmentPreservationScheduler, PrefillShare 자체는 시스템 레이어)은 클러스터 KV 풀 히트율 최대화 또는 단일 흐름 최적화에 집중했으나, "1-to-N 팬아웃 라우팅 + 세그먼트별 수신자 선택" 조합은 이전 18개 사이클 전체에 없다.
3. **SGLang v0.5.11 DP Attention의 "멀티-GPU KV 중복 제거 + 압축 이중 절감" 정량화 방향이 Activity C의 새 평가 축을 형성한다**: DP Attention이 N-GPU 환경에서 KV 복사본을 1/N으로 줄이고 Activity C의 압축 기법이 개별 KV 크기를 추가로 절감하면 두 절감의 곱으로 효과가 증폭된다. 단일 GPU 환경과 멀티 GPU 환경에서 압축 효과의 의존성을 정량화하는 "DP Attention 인식 압축 코덱"은 이전 C 기법 전체가 단일 GPU 환경만 가정했던 것과 근본적으로 다른 평가 차원이다.

## 이번 사이클 초점
**A+B+C** (Activity A: AMPD 지연 페치 원칙 기반 세그먼트 스케줄러 + PrefillShare 1-to-N 팬아웃 KV 라우터, Activity B: AMPD 지연 로드 세그먼트 파이프라인, Activity C: DP Attention 인식 압축 코덱)

**선택 이유**: 오늘 트렌드의 세 신규 항목(AMPD, PrefillShare, SGLang DP Attention)이 각각 A, A+B, C의 새 메커니즘을 제공하며 A+B+C 통합 설계에서 강한 시너지가 성립한다. AMPD 지연 읽기(A)가 세그먼트 확정 전 불필요한 전송을 차단하고, AdapShot RoPE 재인코딩(B)과 결합한 "세그먼트 지연-로드 파이프라인(A+B)"이 재조합 지연을 줄이며, DP Attention 인식 압축(C)이 멀티-GPU 환경에서 중복 제거와 압축을 동시에 달성하는 완전한 스택이 가능하다. 세 기법 모두 이전 18개 사이클 ~180개 아이디어에 없는 신규 메커니즘·평가 차원을 도입한다.

## Activity A 아이디어

### 아이디어 A-1: AMPDLazySegmentFetchScheduler (AMPD 지연-페치 비연속 세그먼트 스케줄러)
- **가설**: AMPD(2602.14516)의 "KV 지연 읽기(lazy read)" 원칙 — 스케줄 확정 시점에만 KV를 읽고 메타데이터만 먼저 전달하는 pull-on-demand 방식 — 을 Activity B의 비연속 세그먼트 재조합 경로에 적용하면, 재사용할 세그먼트 집합이 최종 확정되기 전에 불필요하게 KV 블록을 선제 전송하는 낭비를 제거할 수 있으며, 세그먼트 집합 확정 후에만 해당 세그먼트를 한 번에 pull하는 방식으로 스케줄링 오버헤드와 불필요한 KV 전송을 동시에 줄일 수 있다. 이전 모든 A 기법(KVRMStaticGraphCoalescingSegmentScheduler, HMAMultiConnectorScheduler, DCPPCPFeedbackScheduler 등)은 KV를 "push"하거나 "선제 오프로딩"하는 방식이었으며, 세그먼트 재사용 결정이 확정될 때까지 KV 전송을 보류하는 pull-on-demand 스케줄러는 전혀 없었다.
- **접근 방법**:
  1. `AMPDLazySegmentFetchScheduler` 클래스 (`src/scheduler/ampd_lazy_segment_fetch.py`): `BaseScheduler` 상속
  2. **세그먼트 메타데이터 선행 전달 (pre-resolve 단계)**:
     - 요청 수신 시 세그먼트 내용 해시 + 위치 범위만 담은 경량 메타데이터 패킷(segment_id, position_range, source_node_id, approx_kv_size)을 스케줄러가 수집. KV 데이터 자체는 전송하지 않음.
     - `SegmentMetadataRegistry` (`src/scheduler/segment_metadata_registry.py`): 세그먼트 ID → {source_node, tier, approx_size} 매핑을 추적하는 경량 레지스트리. gRPC 스트림으로 노드 간 메타데이터 동기화.
  3. **스케줄 확정 후 pull-on-demand 페치**:
     - 스케줄러가 재사용 세그먼트 집합 `S_reuse = {seg_1, ..., seg_k}`를 최종 확정한 순간에만 해당 세그먼트들의 KV 데이터를 소스 노드에서 pull.
     - 확정 판단 기준: Louver 반공간 범위 탐색(이미 보고) 결과로 히트 보장 세그먼트 집합이 수렴했을 때.
     - `fetch_segments_lazy(confirmed_segment_ids: List[str]) → AsyncIterator[KVSegment]`: 비동기 스트림으로 KV를 수신.
  4. **전송-계산 오버랩 파이프라인**:
     - 먼저 확정된 세그먼트부터 순차적으로 fetch + AdapShot RoPE 재인코딩(Activity B)을 오버랩. 세그먼트 페치 완료 즉시 재인코딩 시작, 재인코딩 완료 즉시 어텐션 계산에 투입.
     - CUDA Stream: fetch stream (RDMA 수신) + reencoding stream + compute stream 3-way 파이프라인.
  5. **불필요 전송 제거 정량화**:
     - `unnecessary_transfer_ratio = (cancelled_segment_fetches) / (pre-resolved_candidates)` 를 `results/<exp-name>/metrics.json`에 기록. AMPD 원논문의 "메타데이터만 선행 전달 후 확정 시 페치"와 동일 원리가 비연속 세그먼트에서 얼마나 불필요 전송을 줄이는지 정량화.
  6. **이전 A 기법과의 차별점**:
     - KVRMStaticGraphCoalescingSegmentScheduler(05-16 A-1)은 "선택된 세그먼트 블록들을 합체 전송"하는 전송 최적화이고, DCPPCPFeedbackScheduler(05-17 A-2)는 "오프로딩 완료 이벤트 기반 배치 편성"이다. 이 기법은 "세그먼트 재사용 결정 확정 전까지 KV 전송 자체를 보류"하는 pull-on-demand 지연 정책으로 이들과 근본 원리가 다르다. 특히 HMAMultiConnectorScheduler(05-17 A-1)가 "어떤 압축 커넥터를 선택할지" 결정하는 메타-스케줄러라면, AMPDLazySegmentFetchScheduler는 "세그먼트 재사용 결정이 확정되기 전에는 KV를 읽지 않는다"는 전송 타이밍 제어다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 세그먼트 메타데이터를 노드 간 경량 gRPC로 전파 (KV 데이터는 확정 후 RDMA pull). 소스 노드가 여러 개인 경우 `SegmentMetadataRegistry`가 최적 소스 노드(가장 낮은 전송 지연)를 선택해 pull. 노드 간 대역폭 압박이 높은 환경에서 불필요 전송 제거 효과가 극대화됨(InfiniBand: 200Gbps 공유 대역폭 경쟁 완화).
- **예상 효과**: 불필요 KV 전송 −40~65% (사전 확정되지 않은 후보 세그먼트 전송 제거), 처리량 +15~25% (전송 병목 해소 + 전송-계산 오버랩), Scheduling overhead TTFT +3% 이내 (메타데이터 선행 해석 오버헤드 < 0.1ms/요청)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: PrefillShareFanoutKVDistributionRouter (PrefillShare 1-to-N 팬아웃 KV 배포 라우터)
- **가설**: PrefillShare(2602.12029)의 "단일 프리필 노드가 KV를 한 번만 생성하고 N개 디코딩 노드에 배포"하는 1-to-N 팬아웃 KV 라우팅을 Activity B의 비연속 세그먼트 캐시와 연결하면, 공통 세그먼트를 한 번만 프리필하고 N개 디코딩 워커에 각자의 비연속 세그먼트 집합을 팬아웃 배포하는 "세그먼트-선택적 1-to-N 라우터"가 가능하다. 기존 MooncakeRelayKVRoutingScheduler(05-15 A-2)는 이전 에이전트의 전체 KV를 클러스터 풀로 라우팅했으나, PrefillShare 스타일 "1-to-N 팬아웃 + 디코딩 노드별 세그먼트 선택"은 이전 18개 사이클 전체에 없는 새 라우팅 패턴이다.
- **접근 방법**:
  1. `PrefillShareFanoutRouter` 클래스 (`src/scheduler/prefill_share_fanout_router.py`): `BaseScheduler` 상속
  2. **공유 프리필 노드 역할 분리**:
     - 공유 세그먼트(공통 시스템 프롬프트, 툴 디스크립션, RAG 컨텍스트 등)를 식별하는 "공유 세그먼트 감지기": 요청 배치에서 공통 접두사 + 공통 비연속 세그먼트를 동시에 식별.
     - 공유 세그먼트는 단일 프리필 노드(또는 현재 가장 캐시가 따뜻한 노드)에서 1회만 프리필 + KV 생성.
     - 비공유(태스크-특화) 세그먼트는 각 디코딩 노드 로컬에서 처리.
  3. **1-to-N KV 팬아웃 배포**:
     - 프리필 노드가 공유 세그먼트 KV 생성 완료 시 → N개 디코딩 노드에 KV를 팬아웃 전송.
     - 팬아웃 전송 구현: IP Multicast 또는 P2P RDMA Write N회(N ≤ 8 시 RDMA P2P, N > 8 시 tree-cast). 수신 노드별 세그먼트 ID 인덱싱은 `SegmentedHashCache.put(segment_id, kv_tensor)` 호출로 통일.
  4. **디코딩 노드별 세그먼트 선택적 수신**:
     - 각 디코딩 노드가 자신에게 필요한 세그먼트 ID 목록을 "팬아웃 요청 메시지"에 포함해 프리필 노드에 전달. 프리필 노드는 해당 세그먼트만 해당 디코딩 노드로 선택 전송.
     - `fanout_request = {node_id: [segment_id_1, segment_id_3, ...], ...}` 형태의 팬아웃 계획을 사전 수립하는 `FanoutPlannerPolicy`.
  5. **vLLM HMA 멀티-커넥터 통합 경로**:
     - 각 디코딩 노드에서 수신한 공유 KV를 HMA DDR 커넥터에 저장 → HMAMultiConnectorScheduler(05-17 A-1) 기구현 레지스트리에 등록. 팬아웃 수신 KV가 기존 HMA 커넥터 파이프라인에 자동 통합.
  6. **이전 A 기법과의 차별점**:
     - MooncakeRelayKVRoutingScheduler(05-15 A-2)는 이전 에이전트 전체 KV를 분산 풀 → 다음 에이전트 라우팅(1-to-1 or 1-to-pool)이고, PBKVAgentSegmentPreservationScheduler(05-13 A-1)는 에이전트 호출 예측 기반 GPU 보존 정책이다. PrefillShareFanoutRouter는 "공유 세그먼트를 1회 프리필 후 N개 디코딩 노드에 선택적 팬아웃"이라는 1-to-N 동시 배포 패턴으로, 이전 어떤 A 기법과도 다른 라우팅 위상을 도입한다.
- **대상 환경**: 멀티 노드 (1-to-N 팬아웃 필수)
- **멀티 노드 고려사항**: InfiniBand 환경: RDMA Write N회 (N ≤ 8) 또는 tree-cast (N > 8). NVLink 환경: NVSwitch allgather로 팬아웃 구현. RDMA 대역폭 포화 방지: 팬아웃 전송을 배치 내 첫 디코딩 스텝 완료 전에 파이프라인화. 네트워크 토폴로지 인식: 동일 랙 내 디코딩 노드에 우선 배포 후 크로스-랙 배포.
- **예상 효과**: 처리량 +20~35% (공유 세그먼트 중복 프리필 제거 + N개 디코딩 노드 병렬화), TTFT −50~70% (공유 KV 재사용으로 프리필 스킵), Scheduling overhead TTFT +4% 이내 (팬아웃 계획 수립 오버헤드: 배치 크기 N에 선형, N=8 기준 < 0.5ms)
- **구현 난이도**: medium
- **우선순위**: 2

## Activity B 아이디어

### 아이디어 B-1: AMPDAdapShotLazyLoadSegmentPipeline (AMPD 지연 로드 + AdapShot 재인코딩 세그먼트 파이프라인)
- **가설**: AMPD(2602.14516)의 지연 읽기 원칙(스케줄 확정 순간에만 KV를 읽음)과 AdapShot(2605.03644)의 RoPE 회전 기반 위치 재인코딩(비연속 세그먼트를 무손실 재배치)을 결합하면, "세그먼트 집합 확정 후 비동기 로드 + 즉시 재인코딩 오버랩"이라는 새로운 비연속 재조합 파이프라인이 가능하다. 이전 B 기법 전체(HMASegmentTierCache, RelayUShapeLayerSelectiveSegmentCache, KVFoldAccumulativeRadixCache, SPKVSparsityProfiledSegmentReusePolicy 등)는 세그먼트가 이미 메모리에 있다고 가정하고 재사용 결정을 내렸으나, "로드 지연 + 재인코딩 오버랩"이라는 파이프라인 설계는 없었다. 트렌드 리포트의 "세그먼트 지연-로드 파이프라인 (AMPD + AdapShot 결합)" 제안을 직접 구현한다.
- **접근 방법**:
  1. `AMPDAdapShotLazyLoadPipeline` 클래스 (`src/cache/ampd_adapshot_lazy_pipeline.py`): `CacheStore` 인터페이스 상속
  2. **3단계 비동기 파이프라인 설계**:
     - **Stage 1 (Segment Resolution)**: Louver 반공간 범위 탐색으로 히트 보장 세그먼트 집합 S_reuse 확정. 이 단계에서는 KV 데이터를 메모리로 가져오지 않고 세그먼트 ID와 소스 위치(source_tier: HBM/DDR/REMOTE)만 결정. `resolve_segments(query_tokens) → List[SegmentMeta]` 인터페이스.
     - **Stage 2 (Lazy Load)**: S_reuse 확정 후 비동기 로드 시작. HBM 세그먼트: 즉시 반환. DDR 세그먼트: `torch.cuda.Stream`으로 비동기 H2D 전송. REMOTE 세그먼트: RDMA pull (AMPDLazySegmentFetchScheduler의 pull-on-demand 경로 활용). 세그먼트별 `asyncio.Event` 완료 신호.
     - **Stage 3 (RoPE Reencoding Overlap)**: 각 세그먼트 로드 완료 이벤트 수신 즉시 AdapShot RoPE 재인코딩 시작 (Stage 2와 오버랩). 재인코딩 완료 즉시 어텐션 계산 큐에 투입. 세그먼트 간 재인코딩 오버랩으로 전체 재조합 지연 = max(로드 지연, 재인코딩 지연) (이론적 최솟값).
  3. **파이프라인 스케줄링 로직**:
     - 세그먼트 처리 우선순위: HBM 세그먼트(즉시) > DDR 세그먼트(HBM 용량에 따라) > REMOTE 세그먼트(네트워크 지연 최장).
     - 재인코딩 우선순위: 로드 완료된 세그먼트 중 어텐션 계산에서 먼저 참조될 세그먼트 우선.
     - 예측 프리페치: 이전 배치에서 자주 공동 히트한 세그먼트 쌍을 학습해 "동반 세그먼트(companion segments)"를 Stage 1 이전에 DDR에서 HBM으로 프리페치.
  4. **AdapShot RoPE 재인코딩 통합**:
     - 각 세그먼트의 원본 위치 범위 `[pos_start, pos_end]`와 현재 컨텍스트에서의 타겟 위치 `[target_start, target_end]`를 기반으로 RoPE 위상 오프셋 Δθ 계산.
     - 재인코딩 커널: `torch.einsum` 기반 배치 회전 연산, 세그먼트 길이 × d_head 크기 텐서에 적용.
     - 재인코딩 오버헤드: 세그먼트 길이 S 기준 O(S × n_heads × d_head) FP16 연산 ≈ 로드 지연의 10~30% (REMOTE 세그먼트 기준으로는 무시 가능).
  5. **기존 B 기법과의 차별점**:
     - HMASegmentTierCache(05-17 B-1)이 "세그먼트 캐시 자체의 HBM/DDR/DISK 멀티-티어 관리"를 다룬다면, AMPDAdapShotLazyLoadPipeline은 "세그먼트 집합 확정 후 로드-재인코딩 파이프라인의 비동기 오버랩 설계"라는 실행 파이프라인 최적화다. 세그먼트 스토리지 관리(어디에 저장)와 실행 파이프라인(어떻게 로드하고 재인코딩할지)을 명확히 분리한다. RelayUShapeLayerSelectiveSegmentCache(05-15 B-1)은 세그먼트가 메모리에 있을 때 레이어 선택 재사용을 결정하는 반면, 이 기법은 세그먼트 로드 자체를 지연시키고 로드와 재인코딩을 파이프라인화한다.
- **예상 효과**: 비연속 재조합 지연 −30~50% (로드-재인코딩 오버랩으로 직렬 지연 → 병렬 지연), TTFT −20~35% (재조합 병목 해소), 불필요 세그먼트 로드 −35~55% (Stage 1에서 확정된 세그먼트만 로드)
- **구현 난이도**: medium
- **우선순위**: 1

## Activity C 아이디어

### 아이디어 C-1: DPAttentionAwareCompressionCodecEvaluator (DP Attention 인식 압축 코덱 평가기 + 환경별 압축 정책)
- **가설**: SGLang v0.5.11의 DP Attention이 N-GPU 환경에서 KV 복사본을 1/N으로 줄이고, Activity C의 압축 기법(SP-KV write-time 희소화, GlobalRetentionGateEvictionCodec, FibQuantVQCodec 등)이 각 KV의 크기를 추가로 절감하면, 두 절감의 곱으로 총 메모리 절감이 증폭된다. 그러나 DP Attention이 없는 단일 GPU 환경과 있는 멀티 GPU 환경에서 동일 압축 기법의 효과가 다를 수 있으며, 특히 "DP Attention이 이미 KV 중복을 제거한 멀티-GPU 환경에서 추가 압축의 한계 효용"은 이전 어떤 C 기법도 정량화하지 않았다. 이를 측정하는 "DP Attention 인식 압축 정책 선택기"가 단일/멀티 GPU 환경 모두에서 최적 압축 기법을 자동 선택하는 새 설계 차원을 형성한다.
- **접근 방법**:
  1. `DPAttentionAwareCompressionSelector` 클래스 (`src/cache/dp_attention_aware_compression.py`): `CompressionCodec` 인터페이스 구현
  2. **환경 감지 및 DP Attention 상태 조회**:
     - 초기화 시 실행 환경 감지: `n_gpus = torch.cuda.device_count()`, DP Attention 활성화 여부 `dp_attn_enabled` (vLLM/SGLang 설정에서 읽거나 환경 변수로 주입).
     - "유효 KV 복사본 수(effective_kv_replicas)" 계산: DP Attention 비활성화 시 = `n_gpus` (각 GPU에 전체 KV 복사본), DP Attention 활성화 시 = 1 (KV 중복 제거 완료).
     - `effective_kv_replicas`를 이후 압축 정책 선택의 핵심 입력으로 사용.
  3. **환경별 압축 정책 선택 로직**:
     - **단일 GPU 또는 DP Attention 비활성화 환경** (`effective_kv_replicas > 1`): KV 중복 제거가 없으므로 개별 KV 압축이 전체 메모리 절감에 직접 기여. 고압축 코덱 우선 선택: `SPKVWriteTimeCodec`(write-time 3~10× 희소화) 또는 `FibQuantVQCodec`(10×+ VQ).
     - **멀티 GPU + DP Attention 활성화 환경** (`effective_kv_replicas = 1`): KV 중복이 이미 제거됐으므로 "개별 압축의 한계 효용 vs. 압축 오버헤드"를 비교. 한계 효용이 낮은 레이어는 압축 스킵(overhead 절감), 높은 레이어만 선택적 압축. `GlobalRetentionGateEvictionCodec`의 전역 예산 기반 퇴거가 DP Attention 환경에서 더 효과적 (중복이 없으므로 퇴거 대상이 실질 메모리).
     - 선택 결정을 `configs/dp_attn_compression_policy.yaml`에 외부화.
  4. **"DP Attention × 압축" 이중 절감 정량화 실험 설계**:
     - 실험 행렬: {DP Attention ON/OFF} × {압축 기법: None/SPKVWriteTime/GlobalRetention/FibQuant/RateQuant} × {n_gpus: 1/2/4/8}.
     - 측정 지표: 실효 메모리 절감율 = (1 - 1/(effective_kv_replicas × compression_ratio)), 처리량(tokens/sec), accuracy delta.
     - 이론 최솟값: DP Attention(N-GPU) + FibQuant(10×) 결합 시 실효 절감 = 1 - 1/(N × 10). N=4, ratio=10 → 97.5% 절감 — 이론 최솟값 검증.
     - 결과를 `results/<exp-name>/dp_attn_compression_matrix.json`에 기록.
  5. **런타임 압축 정책 전환**:
     - 서빙 중 DP Attention 상태 변화(예: 노드 장애로 DP Attention 비활성화) 감지 시 압축 정책 자동 전환. `register_env_change_callback(callback)` 인터페이스.
  6. **이전 C 기법과의 차별점**:
     - 이전 18개 사이클의 모든 C 기법(RLAdaptivePrecisionQuantizer, NVFPFourBudgetAdapter, GlobalRetentionGateEvictionCodec, SPKVWriteTimeCodec, FibQuantVQCodec, ABSparseCodec, FluxionSparseRestoreCodec, RateQuantReverseWaterfillingCodec, LookaheadKVEvictionCodec, SRFTFusedINT4KVKernel, SMDKVCompressor 등)은 모두 단일 GPU 환경을 암묵적으로 가정하거나 멀티 GPU 환경을 고려하지 않았다. DPAttentionAwareCompressionSelector는 "DP Attention에 의한 KV 중복 제거 상태를 압축 정책 결정의 입력으로 사용"하는 최초의 C 기법으로, 멀티-GPU 환경에서의 압축 효과 환경 의존성 정량화라는 완전히 새로운 설계 차원을 도입한다.
- **예상 효과**: 메모리 −70~97% (환경별 최적 압축 선택 + DP Attention 중복 제거 곱 효과), accuracy delta ±0.7% 이내 (환경별 적합한 압축 강도 선택으로 과압축 방지), 압축 오버헤드 −20~40% (DP Attention 환경에서 과압축 스킵)
- **accuracy-preserving 근거**:
  (1) DP Attention 활성화 환경에서 KV 중복이 이미 제거됐으므로 추가 압축의 필요 강도가 단일 GPU 대비 낮다. 낮은 압축 강도 선택으로 accuracy delta가 자동으로 낮아진다.
  (2) 단일 GPU 환경에서는 고압축 코덱(FibQuant, SP-KV)을 선택하되, 각 코덱의 기존 accuracy-preserving 보장(FibQuant: ±0.5% at 10×, SP-KV: ±0.5% at 3~10×)을 그대로 상속한다.
  (3) 실험 행렬에서 각 {환경, 코덱} 조합의 accuracy delta를 WikiText-2 perplexity ±1% 기준으로 독립 검증. DP Attention ON/OFF 간 accuracy delta 차이를 정량화해 "환경 변화에 따른 정확도 안전성"을 보장한다.
  검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + {DP Attn ON/OFF} × {압축 기법} 교차 행렬 + 처리량·메모리·accuracy delta 3방향 결과.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: MultiRoundPrefillShareKVCompressionCodec (멀티-라운드 프리필 공유 KV 압축 코덱)
- **가설**: AMPD(2602.14516)의 멀티-라운드 KV 지연 읽기 + PrefillShare(2602.12029)의 공유 프리필 KV가 결합된 환경에서, "공유 KV는 여러 라운드·여러 모델이 반복 참조하므로 높은 압축률이 오히려 중요 정보 손실을 유발할 수 있다"는 관찰을 기반으로, "공유 KV는 낮은 압축률(정확도 중심) + 비공유 태스크-특화 KV는 높은 압축률(메모리 중심)"의 차등 압축 정책을 구현하면, 멀티-라운드·멀티-모델 환경에서 전체 정확도를 보존하면서 비공유 KV 메모리를 극대화 절감할 수 있다. 이전 C 기법 전체는 공유/비공유 KV를 구분하지 않고 단일 압축 정책을 적용했다.
- **접근 방법**:
  1. `PrefillShareAwareCompressionCodec` 클래스 (`src/cache/prefill_share_aware_compression.py`): `CompressionCodec` 인터페이스 구현
  2. **공유/비공유 KV 분류**:
     - 공유 KV: PrefillShare 방식으로 생성된 KV(공통 프롬프트, 시스템 컨텍스트, RAG 공통 청크에서 유래). 분류 태그: `kv_share_type = "shared"`.
     - 비공유 KV: 개별 요청 또는 멀티-라운드 증분 KV. 분류 태그: `kv_share_type = "private"`.
     - 분류 기준: 세그먼트 ID의 공유 플래그(PrefillShareFanoutRouter(A-2)가 팬아웃 배포 시 태그 설정) 또는 `n_references ≥ share_threshold`(기본 3회 이상 참조).
  3. **차등 압축 정책**:
     - 공유 KV (`kv_share_type = "shared"`): 낮은 압축 강도 적용. `GlobalRetentionGateEvictionCodec`의 전역 예산을 공유 KV에 대해 더 높게 설정(budget_ratio_shared = 0.6 vs. 기본 0.3). 양자화: FP8 또는 INT8(낮은 정보 손실 우선).
     - 비공유 KV (`kv_share_type = "private"`): 높은 압축 강도 적용. `FibQuantVQCodec` 10× 또는 `SPKVWriteTimeCodec` write-time 희소화. 멀티-라운드 증분 KV는 라운드 간 참조 빈도가 낮으므로 공격적 압축 허용.
  4. **멀티-라운드 참조 빈도 추적**:
     - 각 KV 세그먼트의 라운드 간 참조 횟수를 `n_references_per_round` 카운터로 추적. 카운터가 `share_threshold` 초과 시 자동으로 "공유 KV"로 재분류 + 낮은 압축으로 전환.
     - AMPD 지연 읽기 이벤트(세그먼트 pull 완료) 수신 시 카운터 증가.
  5. **이전 C 기법과의 차별점**:
     - 이전 모든 C 기법은 KV의 출처(공유 프리필 vs. 개별 요청)를 구분하지 않고 단일 압축 정책을 적용했다. PrefillShareAwareCompressionCodec은 "PrefillShare/AMPD 환경에서 공유 KV와 비공유 KV의 참조 패턴 차이"를 압축 강도 결정에 반영하는 최초의 C 기법이다. DPAttentionAwareCompressionSelector(C-1)가 "멀티 GPU 환경(DP Attention 상태)"을 정책 입력으로 사용한다면, 이 기법은 "KV의 공유 여부(PrefillShare 상태)"를 정책 입력으로 사용한다.
- **예상 효과**: 메모리 −55~80% (비공유 KV 고압축 + 공유 KV 저압축 균형), accuracy delta ±0.4% 이내 (공유 KV 정밀도 보존으로 멀티-라운드 누적 오류 방지), 멀티-라운드 추론 정확도 단독 C 기법 대비 +5~15% 향상 (공유 KV 정확도 보존 효과)
- **accuracy-preserving 근거**:
  (1) 공유 KV는 여러 라운드와 모델에서 반복 참조되므로 낮은 압축률(FP8/INT8 + 높은 retention budget)을 적용해 중요 정보 보존. 공유 KV 압축 오류가 멀티-라운드에 걸쳐 누적되지 않도록 제한.
  (2) 비공유 KV는 단일 라운드 내에서만 사용되므로 높은 압축률이 accuracy delta에 미치는 영향이 단일 라운드에 국한됨. 기존 FibQuant/SP-KV의 accuracy-preserving 보장(±0.5%)을 그대로 상속.
  (3) 멀티-라운드 참조 빈도 기반 동적 재분류가 압축 강도를 실제 참조 패턴에 맞게 자동 조정.
  검증: WikiText-2 perplexity ±1% + 멀티-라운드 대화 태스크(MT-Bench, ShareGPT) + 공유/비공유 KV 압축 비율별 accuracy delta + 라운드 수(2/4/8) 별 누적 정확도 변화.
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-1: AMPDPrefillShareIntegratedNonContiguousStack (A+B+C) — AMPD 지연 페치 + PrefillShare 팬아웃 + DP Attention 인식 압축 완전 통합 스택
- **조합 근거**:
  AMPDLazySegmentFetchScheduler(A-1)가 세그먼트 집합 확정 전까지 KV 전송을 보류하고, AMPDAdapShotLazyLoadPipeline(B-1)이 확정 후 로드-재인코딩을 비동기 오버랩하며, DPAttentionAwareCompressionSelector(C-1)가 DP Attention 상태에 따라 최적 압축 코덱을 선택한다. PrefillShareFanoutRouter(A-2)가 공유 세그먼트를 1-to-N 팬아웃 배포하고, PrefillShareAwareCompressionCodec(C-2)가 공유/비공유 KV에 차등 압축을 적용한다. 이 다섯 기법의 통합이 "지연 스케줄링(A-1) → 팬아웃 배포(A-2) → 지연 로드·재인코딩 파이프라인(B-1) → 환경 인식 압축(C-1) → 공유 인식 차등 압축(C-2)"이라는 완전한 멀티-라운드·멀티-모델 비연속 KV 재사용 스택을 형성한다.

  이전 크로스 아이디어 (HMAChainedACPipeline 05-17 Cross-1, NAtHRetentionTierDecider 05-16 Cross-1, LookaheadRelaySegmentCache 05-15 Cross-1 등)와 완전히 다른 A+B+C 통합 축이다: 이전 Cross-1들은 "압축 정책 플러그인 체이닝", "이중 신호 티어 결정", "레이어-선택 + 미래-인식 퇴거" 조합이었으나, 이번 Cross-1은 "지연 스케줄링 + 팬아웃 라우팅 + 로드-재인코딩 파이프라인 + 환경 인식 압축"의 시스템 수준 통합이다.
- **구체 설계**:
  1. `AMPDPrefillShareNonContiguousStack` 모듈 (`src/engine/ampd_prefill_share_stack.py`)
  2. **처리 흐름**:
     - **Step 1 (Segment Resolution + Metadata Pre-resolve)**: 요청 수신 시 Louver 탐색으로 히트 보장 세그먼트 메타데이터 수집. PrefillShareFanoutRouter가 공유 세그먼트 식별 + 팬아웃 계획 수립. 이 단계에서 KV 데이터 전송 없음 (지연 페치 원칙).
     - **Step 2 (Fanout Distribution)**: 공유 세그먼트 KV를 N개 디코딩 노드에 1-to-N 팬아웃 배포. PrefillShareAwareCompressionCodec(C-2)가 배포 전 공유 KV에 낮은 압축 강도 적용(FP8/INT8).
     - **Step 3 (Lazy Load + Reencoding Pipeline)**: 스케줄 확정 후 AMPDLazySegmentFetchScheduler가 비공유 세그먼트를 pull. AMPDAdapShotLazyLoadPipeline이 로드-AdapShot 재인코딩을 비동기 오버랩. PrefillShareAwareCompressionCodec(C-2)가 비공유 KV에 높은 압축 강도 적용.
     - **Step 4 (DP Attention Aware Compression)**: DPAttentionAwareCompressionSelector(C-1)가 현재 환경(DP Attention ON/OFF, n_gpus)에 따라 추가 압축 코덱 선택 및 적용.
     - **Step 5 (Non-Contiguous Attention Compute)**: 재조합된 세그먼트 KV를 어텐션 연산에 투입.
  3. **모니터링**: 각 단계의 {세그먼트 메타데이터 선행 전달 비율, 팬아웃 배포 완료 지연, 로드-재인코딩 오버랩 효율, 공유/비공유 압축 비율, accuracy delta}를 `results/<exp-name>/metrics.json`에 단계별 기록.
  4. **기존 파이프라인과의 결합**: HMAChainedACPipeline(05-17 Cross-1, 기구현 예정)의 HMA 멀티-커넥터 레지스트리에 C-1, C-2를 추가 커넥터로 등록해 기존 파이프라인 확장. 새 스택은 독립 모듈로 실행하거나 HMA 커넥터로 등록해 선택.
- **예상 복합 효과**: 처리량 +35~55% (지연 스케줄링 + 팬아웃 병렬화 + 로드-재인코딩 오버랩 복합), 메모리 −70~95% (DP Attention + 공유/비공유 차등 압축 곱 효과), accuracy delta ±0.6% 이내 (공유 KV 정밀도 보존 + 환경 인식 압축 강도 제어)
- **구현 난이도**: high
- **우선순위**: 1

### 아이디어 Cross-2: LazyFetchDPAttnCompressionABPipeline (A+C) — 지연 페치 스케줄러 + DP Attention 인식 압축 A+C 통합
- **조합 근거**: AMPDLazySegmentFetchScheduler(A-1)가 세그먼트 확정 후 pull하는 시점에 DPAttentionAwareCompressionSelector(C-1)가 즉시 최적 압축 코덱을 선택·적용하면, "세그먼트 로드 완료 즉시 압축 → 압축된 KV를 HBM에 저장"하는 파이프라인이 가능하다. 이전 HMAChainedACPipeline(05-17 Cross-1)이 "HMA 커넥터 플러그인 체이닝 + DCP/PCP 이벤트 피드백"이라는 A+C 통합이었다면, 이 기법은 "지연 페치 + 환경 인식 압축"이라는 완전히 다른 A+C 시너지 축이다. 이전 18개 사이클 어디에도 이 조합은 없다.
- **구체 설계**:
  1. `LazyFetchCompressionPipeline` 모듈 (`src/engine/lazy_fetch_compression_pipeline.py`)
  2. AMPDLazySegmentFetchScheduler(A-1)의 `fetch_segments_lazy()` 스트림에서 세그먼트가 도착할 때마다 DPAttentionAwareCompressionSelector(C-1)의 `compress(kv, env_state)` 호출.
  3. 환경 상태(DP Attention ON/OFF, n_gpus)는 스택 초기화 시 1회 캐싱 후 재사용(런타임 조회 오버헤드 제거).
  4. 압축된 KV를 HBM에 저장 시 `HMASegmentTierCache(05-17 B-1 기구현)`의 `hot_threshold` 기준으로 즉시 티어 배정.
- **예상 복합 효과**: 처리량 +20~35%, 메모리 −65~90% (지연 페치로 불필요 KV 제거 + DP Attention 인식 압축), accuracy delta ±0.5% 이내
- **구현 난이도**: low
- **우선순위**: 2

## 이번 사이클 최우선 구현 타겟
**A-1 (AMPDLazySegmentFetchScheduler)** 와 **B-1 (AMPDAdapShotLazyLoadPipeline)** 및 **C-1 (DPAttentionAwareCompressionSelector)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. A-1(AMPDLazySegmentFetchScheduler)은 AMPD pull-on-demand 원칙을 비연속 세그먼트 스케줄러에 최초 적용하는 기법으로, 구현 난이도 medium이고 기구현된 Louver 탐색 + SegmentedHashCache와 바로 통합 가능하다. 불필요 KV 전송 −40~65% 효과를 `unnecessary_transfer_ratio` 지표로 즉시 정량화할 수 있어 Activity A 목표 지표(Scheduling overhead +5% 이내)와 직결된다.
2. B-1(AMPDAdapShotLazyLoadPipeline)은 A-1의 "확정 후 pull" 이벤트를 받아 로드-AdapShot 재인코딩을 비동기 오버랩하는 자연스러운 A+B 통합이다. 기구현 AdapShot RoPE 재인코딩 로직을 파이프라인 Stage 3에 재활용할 수 있어 구현 재사용률이 높다. Activity B 목표 지표(비연속 히트율 30% 이상)에 직접 기여.
3. C-1(DPAttentionAwareCompressionSelector)은 단일/멀티 GPU 환경 모두에서 압축 정책을 자동 최적화하는 새로운 평가 차원을 도입한다. 구현 난이도 medium으로, 기구현된 C 코덱들(GlobalRetentionGateEvictionCodec, FibQuantVQCodec, SPKVWriteTimeCodec)을 환경 인식 래퍼로 재활용할 수 있다. Activity C 목표 지표(KV Cache Memory Reduction −30% 이상, Compression Accuracy Delta ±1% 이내)를 환경별로 정량화.
4. Cross-1(AMPDPrefillShareNonContiguousStack)는 A-1, A-2, B-1, C-1, C-2의 통합으로, 각 컴포넌트 완성 후 조립 단계에서 구현. A+B+C 복합 효과(처리량 +35~55%, 메모리 −70~95%)가 목표 지표 달성 가능성이 가장 높다.
5. A-2(PrefillShareFanoutRouter)는 멀티 노드 인프라 가정이 필요해 단일 노드 테스트 환경에서는 모킹(mocking) 기반 단위 테스트로 우선 검증 후 순차 구현 권장. C-2(PrefillShareAwareCompressionCodec)는 A-2 완성 후 공유 KV 태깅 인터페이스를 전달받아 구현.

## planner에게 전달할 요청사항
- **A-1(AMPDLazySegmentFetchScheduler) 구현 시**: `src/scheduler/ampd_lazy_segment_fetch.py` 신규 생성. `BaseScheduler` 상속. `SegmentMetadataRegistry` 클래스: `Dict[segment_id → SegmentMeta(source_node, tier, approx_size)]`, gRPC 스트림 기반 노드 간 동기화. `fetch_segments_lazy(confirmed_segment_ids: List[str]) → AsyncIterator[KVSegment]` 비동기 인터페이스. Louver `HalfspaceRangeSearchKVIndex.search()` 결과를 "스케줄 확정" 이벤트로 연결. `unnecessary_transfer_ratio` 지표를 `results/<exp-name>/metrics.json`에 기록. **스케줄링 오버헤드 검증 필수**: 메타데이터 선행 전달 오버헤드 < 0.1ms/요청, 기구현 KVRMStaticGraphCoalescingSegmentScheduler(05-16 A-1) 대비 전송 오버헤드 비교 + TTFT p50 +3% 이내 검증.
- **B-1(AMPDAdapShotLazyLoadPipeline) 구현 시**: `src/cache/ampd_adapshot_lazy_pipeline.py` 신규 생성. `CacheStore` 인터페이스 상속. `resolve_segments(query_tokens) → List[SegmentMeta]` Stage 1 인터페이스. `asyncio.Event` 기반 세그먼트별 로드 완료 신호. `torch.cuda.Stream` 3-way 파이프라인(fetch + reencoding + compute). AdapShot RoPE 재인코딩: `torch.einsum` 기반 배치 회전, `Δθ = target_position - source_position` YAML 외부화. 예측 프리페치를 위한 동반 세그먼트 통계 수집: `companion_segment_stats.json`. **비연속 히트율 검증 필수**: 지연 로드 + 오버랩 적용 전/후 비연속 재조합 지연 비교 + 로드-재인코딩 오버랩 효율(이론 최솟값 대비 달성률) 측정.
- **A-2(PrefillShareFanoutRouter) 구현 시**: `src/scheduler/prefill_share_fanout_router.py` 신규 생성. 공유 세그먼트 감지기: 배치 내 공통 비연속 세그먼트 집합 탐지 (SimHash 64비트 기반 유사도). 팬아웃 계획 `FanoutPlan = Dict[node_id → List[segment_id]]`. RDMA P2P Write (N ≤ 8) / tree-cast (N > 8) 전송 구현 또는 모킹. `kv_share_type` 태그 설정으로 C-2와 인터페이스 연결. **처리량 검증 필수**: 1-to-N 팬아웃 오버헤드 측정 + 중복 프리필 제거 효과 정량화.
- **C-1(DPAttentionAwareCompressionSelector) 구현 시**: `src/cache/dp_attention_aware_compression.py` 신규 생성. `CompressionCodec` 인터페이스 구현. `n_gpus = torch.cuda.device_count()` 자동 감지. `dp_attn_enabled` 환경 변수 또는 설정 파일 읽기. `effective_kv_replicas` 계산 로직. 기구현 코덱 3종(GlobalRetentionGateEvictionCodec, FibQuantVQCodec, SPKVWriteTimeCodec) 래핑. 실험 행렬 {DP Attn ON/OFF} × {압축 기법} × {n_gpus: 1/2/4/8} 결과를 `results/<exp-name>/dp_attn_compression_matrix.json`에 기록. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 환경별 압축 기법 accuracy delta 교차 검증 + DP Attention ON/OFF 간 정확도 차이 정량화.
- **C-2(PrefillShareAwareCompressionCodec) 구현 시**: `src/cache/prefill_share_aware_compression.py` 신규 생성. `kv_share_type` 태그 기반 공유/비공유 KV 분류. `n_references_per_round` 카운터 추적. 공유 KV `budget_ratio_shared=0.6`, 비공유 KV `budget_ratio_private=0.3` YAML 외부화. **accuracy-preserving 검증 필수**: 멀티-라운드 대화 태스크(MT-Bench) perplexity ±1% + 라운드별 누적 accuracy delta + 공유/비공유 압축 강도 균형 실험.
- **Cross-1(AMPDPrefillShareNonContiguousStack) 구현 시**: `src/engine/ampd_prefill_share_stack.py` 신규 생성. 5단계 처리 흐름 구현 (위 설명 참조). HMAChainedACPipeline(05-17 Cross-1 기구현 예정)의 커넥터 레지스트리와 통합 경로 설계. **복합 효과 측정 필수**: 처리량/메모리/accuracy delta 3방향 비교 — A-1 단독 + B-1 단독 + C-1 단독 + 결합 Cross-1 + 기구현 Cross-1(HMAChainedACPipeline).
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (A-1 지연 페치 + B-1 로드-재인코딩 오버랩 + C-1 환경 인식 압축 복합 처리량 +20% 이상 목표)

  - KV Cache Memory Reduction (C-1 DP Attention × 압축 곱 효과 −70% 이상 목표)
