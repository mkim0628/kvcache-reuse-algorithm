### 아이디어 A-1: KVRMStaticGraphCoalescingSegmentScheduler (정적 그래프 인식 비연속 KV 합체 전송 스케줄러)
- **가설**: CUDA Graph 또는 Triton 정적 커널을 사용하는 LLM 서빙 환경에서 비연속 KV 세그먼트 재사용 시 단편화된 블록들이 각각 개별 전송되어 랜덤 I/O가 발생하는 문제를, KV-RM(2605.09735)의 "논리적 KV 이력과 물리적 스토리지의 분리 + 병합-단계 비연속 KV 합체 전송(merge-staged coalescing transport)"을 비연속 세그먼트 재조합 경로에 적용하면, 단편화된 비연속 세그먼트 블록들을 대형 전송 그룹으로 합체해 랜덤 I/O를 대용량 순차 전송으로 변환하며, 기존 개별 블록 전송 대비 비연속 세그먼트 재사용 전송 오버헤드를 50~70% 감소시킬 수 있다.
- **접근 방법**:
  1. `KVRMCoalescingSegmentScheduler` 클래스 (`src/scheduler/kvrm_coalescing_segment.py`): `BaseScheduler` 상속
  2. **논리-물리 KV 분리 레이어**: 현재 `src/cache/segmented.py`의 세그먼트 블록 매핑을 논리 ID(세그먼트 내용 해시)와 물리 스토리지 위치(GPU 페이지 번호)로 분리하는 블록 페이저(block pager) 추가. 비연속 세그먼트 집합 `{seg_1, seg_2, ..., seg_k}`에 대해 물리 주소 목록을 수집하는 `resolve_physical_addresses(segment_ids)` 인터페이스.
  3. **병합-단계 합체기(Intra-commit Coalescer)**: 재사용 요청 수신 시 필요한 비연속 세그먼트 블록들의 물리 주소를 수집 → 인접하거나 정렬 가능한 블록들을 "합체 전송 그룹(coalesced transfer group)"으로 병합 → 단일 대형 DMA 전송 실행. 합체 기준: 물리 주소 간격이 `coalesce_threshold`(기본 4 페이지 = 32KB) 이내인 블록들을 하나의 그룹으로 통합.
  4. **정적 그래프 레이아웃 정규화**: CUDA Graph 재플레이 시 비연속 블록 합체 전송이 완료된 KV를 정적 그래프가 기대하는 연속 텐서 레이아웃으로 재배치하는 "in-place 레이아웃 정규화기"(`normalize_kv_layout(coalesced_kv, target_static_shape)`) 추가.
  5. **기존 `src/cache/segmented.py`와 통합**: `SegmentedHashCache.fetch_segments_coalesced(segment_ids)` 인터페이스 추가. 기존 개별 페치(`get(segment_id)`) 경로는 유지 (호환성). 합체 경로를 YAML 파라미터(`use_coalescing=true`)로 선택 가능.
  6. **이전 A 기법과의 차별점**: 이전 16개 사이클의 모든 A 기법이 "어떤 요청을 어떤 노드/파티션으로 라우팅할 것인가"에 집중했다면, 이 기법은 "선택된 세그먼트 블록들을 어떻게 물리적으로 전송할 것인가"의 전송 계층 최적화를 최초로 도입한다. 특히 정적 그래프 LLM 서빙이라는 새로운 환경 가정(dynamic shape 없는 CUDA Graph 실행 환경)에서의 비연속 KV 합체 전송을 처음으로 다룬다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽

### 아이디어 A-2: NAtHDDROffloadingMinimalEvictionScheduler (NAtH 의미-인식 DDR 오프로딩 + 최소 퇴거 스케줄러)
- **가설**: 기존 KV 퇴거 기반 메모리 관리 스케줄러가 추론(reasoning) 워크로드에서 3% 퇴거만으로도 정확도가 0~2.5%로 붕괴하는 한계를, NAtH(2605.09490)의 이론적 발견 — "정확도는 영구 퇴거 비율에만 의존하며 DDR 오프로딩은 제로 근사 오류 오프로딩(zero-approximation-error offloading)으로 정확도에 영향 없다" — 을 스케줄러 설계 원칙으로 적용하면, 메모리 압박 시 영구 퇴거 대신 DDR 오프로딩을 기본 정책으로 사용하고 누적 어텐션 점수(cumulative attention score)로 4-티어 메모리 분류를 수행하는 "최소-퇴거 DDR 오프로딩 스케줄러"를 구성할 수 있으며, Activity A의 "캐시 퇴거율 최소화" 목표와 Activity C의 "정확도 ±1% 이내" 목표를 스케줄러 수준에서 동시에 달성할 수 있다.
- **접근 방법**:
  1. `NAtHDDROffloadingScheduler` 클래스 (`src/scheduler/nath_ddr_offloading.py`): `BaseScheduler` 상속
  2. **4-티어 메모리 분류 정책**: 각 KV 토큰에 대해 누적 어텐션 점수 `cumulative_attn_score(token_i) = Σ_t attn(query_t, key_i)` 추적. 점수를 기준으로 실시간 4-티어 분류:
     - **Tier 1 (HBM 즉시 접근)**: 상위 `p1` 백분위 — 최고 중요도, GPU HBM에 FP16 유지
     - **Tier 2 (DDR 프리페치)**: `p1 ~ p2` 백분위 — 중간 중요도, CPU DDR로 오프로딩, 어텐션 스텝 전 전체 정밀도로 비동기 프리페치
     - **Tier 3 (압축 보존)**: `p2 ~ p3` 백분위 — 저중요도, DDR에 INT8로 압축 보존
     - **Tier 4 (영구 퇴거)**: 하위 `1 - p3` 백분위 — 매우 낮은 중요도, 영구 제거 (퇴거 비율 ≤ `max_eviction_ratio`, 기본 3%)
  3. **퇴거 비율 엄격 제약**: `max_eviction_ratio = 0.03` (기본값) 파라미터로 영구 퇴거를 전체 KV의 3% 미만으로 강제. 메모리 압박이 심할 경우 Tier 2 DDR 오프로딩 비율을 먼저 높임.
  4. **제로 근사 오류 DDR 오프로딩 구현**: Tier 2 DDR 오프로딩 토큰은 `torch.cuda.Stream` 비동기 전송으로 CPU DDR → GPU HBM 프리페치. 프리페치된 토큰이 완전 정밀도로 어텐션 계산에 참여하므로 HBM에 있었을 경우와 수치적으로 동일.
  5. **누적 어텐션 점수 EMA 추적**: 매 디코딩 스텝마다 `cumulative_attn_score` 지수이동평균(EMA, α=0.95)으로 갱신. 오래된 토큰의 점수가 자연적으로 감소해 시간적 중요도 반영.
  6. **`src/cache/temporal_tiered_kv.py`(기구현 TemporalTieredKVManager)와의 차별점**: TemporalTieredKVManager가 시간적 거리(접근 age)를 계층화 기준으로 사용한 반면, NAtHDDROffloadingScheduler는 누적 어텐션 점수(의미적 중요도)를 기준으로 하고, 영구 퇴거 비율을 3%로 엄격히 제한하며, DDR 오프로딩이 "제로 근사 오류"임을 보장하는 이론적 기반을 갖춘다. 이전 16개 사이클의 모든 A 기법이 메모리 압박 시 퇴거를 기본 정책으로 사용하거나 오프로딩 시 정확도 근사 오차를 허용한 것과 근본적으로 다르다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드에서 Tier 2(DDR) 오프로딩을 로컬 CPU DDR로 우선 처리하고, 로컬 DDR 포화 시 원격 노드 DDR로 RDMA 확장. 노드 간 누적 어텐션 점수는 마스터 노드에서 집계.
- **예상 효과**: 추론 워크로드 정확도 보존 — 영구 퇴거 비율 3% 유지 시 GSM8K 91% 정확도 보존 (NAtH 실증 기반), KV Cache Memory Reduction −60~80% (DDR 오프로딩으로 HBM 점유 해소), Scheduling overhead TTFT +4% 이내 (누적 어텐션 점수 계산 오버헤드: 디코딩 스텝당 O(n_tokens) 덧셈)
- **구현 난이도**: medium
- **우선순위**: 1

## Activity B 아이디어

### 아이디어 B-1: SPKVStructuralSparsityHeadSelectiveSegmentReusePolicy (SP-KV 구조적 희소성 프로파일 기반 헤드-선택적 비연속 세그먼트 재사용 정책)
- **가설**: 비연속 KV 세그먼트 재사용 시 현재 모든 레이어·헤드를 동등하게 처리하는 all-or-nothing 정책이 불필요한 재사용(희소한 헤드의 KV를 재사용해 근사 오류 유발)이나 불필요한 재계산(밀도 높은 헤드의 KV를 버리고 재계산)을 초래하는 문제를, SP-KV(2605.14037, Meta FAIR)의 write-time 경량 효용 예측기가 학습 완료 후 자연스럽게 드러내는 레이어·헤드별 구조적 희소성 패턴(structured sparsity pattern)을 활용하면, "구조적으로 희소한 레이어·헤드(KV 기록 빈도가 낮음)는 재사용 대신 재계산이 더 저렴하고, 구조적으로 밀도 높은 레이어·헤드(KV 기록 빈도가 높음)는 재사용이 필수적이다"는 데이터-기반 재사용 정책을 수립하여 비연속 세그먼트 히트율을 높이면서 재사용-유발 정확도 손실을 최소화할 수 있다.
- **접근 방법**:
  1. `SPKVSparsityProfiledSegmentReusePolicy` 클래스 (`src/cache/spkv_sparsity_segment_reuse.py`): `CacheStore` 인터페이스 상속
  2. **SP-KV 구조적 희소성 프로파일 수집 (오프라인 1회)**:
     - 보정 데이터셋(100~200 시퀀스)에서 각 레이어 l, 헤드 h에 대해 "기록 결정 비율(write rate)" = SP-KV 효용 예측기가 KV를 실제로 기록하기로 결정한 비율을 측정.
     - `write_rate[l][h] = (기록된 KV 수) / (전체 토큰 수)` 계산.
     - `write_rate[l][h] ≥ density_threshold`(기본 0.6): 밀도 높은 헤드 → "재사용 필수" 분류
     - `write_rate[l][h] < density_threshold`: 구조적으로 희소한 헤드 → "재계산 선호" 분류
     - 결과를 `configs/spkv_head_density_profile.yaml`에 저장.
  3. **헤드-선택적 세그먼트 재사용 인터페이스**:
     - `load_segment_for_reuse(segment_id, layer, head)`: 해당 레이어·헤드가 "재사용 필수" 분류이면 캐시에서 KV 로드, "재계산 선호"이면 None 반환 (호출자가 재계산 트리거).
     - `get_reuse_mask(layer_count, head_count) → 2D bool tensor`: 재사용 가능한 레이어·헤드 비트마스크를 반환하는 인터페이스.
  4. **기존 `src/cache/segmented.py`와 통합**: `SegmentedHashCache`를 상속하되 `get()` 호출 시 희소성 프로파일 확인 후 "재사용 필수" 헤드 KV만 반환. "재계산 선호" 헤드는 빈 텐서 반환 + 재계산 플래그 설정.
  5. **이전 B 기법과의 차별점**: 기존 모든 B 기법(RelayUShapeLayerSelectiveSegmentCache 포함)은 코사인 유사도 기반 사후(post-hoc) 측정으로 레이어 범위를 결정했다. SPKVSparsityProfiledSegmentReusePolicy는 SP-KV의 write-time 효용 예측기가 사전(a-priori)으로 학습한 구조적 희소성 패턴을 사용하므로 재사용 결정이 추론 시 계산 없이 미리 결정된다. 또한 RelayUShapeLayerSelectiveSegmentCache가 레이어 단위 결정을 내리는 반면, 이 기법은 레이어 × 헤드 2차원 결정을 내린다.
- **예상 효과**: 비연속 히트율 +20~35%p (밀도 높은 헤드 KV 재사용 극대화), 재사용-유발 정확도 손실 −60~80% (희소 헤드 재계산으로 근사 오류 제거), TTFT −30~50% (밀도 높은 레이어·헤드 재사용으로 재계산 비용 절감)
- **구현 난이도**: medium
- **우선순위**: 1

## Activity C 아이디어

### 아이디어 C-1: SPKVWriteTimeUtilityPredictorCodec (SP-KV Write-time 효용 예측 KV 희소화 코덱)
- **가설**: 기존 모든 Activity C 기법(LaProx, LookaheadKV, LKV, RateQuant, FibQuant 등)이 이미 기록된 KV를 사후에 퇴거·압축하는 "post-write eviction" 패러다임에 의존하는 한계를, SP-KV(2605.14037, Meta FAIR)의 "경량 효용 예측기로 각 KV 쌍의 미래 유용성을 예측해 기록 여부를 사전에 결정하는 write-time sparsification"으로 대체하면, 불필요한 KV가 처음부터 캐시에 쓰이지 않아 사후 퇴거 오버헤드를 제거하면서 3~10× 동적 희소화를 달성하고, 기존 post-write 퇴거와 상보적으로 결합해 최대 KV 절감을 이룰 수 있다.
- **접근 방법**:
  1. `SPKVWriteTimeCodec` 클래스 (`src/cache/spkv_write_time_codec.py`): `CompressionCodec` 인터페이스 구현
  2. **경량 효용 예측기 구조**:
     - 각 레이어·헤드당 경량 2층 MLP 효용 예측기 `U_{l,h}`: 입력 = `[k_i, v_i, position, layer_id, head_id]`, 출력 = KV 기록 효용 점수 `u_i ∈ [0, 1]`.
     - 슬라이딩 로컬 윈도우: 최근 `local_window_size`(기본 256) 토큰은 효용 점수와 무관하게 항상 기록 (최근 컨텍스트 접근성 보장).
     - 장기 KV 캐시 기록 결정: `u_i ≥ utility_threshold`(기본 0.3)인 경우에만 장기 KV 캐시에 기록. 미달 시 토큰 KV를 폐기.
  3. **엔드-투-엔드 공동 훈련 (파인튜닝 단계)**:
     - LLM 파라미터와 효용 예측기를 다음-토큰 예측 손실만으로 공동 학습. 추가 감독 신호 불필요.
     - 훈련 비용: SP-KV 원논문 기반 추정 5~10B 토큰 데이터 (소규모 파인튜닝 설정: 1B 토큰으로 경량화). `experiments/train_spkv_predictor.py` 스크립트.
     - 파인튜닝 불가 환경 대비 훈련-프리 근사: 보정 데이터에서 각 토큰의 미래 어텐션 점수를 측정해 `utility_threshold` 직접 설정하는 휴리스틱 모드(`heuristic_mode=True`) 지원.
  4. **동적 희소화**: 입력 시퀀스 길이가 길수록 더 많은 KV를 폐기하는 적응적 동작. 짧은 시퀀스(< 512 토큰)는 모든 KV 기록, 긴 시퀀스(> 4096 토큰)는 효용 예측기 활성화.
  5. **기존 C 기법과의 상보적 결합**:
     - `SPKVWriteTimeCodec` → `LookaheadKVFutureAwareLoRAEvictionCodec`(기구현) 이중 파이프라인: SP-KV가 불필요한 KV를 처음부터 쓰지 않고, LookaheadKV가 이미 쓰인 KV를 미래 패턴으로 퇴거. 두 방향의 결합으로 최대 KV 절감.
     - `blend_mode` 파라미터로 write-time 희소화 단독 / 사후 퇴거 단독 / 결합 선택 가능.
  6. **이전 C 기법과의 차별점**: 이전 모든 C 기법(LaProx, LookaheadKV, LKV, FibQuant, MixedDimKV, FibQuantVQCodec, ABSparseHeadBlockProfiler, FluxionSparseRestoreCodec 등)이 "이미 캐시에 기록된 KV를 언제 버릴 것인가(post-write eviction)"를 다루는 반면, SPKVWriteTimeCodec은 "KV를 처음부터 캐시에 쓸 것인가(write-time selection)"라는 상보적 관점으로 최초 도입. 이전 16개 사이클 전체에 이 패러다임은 존재하지 않았다.
- **예상 효과**: 메모리 −66~90% (3~10× 동적 희소화, SP-KV 원논문 실증), accuracy delta ±0.5% 이내 (다운스트림 태스크 성능 저하 거의 없음, Meta FAIR 실증), 퇴거 사후 처리 오버헤드 0 (애초에 쓰지 않으므로 퇴거 연산 불필요)
- **accuracy-preserving 근거**: SP-KV(2605.14037, Meta FAIR 8인 연구팀)가 긴 시퀀스에서 3~10× 동적 희소화 달성하면서 검증 손실(validation loss) 및 다양한 다운스트림 태스크 성능 저하가 거의 없음을 실증. 엔드-투-엔드 공동 훈련이 다음-토큰 예측 손실을 직접 최소화하므로 효용 예측기의 KV 폐기 결정이 모델 출력 품질을 보존하는 방향으로 학습됨. 로컬 슬라이딩 윈도우(최근 256 토큰 항상 보존)가 단기 컨텍스트 접근성을 보장해 생성 품질 하한을 유지. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 시퀀스 길이(512/1K/4K/8K) 별 희소화 비율 + accuracy delta 곡선. 훈련-프리 휴리스틱 모드(보정 데이터 기반 threshold 설정)도 동일 perplexity 검증 적용.
- **구현 난이도**: high (파인튜닝 포함 시) / medium (훈련-프리 휴리스틱 모드 단독)
- **우선순위**: 2

### 아이디어 C-2: GlobalRetentionGateCrossLayerEvictionCodec (전역 리텐션 게이트 크로스-레이어 경쟁 퇴거 코덱)
- **가설**: 기존 KV 퇴거 기법이 레이어별 또는 헤드별로 독립 퇴거 결정을 내려 "레이어 A에서 중요한 토큰이 레이어 B에서 퇴거되는" 비일관성이 발생하는 문제를, "Make Each Token Count"(2605.09649, Yale+CUHK)의 전역 리텐션 기반 KV 퇴거(global retention-based KV eviction) — 경량 리텐션 게이트로 각 KV 항목의 미래 효용 점수를 할당하고, 공유 최종 점수 투영(shared final score projection)으로 모든 레이어·헤드에 걸쳐 단일 전역 퇴거 정책을 실현해 서로 다른 레이어·헤드·모달리티의 토큰들이 공유 메모리 예산 안에서 직접 경쟁해 최적 퇴거 집합을 결정 — 을 도입하면, 레이어별 독립 퇴거(LaProx 기구현)와 달리 전역 메모리 예산 내 최적 토큰 선택을 달성하며, 긴 컨텍스트에서 전체 캐시를 초과하는 성능까지 달성할 수 있다.
- **접근 방법**:
  1. `GlobalRetentionGateEvictionCodec` 클래스 (`src/cache/global_retention_gate_eviction.py`): `CompressionCodec` 인터페이스 구현
  2. **경량 리텐션 게이트 구조**:
     - 각 KV 항목 `(token_i, layer_l, head_h)`에 학습 가능한 리텐션 게이트 스코어 `r_{i,l,h} = sigmoid(W_r · [k_{i,l,h}, v_{i,l,h}])`. 게이트 파라미터 `W_r`은 레이어·헤드별 독립.
     - 계산 오버헤드 최소화: `W_r`은 `d_model × 1` 선형 투영 (파라미터 수 = n_layers × n_heads × d_model, 매우 경량).
  3. **공유 최종 점수 투영(Shared Final Score Projection)**:
     - 모든 레이어·헤드의 리텐션 게이트 스코어를 단일 공유 투영 `W_final ∈ R^{n_layers × n_heads → 1}`으로 결합해 토큰별 최종 전역 리텐션 점수 `R_i = W_final · [r_{i,1,1}, r_{i,1,2}, ..., r_{i,L,H}]` 계산.
     - 전역 리텐션 점수로 모든 레이어·헤드가 공유하는 단일 퇴거 랭킹 결정.
  4. **전역 메모리 예산 내 경쟁 퇴거**:
     - 전체 KV 예산 `total_budget`(기본 = 최대 컨텍스트 길이 × 0.3)을 설정.
     - 전역 리텐션 점수 `R_i` 상위 `total_budget`개 토큰만 보존. 나머지 퇴거.
     - 레이어별 독립 예산 없음 — 전체 레이어·헤드가 동일 예산 풀에서 경쟁.
  5. **파인튜닝 경량화**: 리텐션 게이트 파라미터(`W_r`)와 공유 투영(`W_final`)만 소규모 보정 데이터(500~1000 샘플)에서 3~5 에포크 파인튜닝. 기존 LLM 가중치 동결. 훈련 비용 < 0.5 GPU-hour.
  6. **LaProxOutputAwareLayerEviction(기구현)과의 앙상블 옵션**: `ensemble_ratio` 파라미터로 LaProx 출력-인식 레이어별 점수 + 전역 리텐션 게이트 점수 혼합. 두 신호의 보완적 특성(레이어별 정밀도 + 전역 경쟁)을 결합하는 2단계 퇴거 파이프라인.
  7. **이전 C 기법과의 차별점**: LaProxOutputAwareLayerEviction(05-11 기구현)이 출력 행렬곱 근사 오차로 레이어별 독립 퇴거 결정을 내리고, LookaheadKVFutureAwareLoRAEvictionCodec(05-15 기구현)이 미래 응답 어텐션 패턴으로 요청별 독립 퇴거를 결정하는 반면, GlobalRetentionGateEvictionCodec은 공유 최종 점수 투영으로 모든 레이어·헤드 KV가 단일 전역 예산 풀에서 경쟁하는 새로운 설계 원칙을 최초로 도입한다. 결과적으로 "퇴거로 전체 캐시 성능을 초과할 수 있다"는 "Make Each Token Count"의 핵심 발견을 구현에 반영한다.
- **예상 효과**: 메모리 −50~80% (전역 예산 30~50% 유지), accuracy delta ±0 ~ −0.5% (긴 컨텍스트에서 오히려 +0~2% 정확도 향상 가능, "Make Each Token Count" 실증), 레이어 간 KV 보존 일관성 향상
- **accuracy-preserving 근거**: "Make Each Token Count"(2605.09649, Yale+CUHK)가 다양한 장문 컨텍스트 언어·비전-언어 추론 및 다중 대화 벤치마크에서 KV 메모리를 대폭 줄이면서 전체 캐시 추론과 동등하거나 초과하는 성능을 달성했다. 핵심 이론적 근거: 긴 컨텍스트에서 관련 없는 토큰들이 어텐션을 희석하므로 선택적 퇴거가 "노이즈 제거 효과"로 정확도를 향상시킬 수 있다. 전역 경쟁 퇴거가 레이어별 독립 퇴거 대비 전체 어텐션 품질을 더 정확히 보존하는 최적 토큰 집합을 선택한다. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 전역 예산(10~70%) 별 정확도 곡선 + LaProx 기준선과 동일 설정 비교.
- **구현 난이도**: medium
- **우선순위**: 1

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: NAtHRetentionGateOffloadingTierDecider (A+C) — 전역 리텐션 게이트 기반 DDR 오프로딩 티어 결정기
- **조합 근거**: NAtHDDROffloadingScheduler(A-2)가 누적 어텐션 점수로 4-티어 메모리 분류를 수행하고, GlobalRetentionGateEvictionCodec(C-2)의 전역 리텐션 게이트 점수가 더 정밀한 미래 효용 기반 분류를 제공한다. 두 신호를 결합하면 "누적 어텐션 점수(NAtH) + 전역 리텐션 게이트(Make Each Token Count)"의 이중 신호로 4-티어 메모리 분류를 더 정확히 수행하는 "리텐션-기반 오프로딩 티어 결정기"가 가능하다. NAtH의 핵심 발견("정확도는 영구 퇴거 비율에만 의존")이 "리텐션 게이트 임계값을 높여 Tier 2(DDR) 비율을 높이되 Tier 4(영구 퇴거) 비율을 3% 미만으로 유지"하는 최적 정책 설계의 수학적 기반이 된다. 이전 16개 사이클에서 A와 C를 결합한 경우(FluxionSparseRestoreCodec + SecurityAwareSegmentIsolationScheduler 등)는 있었으나, NAtH의 퇴거-최소화 이론 + 전역 리텐션 게이트 결합은 완전히 새로운 A+C 시너지다.
- **구체 설계**:
  1. `NAtHRetentionTierDecider` 모듈 (`src/scheduler/nath_retention_tier_decider.py`)
  2. **이중 신호 결합**: 각 토큰 i의 최종 티어 결정 점수 = `α × norm(cumulative_attn_score_i) + (1-α) × global_retention_score_i`. `α = 0.5` 기본값, YAML 외부화.
  3. **티어 경계 동적 조정**:
     - 영구 퇴거 비율이 `max_eviction_ratio`(3%) 초과 감지 시 Tier 3 → Tier 4 경계를 자동으로 높여 퇴거 비율 3% 유지.
     - HBM 압박 감지 시 Tier 1 → Tier 2 경계를 낮춰 DDR 오프로딩 비율 자동 증가.
  4. **GlobalRetentionGateEvictionCodec(C-2)에서 전역 리텐션 점수 조회**: `get_global_retention_score(token_ids) → tensor` 인터페이스로 실시간 조회.
  5. **NAtHDDROffloadingScheduler(A-2)와의 통합**: `NAtHDDROffloadingScheduler`의 누적 어텐션 점수를 보조 신호로, `NAtHRetentionTierDecider`를 주 티어 결정기로 설정하는 플러그인 구조.
- **예상 복합 효과**: 처리량 +20~35% (DDR 오프로딩으로 HBM 해방 + 전역 최적 퇴거로 노이즈 제거), 메모리 −70~85% (DDR 오프로딩 + 전역 경쟁 퇴거 결합), accuracy delta ±0% ~ +1% (영구 퇴거 3% 이하 + 전역 리텐션 최적 선택으로 노이즈 제거 효과)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: SPKVSparsitySegmentPolicyWithWriteTimeCodec (B+C) — SP-KV 구조적 희소성 패턴 기반 헤드-선택 세그먼트 재사용 + Write-time 희소화 통합
- **조합 근거**: SPKVSparsityProfiledSegmentReusePolicy(B-1)가 SP-KV 구조적 희소성 프로파일로 "어떤 레이어·헤드의 KV를 재사용할지" 사전에 결정하고, SPKVWriteTimeCodec(C-1)이 해당 레이어·헤드에서 효용 예측기로 "어떤 토큰 KV를 애초에 기록할지"를 결정한다. 두 SP-KV 기반 기법의 결합이 "기록 단계에서 불필요한 토큰 KV 폐기(C-1) + 재사용 단계에서 불필요한 헤드 KV 무시(B-1)"라는 이중 필터로 최대 KV 절감과 재사용 정확도를 동시에 달성하는 B+C 시너지다. 두 기법이 동일한 SP-KV 구조적 희소성 분석 결과를 공유하므로 프로파일링 비용이 1회로 공유된다.
- **구체 설계**:
  1. `SPKVDualFilterSegmentCache` 모듈 (`src/cache/spkv_dual_filter_segment.py`)
  2. **공유 구조적 희소성 프로파일**: SP-KV 프로파일링 결과(`configs/spkv_head_density_profile.yaml`)를 B-1과 C-1이 공유.
  3. **저장 단계**: `SPKVWriteTimeCodec`이 효용 예측기로 각 토큰 KV 기록 결정 → "재사용 필수" 헤드의 KV만 선택적으로 장기 캐시에 기록 (희소 헤드는 write-time에 이미 폐기).
  4. **재사용 단계**: `SPKVSparsityProfiledSegmentReusePolicy`가 "재사용 필수" 헤드 KV만 로드 → "재계산 선호" 헤드는 on-demand 재계산.
  5. **메모리 효율 극대화**: write-time 희소화(C-1, 3~10×) × 헤드 선택 재사용(B-1, 20~40% 헤드만 재사용) 이중 절감.
  6. 이전 B+C 크로스 아이디어(FibQuantPositionFreeSegmentCache, AdapShotMixedDimSegmentPipeline 등)와의 차별점: 이전 B+C 기법들이 "세그먼트 압축 + 위치 독립 재사용"에 집중했다면, 이 기법은 "write-time 폐기 + 헤드 선택 재사용"이라는 완전히 다른 B+C 시너지 축을 도입한다.
- **예상 복합 효과**: 처리량 +25~40%, 메모리 −75~90% (write-time 3~10× + 헤드 선택 20~40%), 비연속 히트율 +20~35%p (밀도 높은 헤드 재사용 극대화), accuracy delta ±0.5% 이내 (SP-KV 엔드-투-엔드 학습 기반)
- **구현 난이도**: high (파인튜닝 포함) / medium (훈련-프리 모드)
- **우선순위**: 2

### 아이디어 Cross-3: KVRMNAtHStaticGraphReasoningKVPipeline (A+B+C) — 정적 그래프 추론 LLM 완전 통합 KV 최적화 파이프라인
- **조합 근거**: 오늘 트렌드 리포트의 A+B+C 통합 시너지를 직접 구현한다. KVRMStaticGraphCoalescingSegmentScheduler(A-1)가 정적 그래프 환경에서 비연속 세그먼트 블록을 합체 전송하고, SPKVSparsityProfiledSegmentReusePolicy(B-1)가 구조적 희소성 프로파일로 헤드-선택 재사용을 수행하며, NAtHDDROffloadingScheduler(A-2)가 추론 CoT 토큰 KV를 DDR에 오프로딩하고, GlobalRetentionGateEvictionCodec(C-2)가 전역 경쟁 퇴거로 최적 토큰 집합을 선택하며, SPKVWriteTimeCodec(C-1)이 write-time 희소화로 불필요한 KV 기록을 애초에 방지하는 완전 통합 파이프라인. 특히 "정적 그래프 LLM + 추론 워크로드"라는 새로운 환경 조합(CUDA Graph 환경에서 CoT 생성)에서 A+B+C 통합 효과를 검증한다.
- **구체 설계**:
  1. `StaticGraphReasoningKVPipeline` 모듈 (`src/engine/static_graph_reasoning_pipeline.py`)
  2. 처리 순서:
     - **입력 프리필**: SPKVWriteTimeCodec이 write-time 희소화로 필요한 KV만 기록
     - **CoT 디코딩**: NAtHDDROffloadingScheduler가 CoT 토큰 KV를 4-티어로 분류 + DDR 오프로딩 (영구 퇴거 ≤ 3%)
     - **세그먼트 검색**: SPKVSparsityProfiledSegmentReusePolicy로 "재사용 필수" 헤드의 비연속 세그먼트 조회
     - **합체 전송**: KVRMStaticGraphCoalescingSegmentScheduler로 단편화 블록 합체 전송 → 정적 그래프 텐서 레이아웃으로 정규화
     - **전역 퇴거**: GlobalRetentionGateEvictionCodec이 HBM 내 KV에 전역 경쟁 퇴거 적용
  3. CUDA Graph 호환성: 정적 텐서 형상 유지를 위해 합체 전송 결과를 고정 크기 버퍼에 기록. 동적 KV 비율 변화는 마스크 텐서로 처리.
  4. 모니터링: 각 단계의 KV 기록 비율(C-1), 재사용 헤드 비율(B-1), DDR 오프로딩 비율(A-2), 합체 전송 효율(A-1), 전역 퇴거 비율(C-2)을 `results/<exp-name>/metrics.json`에 단계별 기록.
- **예상 복합 효과**: 처리량 +40~60%, 메모리 −80~92% (write-time 희소화 + DDR 오프로딩 + 전역 퇴거 복합), 비연속 히트율 +25~40%p, 추론 정확도 보존 (영구 퇴거 3% 이하), accuracy delta ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 3

## 이번 사이클 최우선 구현 타겟

**C-2 (GlobalRetentionGateEvictionCodec)** 과 **A-2 (NAtHDDROffloadingScheduler)** 및 **Cross-1 (NAtHRetentionTierDecider, A+C)** 를 최우선 구현 타겟으로 권장한다.

이유:
1. C-2(GlobalRetentionGateEvictionCodec)는 "Make Each Token Count"의 전역 경쟁 퇴거 원칙을 구현하며, 구현 난이도 medium으로 빠른 검증이 가능하다. 기구현된 LaProxOutputAwareLayerEviction, LookaheadKVFutureAwareLoRAEvictionCodec과 동일 설정에서 직접 비교하면 전역 vs. 레이어별 퇴거의 우위를 정량화할 수 있다. "퇴거로 전체 캐시를 초과하는 성능"이라는 새로운 accuracy-preserving 방향을 검증하는 가장 중요한 C 기법이다.
2. A-2(NAtHDDROffloadingScheduler)는 추론 워크로드의 정확도 보존 목표("±1% 이내")를 스케줄러 수준에서 이론적으로 보장하는 최초의 A 기법이다. 구현 난이도 medium으로 기구현된 TemporalTieredKVManager와 차별화된 새 계층화 기준을 즉시 통합 가능하다.
3. Cross-1(NAtHRetentionTierDecider)는 A-2와 C-2의 자연스러운 A+C 통합으로, 이중 신호(누적 어텐션 + 전역 리텐션)로 4-티어 결정의 정밀도를 높이는 가장 임팩트 높은 크로스 아이디어다.
4. B-1(SPKVSparsityProfiledSegmentReusePolicy)은 오프라인 프로파일링만으로 즉시 적용 가능하며, 기구현된 RelayUShapeLayerSelectiveSegmentCache와 보완적으로 결합 가능하다 (U자형 프로파일 대신 SP-KV write_rate 기반 프로파일).
5. C-1(SPKVWriteTimeCodec)은 파인튜닝이 필요해 구현 비용이 높지만 write-time 희소화라는 완전히 새로운 압축 패러다임을 도입하므로 이후 사이클에서 훈련-프리 휴리스틱 모드로 우선 검증 후 파인튜닝 버전을 순차 구현 권장.

## planner에게 전달할 요청사항
- **C-2(GlobalRetentionGateEvictionCodec) 구현 시**: `src/cache/global_retention_gate_eviction.py` 신규 생성. `CompressionCodec` 인터페이스 구현. 리텐션 게이트: `W_r ∈ R^{d_model×1}` per (layer, head) — `nn.Linear(d_model, 1)` 구조. 공유 투영: `W_final ∈ R^{n_layers × n_heads}` — 행 벡터로 단순 선형 결합. 파인튜닝: 보정 데이터 500~1000 샘플, 3~5 에포크, LLM 가중치 동결. 전역 예산 `total_budget` 파라미터 YAML 외부화. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 전역 예산(10%/20%/30%/50%/70%) 별 정확도 곡선. 기구현 LaProxOutputAwareLayerEviction(05-11) + LookaheadKVFutureAwareLoRAEvictionCodec(05-15)과 동일 설정 비교 필수. "전체 캐시 초과 성능" 구간(긴 컨텍스트 > 8K 토큰) 확인 포함.
- **A-2(NAtHDDROffloadingScheduler) 구현 시**: `src/scheduler/nath_ddr_offloading.py` 신규 생성. 누적 어텐션 점수 EMA: `torch.cuda.Event` 기반 비동기 어텐션 점수 수집, EMA α=0.95. DDR 전송: `torch.cuda.Stream` 비동기 (`torch.cuda.current_stream().wait_stream()`으로 어텐션 전 동기화). `max_eviction_ratio=0.03` 엄격 제약 파라미터 YAML 외부화. `tier_boundaries = [p1, p2, p3]` YAML 외부화 (기본 [0.3, 0.7, 0.97]). **정확도 검증 필수**: GSM8K + MATH-500 추론 태스크에서 퇴거 비율(1%/3%/5%/10%) 별 정확도 변화 측정. "정확도는 퇴거 비율에만 의존"하는 NAtH 이론 검증. 기구현 TemporalTieredKVManager(05-10)와 동일 설정 정확도 비교 필수.
- **B-1(SPKVSparsityProfiledSegmentReusePolicy) 구현 시**: `src/cache/spkv_sparsity_segment_reuse.py` 신규 생성. 오프라인 프로파일 스크립트: `experiments/run_spkv_write_rate_calibration.py`. 보정 데이터 100~200 시퀀스. `write_rate[l][h]` 행렬을 `configs/spkv_head_density_profile.yaml`에 저장. `density_threshold=0.6` YAML 외부화. `SegmentedHashCache` 상속, `load_segment_for_reuse(segment_id, layer, head)` 인터페이스 추가. **비연속 히트율 측정 필수**: 헤드 선택 ON/OFF 비교 + "재계산 선호" 헤드 재계산 비용 측정.
- **Cross-1(NAtHRetentionTierDecider) 구현 시**: `src/scheduler/nath_retention_tier_decider.py` 신규 생성. 이중 신호 결합 가중치 `α=0.5` YAML 외부화. `GlobalRetentionGateEvictionCodec.get_global_retention_score()` 인터페이스 연결. 영구 퇴거 비율 실시간 모니터링 + 자동 경계 조정 로직. **A+C 복합 효과 측정 필수**: 단독 A-2 + 단독 C-2 + 결합 Cross-1의 처리량/메모리/정확도 3방향 비교.
- **C-1(SPKVWriteTimeCodec) 구현 시** (이후 사이클 권장): `src/cache/spkv_write_time_codec.py` 신규 생성. 훈련-프리 휴리스틱 모드 우선 구현 (`heuristic_mode=True`): 보정 데이터에서 토큰별 미래 어텐션 점수 측정 → `utility_threshold` 직접 설정. 파인튜닝 모드는 별도 스크립트 (`experiments/train_spkv_predictor.py`). 로컬 윈도우 `local_window_size=256` YAML 외부화. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + 시퀀스 길이별 희소화 비율 곡선 + 기구현 LookaheadKVEvictionCodec(05-15)과 동일 설정 비교.
- **목표 지표 중 이번 사이클에서 측정할 지표**:
  - Inference Throughput (목표 +20% 이상)
  - KV Cache Memory Reduction (목표 −30% 이상, C-2 전역 리텐션 게이트 구현 시 −50~80% 목표)
  - Compression Accuracy Delta (필수: ±1% 이내 — 전역 예산별 정확도 곡선 측정)
  - Cache Eviction Rate (목표 최소화 — A-2 NAtH DDR 오프로딩으로 영구 퇴거 3% 이하 검증)
  - TTFT p50 (목표 Scheduling overhead +5% 이내)
- **Activity C 구현 시 공통 필수 사항**: 모든 C 아이디어는 `src/metrics/` 내 perplexity 측정 코드를 포함하고, 압축/퇴거 전후 perplexity를 `results/<exp-name>/metrics.json`에 함께 기록해야 한다. C-2(GlobalRetentionGateEvictionCodec)는 "전체 캐시 초과 성능" 구간 측정을 포함하고, LaProx + LookaheadKV 기구현 기법과 동일 설정 3방향 비교를 필수로 포함.
