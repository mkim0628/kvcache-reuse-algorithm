- **Activity B+C 결합의 새 차원**: R-KV 중복성 스코어를 세그먼트 퇴거 우선화에 적용하면 의미 중복 비연속 세그먼트(RAG 중복 문서 등)를 식별해 정확도 마진을 확보하면서 더 공격적으로 압축·퇴거할 수 있다. 이전 사이클의 DHD + TurboQuant 조합이 "유사도 기반 재사용 + 압축 저장"이라면, 이번은 "중복성 식별 기반 선택적 퇴거 + 압축"으로 방향이 다르다.

**A+B+C** — 워크플로우 DAG 인식 스케줄링(A) + 중복성 인식 비연속 세그먼트 관리(B) + 시스템-레벨 압축 공동 설계(C) 통합

**선택 이유**:
1. 이전 사이클(2026-05-03)이 B+C(SemanticSegmentCache+TurboQuantCodec)에 최우선을 두고 A는 DualMapSemanticRouter로 보완했다. 이번 사이클에서는 A 계층에 DAG 위상 인식을 새로 추가하면 B/C에서 확보된 메모리·히트율 이득이 A 스케줄링의 예측 정밀도 향상으로 연쇄 증폭된다.
2. DMS 8× 압축 + Harvest NVLink 피어 캐시의 A+C 조합이 이전 사이클에 없던 "클러스터 전체 유효 KV 캐시 용량 확장"이라는 새 목표를 제시하며, 처리량 +20% + 메모리 −30% 목표를 동시에 초과 달성하는 경로를 열어준다.
3. SideQuest 자기-지시형 퇴거를 B 세그먼트 선택에 통합하면 외부 휴리스틱 없이 모델 의미 이해를 활용해 비연속 히트율 30% 이상 목표의 정확도 마진을 확보할 수 있다.

## Activity A 아이디어

### 아이디어 A-1: SAGA+Pythia 워크플로우 DAG 위상 기반 KV 선제 보존 스케줄러
- **가설**: 에이전트 워크플로우의 DAG 구조(도구 호출 순서, 에이전트 의존 관계)를 스케줄러 메타데이터로 수집하고, DAG 위상 순서(topological order)에서 후속 노드가 현재 노드의 KV를 재사용할 확률을 Pythia 스타일 룩어헤드 예측기로 추정하면, Bélády 정책에 근접하는 선제 KV 보존 결정이 가능하다. 이는 이전 사이클의 GSA 경쟁비 스케줄러(요청 단위 경쟁비 보장)나 DualMap(노드 간 부하 균형)과 달리 에이전트 파이프라인 전체를 최적화 단위로 사용한다.
- **접근 방법**:
  1. `DAGTopologyScheduler` 클래스 (`src/scheduler/dag_topology_scheduler.py`): `BaseScheduler` 상속
  2. DAG 메타데이터 수집 인터페이스: `register_workflow(dag_id, node_graph)` — 각 노드는 `{agent_id, tool_calls, expected_kv_tokens, parent_ids}` 포함
  3. 위상 순서 분석: BFS/DFS로 DAG 순회 → 후속 노드 수(out_degree)와 KV 공유 확률(Pythia 스타일 소수 프로파일링으로 학습) 계산
  4. KV 보존 결정: 후속 노드 KV 공유 확률 > threshold_retain인 KV 세그먼트를 LRU 퇴거에서 제외(핀 고정)
  5. 룩어헤드 예측기: 워크플로우별 출력 토큰 수 분포를 경량 히스토그램으로 프로파일링(최소 10회 실행 후 예측 개시)
  6. SAGA 교훈 적용: DAG 구조가 알려진 경우 KV 퇴거 최적성 상한(upper bound)을 사전 계산해 실제 정책과 gap 측정
  7. DAG 구조 미지(unknown) 폴백: 기존 `CacheAwareScheduler` 위임
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: DAG 노드별 GPU 할당 정보를 토폴로지 메타데이터에 포함. 후속 에이전트가 다른 노드에 배정된 경우 KV 전송 비용(RDMA/NVLink latency)을 보존 결정 스코어에 감산하여 전송 비용이 높으면 재계산을 선택하는 자동 임계값 조정.
- **예상 효과**: 처리량 +25~35% (에이전트 워크플로우 Bélády 근접 시), 캐시 히트율 +20~30%p (DAG 인식 보존), TTFT p50 오버헤드 +3% 이내
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 최우선 타겟)

### 아이디어 A-2: Harvest NVLink 피어 GPU 캐시 + DMS 8× 압축 결합 계층 (A+C 복합)
- **가설**: NVLink로 연결된 피어 GPU의 미사용 HBM을 KV 캐시 중간 계층으로 활용할 때(Harvest), 해당 피어 캐시에 저장되는 KV를 DMS 8× 압축으로 저장하면 동일 피어 HBM 용량에 8배 더 많은 KV를 보관할 수 있어 클러스터 전체 유효 KV 캐시 용량이 대폭 확장된다. CPU 오프로딩(PCIe 병목)을 완전히 우회하는 동시에 메모리 밀도를 극대화한다.
- **접근 방법**:
  1. `NVLinkCompressedTierScheduler` 클래스 (`src/scheduler/nvlink_compressed_tier.py`)
  2. 피어 GPU 계층 모델: `{peer_gpu_id, available_hbm_mb, nvlink_bandwidth_gbps, compressed_capacity_mb}` — compressed_capacity = available_hbm × compression_ratio(8×)
  3. KV 저장 정책: GPU 로컬 HBM(핫 세그먼트, FP16) → NVLink 피어 HBM(웜 세그먼트, DMS 8× 압축) → CPU DRAM(콜드, 추가 압축) 계층 순서
  4. 피어 캐시 조회: NVLink 전송 지연(~수십 나노초) vs 재계산 비용 비교 → 빠른 쪽 선택
  5. DMS 압축 통합: `DelayedEvictionCodec` (`src/cache/delayed_eviction_codec.py`) — DMS "지연 퇴거" 메커니즘 구현 (퇴거 표시된 토큰을 짧은 윈도우 동안 접근 가능 유지 후 퇴거)
  6. 동적 피어 가용성 감지: 피어 GPU 사용률 주기적 폴링, 가용 HBM > threshold일 때만 피어 캐시 활성화
- **대상 환경**: 멀티 노드 (NVLink 클러스터)
- **멀티 노드 고려사항**: NVLink 토폴로지 맵 사전 구성 필요. NVLink 대역폭 포화 시 RDMA 폴백. 피어 GPU가 자체 추론 워크로드로 busy한 경우 캐시 계층 즉시 비활성화(opportunistic 특성 유지).
- **예상 효과**: 처리량 +30~40% (NVLink 전송 지연 < 재계산 비용 시), KV 캐시 메모리 효율 8× 향상(피어 계층 기준), 유효 컨텍스트 길이 4~8× 확장
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 A-3: llm-d 0.5 LoRA 어댑터 인식 캐시 라우팅 + Kubernetes 계층형 오프로딩 통합
- **가설**: 다중 LoRA 어댑터를 서빙하는 환경에서 요청을 동일 LoRA 어댑터 + 동일 접두사 캐시를 가진 인스턴스로 우선 라우팅하면, 어댑터 교체 비용(GPU 메모리 재로드)과 KV 캐시 재계산 비용을 동시에 제거할 수 있다. llm-d 0.5 구현에서 검증된 LoRA-인식 캐시 라우팅 원리를 단일 노드 시뮬레이션 환경에서 구현한다.

  2. 라우팅 키: `(adapter_id, prefix_hash_top_k)` 복합 키로 인스턴스 선택
  3. 어댑터-캐시 공동 인덱스: 각 인스턴스의 `{active_adapter_id, cached_prefix_hashes}` 상태를 중앙 라우터에 주기적 보고
  4. 라우팅 우선순위: (1) 동일 어댑터 + 접두사 히트 → 최우선, (2) 동일 어댑터 + 접두사 미스 → 차순위, (3) 어댑터 미스 → 부하 균형 기반

- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 어댑터 인덱스 동기화 주기(gossip 프로토콜). 어댑터 교체 비용이 높은 환경(대형 LoRA r=128)에서 어댑터 핀 고정 인스턴스 수 결정 정책.
- **예상 효과**: 처리량 +15~25% (다중 어댑터 시나리오), 어댑터 교체 오버헤드 −80%, KV 캐시 히트율 +10~15%p

- **우선순위**: 2

### 아이디어 B-1: SideQuest 자기-지시형 KV 퇴거를 비연속 세그먼트 선택 정책으로 통합
- **가설**: 모델 자체를 KV 세그먼트 유지·퇴거 결정의 주체로 활용할 때(SideQuest), 이를 "에이전틱 장문 컨텍스트에서 어떤 비연속 세그먼트를 유지할 것인가"를 결정하는 보조 태스크로 정의하면, 215개 샘플 파인튜닝만으로 외부 휴리스틱(어텐션 스코어, L2 노름) 없이 의미 기반 세그먼트 보존 결정이 가능하다. 기존 DHD 의미 유사도 기법(이전 사이클 B-1)이 유사도 스코어로 세그먼트를 선택하는 것과 달리, 이 접근은 모델이 작업 상태 이해를 바탕으로 직접 퇴거 지시를 생성한다.
- **접근 방법**:
  1. `ModelDirectedEvictionPolicy` 클래스 (`src/cache/model_directed_eviction.py`)
  2. 보조 태스크 정의: 메인 추론과 병렬로 "현재 컨텍스트의 어느 세그먼트가 이후 추론에 불필요한가"를 묻는 경량 프롬프트를 실행 (단일 패스, 1~5 토큰 응답 — 세그먼트 인덱스 목록)
  3. 파인튜닝 데이터: 215개 샘플 = 에이전틱 장문 컨텍스트 + 정답 퇴거 세그먼트(golden eviction)를 ForesightKV의 골든 퇴거 알고리즘으로 자동 레이블링
  4. `NonContiguousSegmentCache`의 `evict_candidates()` 훅에 모델 지시 결과를 우선 적용 → 모델 지시 없는 세그먼트는 기존 LRU/어텐션 스코어 폴백
  5. 추론 비용 절감: 보조 태스크를 메인 추론의 크리티컬 패스 밖 CPU 스레드에서 비동기 실행 (FreeKV 투기적 검색과 유사한 비동기 패턴)
  6. 기존 `SegmentedHashCache` 및 `SemanticSegmentCache`와 스택 가능: 세그먼트 퇴거 결정 레이어에만 추가
- **예상 효과**: 비연속 히트율 +15~20%p (모델 의미 이해 기반 보존), TTFT −15~25% (불필요 세그먼트 조기 퇴거로 캐시 압력 감소), 피크 토큰 사용량 −40~60%
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-2: KVCache in the Wild 워크로드 특성 기반 카테고리별 TTL 세그먼트 보존 정책
- **가설**: 클라우드 실측 데이터(KVCache in the Wild)에서 요청 카테고리별 KV 재사용 시간과 확률이 예측 가능하다면, 카테고리(코드 생성 / 대화 / RAG / 에이전틱)별로 다른 TTL(Time-To-Live)을 세그먼트에 할당하는 정책이 단일 LRU 정책 대비 캐시 히트율을 23.9% 이상 향상시킬 수 있다. 이는 이전 사이클의 모든 Activity B 기법이 세그먼트 내용(해시, 의미 유사도, 서명 벡터)만 보고 사용 패턴(시간·카테고리)은 무시한 것과 다르다.
- **접근 방법**:
  1. `WorkloadAwareTTLCache` 클래스 (`src/cache/workload_ttl_cache.py`): `CacheStore` 인터페이스 구현
  2. 카테고리 분류기: 요청 헤더 또는 경량 토큰 분류기(k-NN on 첫 32 토큰 임베딩)로 카테고리 결정 — 추가 모델 불필요
  3. 카테고리별 TTL 프로파일: `{category: {median_reuse_gap_sec, reuse_probability, ttl_multiplier}}` — 초기값은 KVCache in the Wild 논문 Table 3 참고, 이후 온라인 업데이트
  4. TTL 기반 퇴거: 세그먼트 TTL 만료 시 즉시 퇴거 후보로 표시 → 메모리 부족 시 TTL 만료된 세그먼트 우선 퇴거
  5. 온라인 TTL 갱신: 실제 히트·미스 패턴을 카테고리별로 집계하여 TTL 프로파일 주기적 업데이트 (EMA 방식)
  6. 단일 턴 재사용 최적화: 단일 턴 요청도 멀티 턴과 동등하게 TTL 기반 보존 — KVCache in the Wild의 핵심 발견 반영
- **예상 효과**: 비연속 히트율 +20~25%p (TTL 기반 보존으로 만료 전 재사용), TTFT −30~42% (캐시 히트 증가), Cache Eviction Rate −30~40%
- **구현 난이도**: low
- **우선순위**: 1 (이번 사이클 구현 타겟)

## Activity C 아이디어

### 아이디어 C-1: R-KV 이중 스코어(중요도+중복성)를 비연속 세그먼트 퇴거 우선화에 적용
- **가설**: CoT/에이전틱 컨텍스트에서 반복적 자기 성찰 패턴으로 인해 비연속 세그먼트 간 고도로 중복적인 KV 블록이 존재한다. R-KV의 "어텐션 중요도 스코어 + 동적 중복성 스코어" 이중 평가를 세그먼트 단위로 적용하면, 단순 중요도 기반 퇴거(H2O, SnapKV) 대비 중복 세그먼트를 우선 퇴거해 정확도 마진을 확보하면서 10% KV 예산으로 100% 성능 근접이 가능하다. 이는 이전 사이클의 HybridEvictionPolicy(CAOTE+StructKV)가 어텐션 출력 오류 + 글로벌 in-degree centrality를 결합한 것과 달리, 중복성 차원을 명시적으로 도입한다.
- **접근 방법**:
  1. `RedundancyAwareEvictionPolicy` 클래스 (`src/cache/redundancy_eviction.py`)
  2. 중요도 스코어: 누적 어텐션 스코어(기존 구현의 `attention_score` 메트릭 재활용)
  3. 중복성 스코어: 세그먼트 쌍 간 Key 벡터 코사인 유사도 평균 → 유사도 높을수록 중복성 높음 → 퇴거 우선
  4. 이중 스코어 결합: `eviction_score = (1 - importance_score) × redundancy_score` — 중요도 낮고 중복성 높은 세그먼트 우선 퇴거
  5. 세그먼트 단위 중복성 계산: 새 세그먼트 삽입 시 캐시 내 상위 N 세그먼트와 유사도 계산 (N≤100이면 brute-force O(N×chunk_size×d_head) 허용)
  6. RAG 특화: 동일 문서가 여러 청크로 분할되어 반복 등장하는 경우 문서 ID 기반 중복 감지 단축 경로 제공
  7. 기존 `CompressedSegmentCache.evict()` 정책으로 drop-in 교체 가능
- **예상 효과**: 메모리 −80~90% (10% KV 예산 목표), accuracy delta ±0.8% 이내, 처리량 +5~6× (R-KV 실증 기준)
- **accuracy-preserving 근거**: R-KV(NeurIPS 2025, Zefan Cai et al.)가 추론 모델에서 10% KV → ~100% 성능 유지를 실증(Qwen, DeepSeek-R1 계열). 중복성 스코어는 이미 중요도 스코어가 낮은 세그먼트들 중에서만 우선순위를 조정하므로 중요 토큰 퇴거 위험 없음. 이중 스코어의 곱 형태 결합이 단독 중요도 기반보다 false negative(중요 토큰 퇴거) 감소. 검증: WikiText-2 perplexity ±1% + LongBench QA/요약 + AIME 추론 벤치마크(CoT 중복성 효과 확인).
- **구현 난이도**: low
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 C-2: PackKV 캐시-상주 디컴프레션 — GPU 공유 메모리 내 압축 해제 시스템 공동 설계
- **가설**: KV 캐시 압축 해제 비용을 어텐션 커널 내부 GPU 공유 메모리에서 처리하면(PackKV 원리), 압축 해제된 KV를 전역 메모리에 쓰고 다시 읽는 왕복 비용이 제거되어 기존 SOTA 양자화(INT4/INT8) 대비 KV 처리량이 K 75.6%, V 171.6% 향상된다. 이전 사이클의 TurboQuantCodec이 알고리즘 수준 최적화(PolarQuant+QJL)라면, 이 접근은 동일 압축 결과를 하드웨어 친화적 실행으로 가속하는 시스템-레벨 직교 최적화다.
- **접근 방법**:
  1. `SharedMemDecompressor` 클래스 (`src/cache/shared_mem_decomp.py`)
  2. CUDA 커널 설계 (PyTorch custom op): 압축된 KV를 GPU 공유 메모리로 직접 로드 → 공유 메모리 내에서 압축 해제 → FlashAttention에 직접 전달 (전역 메모리 쓰기 없음)
  3. 오류-제어 양자화: PackKV 원리 — 캘리브레이션 통계로 채널별 정밀도 결정 (정밀도 민감 채널 FP8, 나머지 INT4)
  4. 기존 `TurboQuantCodec`의 encode 출력을 `SharedMemDecompressor` 친화적 포맷(block-wise 인터리빙)으로 재배치하는 변환 레이어 추가
  5. vLLM PagedAttention 통합 경로: 페이지 경계를 공유 메모리 로드 경계와 정렬하여 단편화 없는 공유 메모리 로드 실현
  6. PyTorch fallback: CUDA 커널 미지원 환경에서는 표준 압축 해제 경로로 자동 전환
- **예상 효과**: KV 처리량 K +75%, V +170% (PackKV 실증 기준), 메모리 K 153%, V 179% 추가 압축률 향상, accuracy delta ±0.3% 이내
- **accuracy-preserving 근거**: PackKV(IPDPS 2026)가 오류-제어 양자화로 "동등 정확도 기준 압축률 향상"을 실증 — 정확도를 희생하지 않고 압축률을 높이는 구조. 공유 메모리 내 디컴프레션은 수치 정밀도 경로를 변경하지 않고 메모리 접근 패턴만 최적화하므로 정확도 영향 없음. 채널별 정밀도 결정이 민감 채널을 보호. 검증: MMLU + WikiText-2 perplexity + LLM throughput benchmark.
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 C-3: DMS 지연 퇴거(Delayed Eviction) + ForesightKV RL 정책 통합
- **가설**: DMS의 "퇴거 표시된 토큰을 짧은 윈도우 동안 접근 가능하게 유지" 메커니즘(지연 퇴거)과 ForesightKV의 RL 기반 장기 기여도 학습을 결합하면, 지연 퇴거 윈도우를 RL 정책이 동적으로 결정하여 "언제 퇴거해도 안전한가"를 학습할 수 있다. DMS 단독(1K 스텝 파인튜닝)보다 적은 훈련으로 더 정밀한 퇴거 타이밍을 달성하고 AIME 류 추론 벤치마크에서 정확도를 추가로 향상시킨다.
- **접근 방법**:
  1. `RLDelayedEvictionCodec` 클래스 (`src/cache/rl_delayed_eviction.py`)
  2. MDP 정의(ForesightKV 방식): 상태 = {현재 KV 캐시 사용률, 토큰 나이, 누적 어텐션 스코어, 지연 퇴거 윈도우 내 토큰 수}, 행동 = {즉시 퇴거 / 지연 연장 / 보존}, 보상 = 추론 정확도 + 메모리 효율 가중합
  3. Golden Eviction 레이블링(ForesightKV 방식): 미래 어텐션 스코어 시뮬레이션으로 최적 퇴거 시점 결정 → SL 사전학습 후 GRPO 정제
  4. 지연 퇴거 윈도우 동적 결정: RL 정책 출력이 {즉시 퇴거, 16 토큰 지연, 64 토큰 지연, 보존} 중 선택
  5. DMS FlashAttention 호환성 유지: 지연 퇴거 윈도우 토큰을 별도 버퍼에 유지하고 마스크 어텐션으로 통합
  6. 1K 스텝 파인튜닝(DMS 방식) 후 GRPO 100 스텝 미세조정으로 전체 훈련 비용 최소화
- **예상 효과**: 메모리 −85~90% (DMS 8× + RL 최적화), accuracy delta ±0.5% 이내, AIME 24 기준 +8~14점 (DMS +12점 기반)
- **accuracy-preserving 근거**: DMS(NeurIPS 2025, NVIDIA)가 8× 압축에서 기존 훈련-무료 희소 어텐션 대비 우수한 정확도를 실증. ForesightKV(NeurIPS 2025)가 RL 기반 장기 기여도 학습으로 절반 캐시 예산에서 기존 방법 일관 초과. 지연 퇴거 윈도우 동안 퇴거 후보 토큰이 컨텍스트에 합산되므로 중요 정보가 누락되지 않는 구조적 보장. 검증: AIME 2024/2025 + GPQA + LiveCodeBench (DMS 기준 벤치마크 재현).
- **구현 난이도**: high
- **우선순위**: 4

### 아이디어 C-4: ScoutAttention 레이어-선행 CPU 선계산 + 카테고리별 TTL 계층 오프로딩
- **가설**: ScoutAttention의 "CPU가 GPU보다 한 레이어 앞서 어텐션 패턴을 계산"하는 레이어-선행 CPU 선계산 원리를, Activity B의 WorkloadAwareTTLCache(B-2) 카테고리별 TTL 정보와 결합하면 CPU 선계산 대상 블록을 TTL 만료 예정 세그먼트로 한정하여 CPU 계산 부하를 추가로 감소시킬 수 있다. 즉, "곧 퇴거될 세그먼트의 어텐션 기여도를 CPU에서 미리 계산해 GPU 메모리 반환을 가속"한다.
- **접근 방법**:
  1. `TTLGuidedScoutAttention` 클래스 (`src/cache/ttl_scout_attention.py`)
  2. TTL 만료 임박 세그먼트 감지: WorkloadAwareTTLCache에서 `ttl_remaining < threshold_evict_soon` 세그먼트 목록 조회
  3. CPU 선계산 우선화: ScoutAttention 원리 — TTL 임박 세그먼트의 어텐션 기여도를 CPU에서 레이어-선행 계산
  4. GPU-CPU 파이프라인: GPU가 레이어 L 처리 중 CPU가 레이어 L+1에서 TTL 임박 세그먼트의 상위-k 어텐션 블록 계산
  5. KV 오프로딩 가속: TTL 임박 세그먼트를 CPU 선계산 결과로 "어텐션 요약(attention summary)"으로 압축 후 퇴거 → 원본 KV 메모리 반환
  6. 정확도 손실 2.1% 이내(ScoutAttention 실증) 유지를 TTL 임계값 조정으로 보장 — TTL이 긴 중요 세그먼트는 GPU에서 정밀 계산
- **예상 효과**: 처리량 +2.1× (ScoutAttention 기준), 메모리 −40~60% (TTL 안내 선제 오프로딩), accuracy delta ±2% 이내 (ScoutAttention ±2.1% 기준 동등 또는 개선)
- **accuracy-preserving 근거**: ScoutAttention(arXiv 2603.27138)이 인접 디코딩 토큰의 어텐션 패턴 강한 시간적 국소성을 활용해 CPU 계산으로 대체해도 정확도 2.1% 이내 손실을 실증. TTL 임박 세그먼트는 이미 재사용 확률이 낮아 정밀도 요구가 낮으므로 CPU 근사 계산의 정확도 영향이 더 작다. 검증: LongBench + RULER long-context + WikiText-2 perplexity ±1% 기준 재검증.
- **구현 난이도**: medium
- **우선순위**: 2

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: A+B — DAGTopologyScheduler + WorkloadAwareTTLCache (DAG 위상 인식 TTL 보존)
- **조합 근거**: DAGTopologyScheduler(A-1)는 워크플로우 DAG에서 후속 노드가 현재 노드의 KV를 재사용할 확률을 예측하고, WorkloadAwareTTLCache(B-2)는 카테고리별 TTL로 세그먼트를 보존한다. 두 기법을 결합하면 DAG 위상 분석이 TTL 프로파일을 동적으로 오버라이드할 수 있다: "DAG 후속 노드가 현재 KV를 사용할 것으로 예측되면 TTL을 연장하고, 사용하지 않을 것으로 예측되면 TTL을 단축"하는 DAG-인식 TTL 조정 정책.
- **구체 설계**:
  1. `DAGAwareTTLAdjuster` 모듈: DAGTopologyScheduler에서 `{segment_id, dag_reuse_probability}` 이벤트를 구독
  2. TTL 조정: `adjusted_ttl = base_ttl × (1 + dag_reuse_probability × alpha)` — alpha=2.0 기본값
  3. 후속 노드 완료 시 TTL 즉시 단축(조기 퇴거 허용): 후속 노드 처리 완료 → 해당 KV 세그먼트 TTL 0으로 설정
  4. DAG 미지 요청: 기존 카테고리별 TTL 그대로 사용
- **예상 복합 효과**: 처리량 +30~40%, 비연속 히트율 +25~35%p, TTFT p50 오버헤드 +3% 이내, Cache Eviction Rate −35~45%
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 최우선 통합 타겟)

### 아이디어 Cross-2: B+C — WorkloadAwareTTLCache + RedundancyAwareEvictionPolicy (TTL-중복성 이중 퇴거 필터)
- **조합 근거**: WorkloadAwareTTLCache(B-2)가 "언제 퇴거할 것인가"를 TTL로 결정하고, RedundancyAwareEvictionPolicy(C-1)가 "어느 세그먼트를 퇴거할 것인가"를 중요도+중복성으로 결정한다. 두 기법을 직렬 결합하면 TTL 만료 후보 중에서만 이중 스코어를 적용해 연산 비용을 줄이면서도 퇴거 품질을 유지하는 효율적 퇴거 파이프라인이 가능하다.
- **구체 설계**:
  1. 1단계 필터: TTL 만료 세그먼트 목록 생성 (WorkloadAwareTTLCache)
  2. 2단계 스코어링: TTL 만료 세그먼트 중 `eviction_score = (1 - importance) × redundancy` 계산 (RedundancyAwareEvictionPolicy)
  3. 최종 퇴거: 이중 스코어 상위 K 세그먼트 퇴거 (메모리 압력에 따라 K 결정)
  4. TTL 연장 예외: 이중 스코어가 낮음(=중요하고 고유한 세그먼트)에도 불구하고 TTL 만료 시 TTL 연장 신호 생성
- **예상 복합 효과**: 처리량 +15~20%, 메모리 −85~90%, 비연속 히트율 +22~28%p, accuracy delta ±0.8% 이내
- **구현 난이도**: low
- **우선순위**: 2

### 아이디어 Cross-3: A+C — NVLinkCompressedTierScheduler + RedundancyAwareEvictionPolicy (압축 피어 캐시 + 중복성 인식 퇴거)
- **조합 근거**: NVLinkCompressedTierScheduler(A-2)는 피어 GPU HBM에 DMS 압축 KV를 저장해 클러스터 전체 유효 캐시 용량을 확장하지만, 피어 HBM 용량은 여전히 한정적이다. RedundancyAwareEvictionPolicy(C-1)로 피어 캐시 내 중복 세그먼트를 우선 퇴거하면 피어 캐시의 고유 KV 밀도를 높여 히트율을 추가로 향상시킨다.
- **구체 설계**:
  1. 피어 캐시 퇴거 정책: `NVLinkCompressedTierScheduler`의 피어 캐시 퇴거 로직을 `RedundancyAwareEvictionPolicy`로 교체
  2. 피어 캐시 중복성 계산: 피어 노드 간 공유 세그먼트 목록 교환 → 동일 세그먼트가 여러 피어에 복제된 경우 중복성 스코어 부여
  3. 압축 + 중복성: DMS 압축 세그먼트 중 중복성 높은 것은 단 하나의 피어에만 보존(deduplication), 나머지 피어에서 퇴거
- **예상 복합 효과**: 처리량 +35~45%, 메모리 −88~92%, 피어 캐시 히트율 +15~20%p 추가, accuracy delta ±0.8% 이내
- **구현 난이도**: high
- **우선순위**: 4

### 아이디어 Cross-4: A+B+C — DAGTopologyScheduler + WorkloadAwareTTLCache + RedundancyAwareEvictionPolicy 완전 통합
- **조합 근거**: 이번 사이클의 세 핵심 아이디어(A-1 DAG 위상 스케줄러, B-2 TTL 세그먼트 캐시, C-1 중복성 인식 퇴거)를 완전 통합한다. DAG 위상 분석이 TTL을 동적 조정(Cross-1)하고, TTL 만료 후보에 중복성 이중 스코어를 적용(Cross-2)하며, 전체 파이프라인이 처리량·메모리·정확도 목표를 동시에 달성하는 엔드-투-엔드 경로를 제공한다.
- **구체 설계**:
  1. 요청 수신 → 카테고리 분류 → WorkloadAwareTTLCache에 카테고리별 TTL 세그먼트 저장
  2. 워크플로우 DAG 등록 → DAGTopologyScheduler가 후속 노드 KV 재사용 확률 계산 → TTL 동적 조정 (Cross-1)
  3. 메모리 압력 발생 → TTL 만료 후보 생성 → RedundancyAwareEvictionPolicy 이중 스코어 정렬 → 퇴거 실행 (Cross-2)
  4. 퇴거 결과를 DAGTopologyScheduler에 피드백: 퇴거된 세그먼트의 DAG 노드 재배정 여부 결정
  5. 단일 노드 기준 구현 후 멀티 노드(NVLink 피어 캐시) 확장은 Cross-3로 진행
- **예상 복합 효과**: 처리량 +35~50%, 메모리 −80~90%, 비연속 히트율 +30~40%p (전체 히트의 50%+ 목표), accuracy delta ±0.8% 이내
- **구현 난이도**: high
- **우선순위**: 3

## 이번 사이클 최우선 구현 타겟

**Cross-1(A+B): DAGTopologyScheduler(A-1) + WorkloadAwareTTLCache(B-2) + DAGAwareTTLAdjuster**

구현 순서:
1. **B-2** `WorkloadAwareTTLCache`: 카테고리 분류기(k-NN 경량) + 카테고리별 TTL 프로파일 + LRU-TTL 복합 퇴거. `CacheStore` 인터페이스 구현. 독립 모듈로 즉시 배포 가능.
2. **C-1** `RedundancyAwareEvictionPolicy`: 중요도 × 중복성 이중 스코어 계산. 기존 퇴거 정책 drop-in 교체. 단순 구현(low 난이도)으로 빠른 검증.
3. **Cross-2 통합**: B-2 TTL 필터 + C-1 이중 스코어 직렬 파이프라인. WorkloadAwareTTLCache의 `evict_candidates()` 훅에 RedundancyAwareEvictionPolicy 연결.
4. **A-1** `DAGTopologyScheduler`: DAG 메타데이터 수집 인터페이스 + 위상 분석 + KV 보존 결정. `BaseScheduler` 상속.
5. **Cross-1 통합**: DAGAwareTTLAdjuster 모듈로 A-1과 B-2 연결. DAG 재사용 확률 → TTL 동적 조정.

이유:
- B-2 WorkloadAwareTTLCache는 구현 난이도 low + 즉각적인 캐시 히트율 향상(23.9%, KVCache in the Wild 산업 실증)으로 초기 가치를 신속하게 검증 가능.
- C-1 RedundancyAwareEvictionPolicy는 구현 난이도 low + 10% KV → 100% 성능(R-KV NeurIPS 2025 실증)으로 메모리 목표 −30% 달성 가능성이 높다.
- A-1 DAGTopologyScheduler는 이전 사이클의 DualMap(의미 히트율), GSA(경쟁비), CXL(토폴로지 비용)과 알고리즘·자료구조 모두 다른 새 접근(DAG BFS/위상 분석 + Bélády 상한 계산)으로 SIGNIFICANT_CHANGE 기준 충족.
- Cross-1 통합이 처리량 +30~40% + 비연속 히트율 +25~35%p + 오버헤드 +3% 이내를 동시 달성해 연구 목표 지표를 폭넓게 커버.

## planner에게 전달할 요청사항
- **Activity B-2(WorkloadAwareTTLCache) 구현 시**: `src/cache/workload_ttl_cache.py` 신규 생성. 카테고리 분류기는 첫 32 토큰 임베딩 k-NN(K=5) 또는 키워드 룰 기반으로 단순화 가능. TTL 프로파일 초기값은 논문 참고값(코드 생성 TTL 600s, 대화 TTL 300s, RAG TTL 120s 등)을 `configs/` YAML로 외부화. EMA 온라인 업데이트 파라미터(`ttl_ema_alpha`) 설정 파라미터로 노출.
- **Activity C-1(RedundancyAwareEvictionPolicy) 구현 시**: `src/cache/redundancy_eviction.py` 신규 생성. 중복성 스코어 계산은 새 세그먼트 삽입 시 O(N×chunk×d_head) brute-force (N≤100). RAG 단축 경로(문서 ID 기반 중복 감지) 선택적 구현. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + AIME 추론(CoT 중복성 효과 확인).
- **Activity A-1(DAGTopologyScheduler) 구현 시**: `src/scheduler/dag_topology_scheduler.py` 신규 생성. DAG 메타데이터는 JSON 포맷으로 수신 가능하도록 설계. Bélády 상한 계산(시뮬레이션 기반)을 벤치마크 비교 도구로 포함. `kv_reuse_probability` 히스토그램을 결과 파일에 저장.
- **Cross-1 통합 시**: DAGAwareTTLAdjuster를 별도 모듈(`src/scheduler/dag_ttl_adjuster.py`)로 구현. DAG 이벤트-TTL 조정 간 지연(이벤트 수신 → TTL 갱신)을 측정해 오버헤드 보고.
- **측정 지표 (이번 사이클)**: (1) 비연속 히트율 — TTL 기반 히트 vs 정확 해시 히트 분리 측정, (2) KV 메모리 감소율 — 중복성 퇴거 + TTL 퇴거 각각 기여분, (3) Cache Eviction Rate — TTL 만료 퇴거 vs 메모리 압력 퇴거 분리, (4) 처리량 — tokens/sec 베이스라인 대비, (5) DAG 재사용 확률 예측 정확도(예측 vs 실제 히트 비율).
