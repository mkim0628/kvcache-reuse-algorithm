## 변화 요약
- **Activity A 선점형 패러다임 전환**: TokenFlow(EuroSys 2026)의 선점 + 능동적 KV CPU 전송, GORGO의 크로스-리전 3차원 공동 최적화, WAIT 알고리즘의 유체역학 임계값 배칭이 이전 사이클의 모든 A 기법(이론 최적/DAG 위상/쿼리 플랜/공간-시간 분리 등)이 다루지 않은 "선점 + 능동 전송 오버랩" 및 "지리적 멀티-리전 라우팅"이라는 두 새로운 설계 차원을 동시에 열었다.
- **Activity B 에이전트 메모리 갱신 특화 등장**: KEEP(arXiv 2602.23592)의 Static-Dynamic 분리가 기존 B 기법들이 다루지 않은 "에이전트 행동 → 메모리 갱신 → 비연속 KV 무효화" 사이클을 KV 캐시 수준에서 관리하는 첫 메커니즘을 제시한다.
- **Activity C 스펙트럼·잔차 패러다임 신규 등장**: eOptShrinkQ(랜덤 행렬 이론 자동 저랭크) + DeltaKV(장거리 잔차) + DASH-KV(비대칭 딥 해싱 O(N)) + GVote(몬테카를로 자동 예산) + ManifoldKV(유클리드 아웃라이어)가 이전 사이클 C 기법 전체와 이론적 기반이 다른 새 기둥군을 형성하며, 특히 DeltaKV의 잔차 패러다임이 B+C 통합에 새 경로를 제시한다.

## 이번 사이클 초점
**A+C (선점형 스케줄링 기반 압축 파이프라인 + 크로스-리전 라우팅)** — 보조로 **B** StaticDynamicSegmentCache, **B+C** DeltaKV 잔차 저장 통합

**선택 이유**:
1. 이전 두 사이클(2026-05-05, 2026-05-06)이 B+C에 집중했으므로, 이번 사이클에서 Activity A의 선점형 패러다임(TokenFlow)과 Activity C의 자동 저랭크 압축(eOptShrinkQ)을 결합하면 A+C 통합의 새 경로가 열린다. 선점 시 CPU로 이동되는 KV를 eOptShrinkQ로 자동 압축하면 선점 오버헤드를 압축 비율만큼 추가로 줄일 수 있다.
2. GORGO 크로스-리전 라우팅이 이전 사이클 어디에도 없던 "지리적 멀티-리전" 환경 가정을 새로 도입하며, 이를 이전 사이클의 DeltaKV 잔차 압축과 결합하면 원격 리전 전송 비용 임계값을 낮출 수 있다.
3. Activity B의 StaticDynamicSegmentCache는 구현 독립성이 높아 병행 설계가 용이하며, 에이전트 메모리 관리 시나리오에서 B 히트율 목표(≥30%)를 달성하는 새 경로를 제공한다.
4. Activity C의 GVote 자동 예산 결정이 Activity B의 세그먼트별 중요도 예측과 결합될 때 "세그먼트-레벨 GVote" 변형이 B+C 통합에서 새로운 시너지를 만든다.

## Activity A 아이디어

### 아이디어 A-1: PreemptiveKVOffloadScheduler — 선점형 요청 스케줄링 + 능동적 GPU-CPU KV 전송 파이프라인
- **가설**: 표준 비선점형(non-preemptive) 요청 스케줄링은 요청 폭주(burst) 시 "머리-꼬리 블로킹(head-of-line blocking)"이 발생해 P99 TTFT가 급등하는데, 실시간 토큰 버퍼 점유율과 소비율을 기준으로 요청을 동적으로 선점하고 GPU-CPU KV 전송을 백그라운드에서 능동적으로 실행하면(TokenFlow 원리), 이전 사이클의 모든 Activity A 기법(OnlineLatencyOptimalScheduler의 이론 보장, ContiguousChunkIOScheduler의 I/O 정렬, DAGTopologyScheduler의 위상 분석 등)이 다루지 않은 "선점 타이밍의 실시간 적응성 + I/O-컴퓨팅 오버랩"으로 P99 TTFT를 80% 이상 감소시킬 수 있다.
- **접근 방법**:
  1. `PreemptiveKVOffloadScheduler` 클래스 (`src/scheduler/preemptive_kv_offload.py`): `BaseScheduler` 상속
  2. 선점 트리거: 실시간 토큰 버퍼 점유율(`buffer_occupancy_ratio`) + 소비율(`token_consumption_rate`) 계산. `buffer_occupancy_ratio > threshold_preempt` AND `consumption_rate < demand_rate` → 저우선순위 요청 선점
  3. 능동 KV 전송: 선점 결정 시 선점 요청의 KV를 즉시 GPU→CPU 비동기 전송(`torch.cuda.current_stream().synchronize()` 없이 백그라운드 큐에 적재). I/O와 다른 요청의 GPU 컴퓨팅을 오버랩.
  4. 선점 오버헤드 최소화 메커니즘: 선점된 요청의 KV 전송 시작 시점을 선점 결정 직후가 아닌 "다음 배치 버블(bubble)" 시간에 배치. 버블이 없을 경우 최소 전송 단위(ContiguousChunk, 이전 A-2)를 이용해 단계적 전송.
  5. 재개(resume) 로직: 선점된 요청이 재스케줄될 때 CPU DRAM에서 GPU로 KV 복원. 복원 비용이 재계산 비용보다 클 경우 재계산 선택.
  6. vLLM v0.9 비동기 KV Connector 통합 경로: vLLM의 비동기 KV Connector API를 PreemptiveKVOffloadScheduler의 전송 백엔드로 사용.
  7. 이전 A 기법과의 차별점: OnlineLatencyOptimalScheduler가 수학적 최적성 보장에 집중하고 ContiguousChunkIOScheduler가 I/O 블록 정렬에 집중한 반면, PreemptiveKVOffloadScheduler는 "요청을 멈추는 시점"과 "KV를 옮기는 타이밍"의 실시간 공동 결정이라는 새 차원을 도입한다.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: 멀티 노드 Disaggregated Prefill 환경에서 선점된 요청의 KV를 Prefill 노드 로컬 CPU DRAM에 오프로드. 이후 재개 시 P/D 노드 간 RDMA로 복원. 노드 간 KV 전송 비용과 재계산 비용의 실시간 비교 로직 포함.
- **예상 효과**: 처리량 +30~50% (요청 폭주 시, TokenFlow 실증 기준), P99 TTFT −80% (TokenFlow EuroSys 2026 실증), P50 TTFT 오버헤드 +5% 이내 (정상 부하 시)
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 A-2: CrossRegionSegmentRouter — 비연속 세그먼트 히트 확률 + 리전 간 지연 3차원 공동 최적화 라우터
- **가설**: 다중 리전 분산 LLM 추론에서 기존 라우터가 KV 캐시 히트율만 최적화하고 리전 간 네트워크 지연을 무시하는 문제를, 이전 사이클의 모든 A 기법이 단일 데이터센터 내 라우팅에 집중한 것과 달리, "세그먼트별 히트 확률 + 리전 간 실측 지연 + 큐 상태" 3차원을 공동 최적화하는 크로스-리전 라우터를 설계하면(GORGO 원리), 세그먼트 단위 히트 가치와 전송 비용의 실시간 비교로 단순 접두사 히트율 기반 라우팅 대비 TTFT를 2× 이상 감소시킬 수 있다.
- **접근 방법**:
  1. `CrossRegionSegmentRouter` 클래스 (`src/scheduler/cross_region_segment_router.py`): `BaseScheduler` 상속
  2. 라우팅 스코어 계산: `score(region_r) = α × segment_hit_value(r) - β × network_latency(r) - γ × queue_pressure(r)`
     - `segment_hit_value(r)`: 해당 리전의 세그먼트 인덱스(SignVQSegmentIndex 또는 SemanticSegmentCache에서 조회)에 대한 요청의 예상 세그먼트 히트 수 × 평균 세그먼트 KV 재사용 절감량
     - `network_latency(r)`: 리전 r까지의 실측 왕복 지연 (경량 핑 기반 이동평균)
     - `queue_pressure(r)`: 리전 r의 현재 대기 큐 길이 추정
  3. 세그먼트-인식 히트 예측: 접두사 전체 일치가 아닌 "세그먼트별 히트 확률"로 비연속 세그먼트 재사용 가치를 계산. 이전 사이클의 DualMapSemanticRouter가 의미 히트율 가중 라우팅을 단일 데이터센터에서 수행한 것과 달리, 세그먼트별 히트를 지리적 라우팅 신호로 활용.
  4. 로컬 우선 정책: 로컬 리전의 `score`가 임계값(`threshold_local_bias`) 이상이면 원격 전송 없이 로컬 처리 (GORGO 원리의 핵심).
  5. 피어 네트워크 메트릭 전파: 각 리전의 현재 메트릭(`segment_hit_rate`, `avg_latency`, `queue_depth`)을 경량 gossip으로 이웃 리전에 주기적 전파.
  6. GORGO-proxy 모드: 중앙화 proxy가 모든 리전의 메트릭을 집계해 전역 최적 라우팅 결정 (단일 데이터센터 내 멀티-노드 환경에서는 proxy = 클러스터 헤드 노드).
  7. 이전 A 기법과의 차별점: DualMapSemanticRouter(이전 A 기법)가 단일 데이터센터 내 의미 히트율 기반 이중 해시 라우팅을 수행한 반면, CrossRegionSegmentRouter는 지리적 다중 리전 + 세그먼트별 히트 가치 + 네트워크 지연의 3차원 공동 최적화라는 환경 가정과 알고리즘 모두 다르다.
- **대상 환경**: 멀티 노드 (크로스-리전 / 크로스-클러스터)
- **멀티 노드 고려사항**: 리전 간 KV 세그먼트 전송 비용(DeltaKV 잔차 압축 후 1/3.4 크기)을 `network_latency` 항에 반영. 리전 간 전송이 필요할 때는 eOptShrinkQ 또는 DeltaKV로 압축 후 전송하는 "compress-before-cross-region-transfer" 정책 통합.
- **예상 효과**: 처리량 +25~35%, 중간 TTFT −2× (GORGO-proxy 기준 실증), P99 TTFT −60~80%, 세그먼트 히트율 +20~30%p (로컬 우선 + 비연속 히트 가치 반영)
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-3: WAITThresholdBatcher — 유체역학 임계값 기반 동적 배칭으로 퇴거 없는 근최적 처리량
- **가설**: KV 캐시가 생성 중 동적으로 증가하는 환경에서 배치 시작 임계값을 정적으로 설정하면 퇴거 연쇄(eviction cascade)가 발생하는데, 유체역학 근사로 시스템을 "부하 균형 근처"에 유지하는 임계값 기반 배칭(WAIT 알고리즘)을 적용하면, 이전 사이클의 OnlineLatencyOptimalScheduler(이론적 레이턴시 최적화 초점)와 달리 "퇴거 없는 근최적 처리량"이라는 실용 목표를 단순한 임계값 파라미터 하나로 달성할 수 있다.
- **접근 방법**:
  1. `WAITThresholdBatcher` 클래스 (`src/scheduler/wait_threshold_batcher.py`): `BaseScheduler` 상속
  2. WAIT 임계값 계산: `threshold_T = available_kv_budget × (1 - safety_margin)`. `safety_margin` = 예상 최대 출력 길이 분산 × 안전 계수 (기본 0.15).
  3. 배칭 결정: 대기 중인 요청들의 예상 KV 사용량 합 < `threshold_T`이면 배치 시작 지연. 초과 시 즉시 배치 시작. 출력 길이 힌트 없으면 히스토그램 기반 추정.
  4. 동적 임계값 갱신: 실제 퇴거 발생 시 `safety_margin` 0.05 증가 (퇴거 0 유지 시 0.01 감소). 온라인 자동 조정.
  5. OnlineLatencyOptimalScheduler와의 차별점: OnlineLatencyOptimalScheduler가 "레이턴시 최소화"라는 다목적 이론 최적성을 추구하는 반면, WAITThresholdBatcher는 "퇴거 없는 처리량 최대화"라는 단일 실용 목표에 집중하며 WAIT 알고리즘의 유체역학 이론적 보장(근최적 처리량)을 직접 구현한다.
  6. PreemptiveKVOffloadScheduler(A-1)와 직렬 결합: A-1이 선점 타이밍을 결정하고 A-3이 배치 시작 타이밍을 결정 — 두 스케줄러가 선점과 배칭의 각 차원을 독립 제어.
- **대상 환경**: 단일 노드 / 멀티 노드 양쪽
- **멀티 노드 고려사항**: `available_kv_budget`을 로컬 노드 HBM 용량으로 한정. 멀티 노드에서는 노드별 독립 WAIT 인스턴스 + CrossRegionSegmentRouter(A-2)가 노드 간 부하 분산 담당.
- **예상 효과**: 처리량 +15~25% (퇴거 없는 배칭으로 KV 재계산 제거), 퇴거율 −95%+ (WAIT 알고리즘 목표), TTFT 오버헤드 +3% 이내
- **구현 난이도**: low
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: StaticDynamicSegmentCache — 에이전트 메모리 갱신 패턴 기반 정적/동적 세그먼트 분리 + Multi-hop 재계산 최소화
- **가설**: 구체화(embodied) 에이전트 및 멀티-턴 대화 환경에서 메모리(시스템 프롬프트, 공통 맥락 등)를 갱신할 때 갱신 세그먼트 이후의 모든 KV 캐시가 무효화되어 대규모 재계산이 발생하는 "에이전트-유도 비연속 KV 무효화" 문제를, 세그먼트를 "정적(불변) 세그먼트"와 "동적(갱신) 세그먼트"로 분리 관리하면(KEEP 원리), 이전 사이클의 모든 Activity B 기법(TTL, 차분 저장, 쿼리 중심 재계산, 정보 흐름 재순서화 등)이 갱신 패턴 자체를 KV 무효화 트리거로 모델링하지 않은 것과 달리 정적 세그먼트는 완전 재사용하고 동적 세그먼트 변경 시 영향 범위만 최소 재계산하는 구조가 가능하다.
- **접근 방법**:
  1. `StaticDynamicSegmentCache` 클래스 (`src/cache/static_dynamic_segment.py`): `CacheStore` 인터페이스 상속
  2. 세그먼트 분류:
     - 정적 세그먼트(Static Segments): 시스템 프롬프트, 문서 컨텍스트, 코드 라이브러리 등 갱신 빈도 낮은 세그먼트. `mark_static(segment_id)` 호출로 명시적 등록 가능. 또는 과거 갱신 이력이 없는 세그먼트를 자동으로 정적 분류.
     - 동적 세그먼트(Dynamic Segments): 에이전트 행동 결과, 도구 호출 출력, 대화 히스토리 갱신 등. `mark_dynamic(segment_id)` 또는 `update_segment(segment_id, new_tokens)`로 갱신.
  3. 무효화 전파 최소화 (Multi-hop Memory Reuse): 동적 세그먼트 갱신 시 해당 세그먼트 이후의 KV 전체를 무효화하는 대신, "갱신된 토큰이 이후 세그먼트에 실제로 영향을 미치는 범위"만 무효화. 영향 범위 계산: 어텐션 마스크 구조 분석으로 갱신 세그먼트와 후속 세그먼트 간 어텐션 의존성(attention dependency graph) 기반 영향 추정.
  4. Multi-hop 재계산 체인 제어: `max_recompute_hops` 파라미터(기본 2)로 재계산 전파 깊이 제한. 깊이 초과 세그먼트는 이전 KV 재사용 (근사 재사용).
  5. 정적 세그먼트 KV 영구 보존: 정적 세그먼트는 LRU 퇴거 대상에서 제외. 메모리 압력 시 동적 세그먼트 우선 퇴거.
  6. 이전 B 기법과의 차별점: DiffAwareSegmentStore(이전 B 기법)가 에이전트 간 KV 차분을 저장하는 반면, StaticDynamicSegmentCache는 단일 에이전트 내의 갱신 패턴에 따른 KV 무효화 범위를 최소화한다는 점에서 문제 설정과 알고리즘이 다르다. WorkloadAwareTTLCache(이전 B 기법)가 시간 기반 카테고리별 TTL을 사용하는 반면, 이 기법은 갱신 이벤트 기반 영향 범위 추정을 사용한다.
- **예상 효과**: 비연속 히트율 +20~30%p (정적 세그먼트 즉시 재사용), TTFT −40~60% (대규모 무효화-재계산 제거), 에이전트 메모리 갱신 처리량 +3~5× (KEEP 실증 기준)
- **구현 난이도**: medium
- **우선순위**: 1

## Activity C 아이디어

### 아이디어 C-1: eOptShrinkQCodec — BBP 위상전이 기반 자동 저랭크 공유 성분 추출 + TurboQuant 잔차 양자화
- **가설**: KV 캐시가 저랭크 공유 맥락 성분(context-shared)과 전랭크 토큰별 잔차(token-specific residual)로 분해된다는 스파이크 랜덤 행렬 모델에 기반해, BBP(Baik-Ben Arous-Péché) 위상전이를 이용해 자동으로 저랭크 공유 성분을 추출하고 잔차를 TurboQuantCodec(이전 사이클 C 기법)으로 압축하면, 이전 사이클의 모든 C 기법(HadamardInt4, TurboQuant 단독, NQKVCodec, FireQCodec, TriAttentionCodec 등)이 레이어/헤드별 랭크를 수동 설정하거나 단일 압축 전략에 의존한 것과 달리, "이론적으로 최적에 가까운 자동 랭크 선택"으로 TurboQuant 3비트 대비 2.2비트에서 동등 성능을 달성할 수 있다.
- **접근 방법**:
  1. `eOptShrinkQCodec` 클래스 (`src/cache/eopt_shrinkq_codec.py`): `CompressionCodec` 인터페이스 구현
  2. 저랭크 공유 성분 추출 (`eOptShrink` 단계):
     - KV 행렬의 특이값 분해(SVD) 수행 후 BBP 위상전이 기준으로 "신호 특이값(signal singular values)"과 "노이즈 특이값(noise singular values)" 자동 구분.
     - `auto_rank_select(kv_matrix, noise_level)` → BBP 임계값 이상의 특이값 수 = 자동 랭크 r
     - 공유 성분: `U[:, :r] @ diag(S[:r]) @ V[:r, :]` — 저랭크 베이스로 레이어별 캐싱
  3. 잔차 양자화 (`TurboQuant` 단계):
     - 잔차 = KV - 저랭크 베이스. 잔차에 TurboQuantCodec(이전 사이클 기구현)의 PolarQuant+QJL 파이프라인 적용.
     - 잔차의 내적 편향이 근-제로로 보장되어 양자화 왜곡 최소.
  4. K/V 비대칭 적용: Key는 낮은 랭크 + 공격적 잔차 양자화(2비트), Value는 높은 랭크 + 보수적 잔차 양자화(3비트). KV-CoRE 실증(Key 압축 가능성 > Value) 결합.
  5. 캘리브레이션: 레이어별 noise_level 추정을 ≥20 요청 샘플로 1회 오프라인 실행 후 저장.
  6. TurboQuantCodec(이전 사이클)과의 차별점: TurboQuantCodec이 KV 전체에 PolarQuant+QJL을 적용하는 반면, eOptShrinkQCodec은 먼저 저랭크 공유 성분을 분리(저장 효율)한 후 잔차에만 TurboQuant를 적용한다. 랭크 선택이 자동화되어 수동 튜닝 불필요.
- **예상 효과**: 메모리 −80~85% (2.2비트 실효 압축), 처리량 +15~20% (메모리 절감에 따른 배치 크기 증가), accuracy delta ±0.5% 이내 (eOptShrinkQ 실증)
- **accuracy-preserving 근거**: eOptShrinkQ(arXiv 2605.02905)가 Llama-3.1-8B, Ministral-8B LongBench 16개 태스크에서 TurboQuant 3비트 대비 2.2비트에서 동등/우수한 성능을 실증. BBP 위상전이의 이론적 보장: (1) 자동 랭크 선택의 일관성, (2) 잔차 내적 편향 근-제로, (3) 좌표 비국소화로 양자화 왜곡률 근최적. 멀티-니들 검색에서 2.2비트로 FP16 비압축과 동등한 수준 달성. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 멀티-니들 NIAH.
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 C-2: DeltaKVCrossSegmentCodec — 비연속 세그먼트 간 장거리 유사 참조 잔차 인코딩
- **가설**: KV 캐시 표현에서 비연속 세그먼트 간 장거리 유사 토큰이 존재하므로, 비연속 세그먼트를 독립적으로 저장하는 대신 "전역에서 검색된 가장 유사한 참조 토큰에 대한 잔차"를 인코딩하면(DeltaKV 원리를 세그먼트 경계 교차로 확장), 이전 사이클의 모든 C 기법이 "원본 KV 선택/결합/양자화"에 머문 것과 달리 "비연속 세그먼트를 공통 기저 + 세그먼트별 잔차"로 저장하는 새 패러다임으로 KV 메모리를 원본의 29% 수준으로 압축하면서 근무손실 정확도를 유지할 수 있다.
- **접근 방법**:
  1. `DeltaKVCrossSegmentCodec` 클래스 (`src/cache/delta_kv_cross_segment.py`): `CompressionCodec` 인터페이스 구현 + `CacheStore` 인터페이스와 공동 작동
  2. 참조 토큰 검색: 신규 세그먼트 S_new의 각 토큰 t에 대해 캐시된 모든 세그먼트에서 코사인 유사도 최근접 참조 토큰 ref(t) 검색 (ManifoldKV 스타일 유클리드 거리로 보완).
  3. 잔차 인코딩: `delta(t) = KV(t) - KV(ref(t))`. 잔차를 FP8 또는 INT8로 경량 양자화 저장.
  4. 참조 인덱스 저장: `(segment_id, token_idx)` 2-tuple로 참조 위치 저장. 참조 체인 깊이 제한(max_reference_depth = 2).
  5. 복원 경로: `KV(t) = KV(ref(t)) + delta(t)`. ref 세그먼트가 캐시에 있으면 즉시 복원, 없으면 재계산.
  6. 온-디맨드 재구성: DeltaKV 원논문처럼 희소 어텐션과 통합 시 선택된 토큰만 복원 가능.
  7. 이전 C 기법과의 차별점: DiffAwareSegmentStore(이전 B 기법)가 에이전트 그룹 내 마스터-차분 저장을 수행하는 반면, DeltaKVCrossSegmentCodec은 세그먼트 경계를 교차하는 전역 참조에 대한 잔차를 Activity C 압축 레이어에서 다룬다. KVSculptDistillationStore(이전 C 기법)가 L-BFGS 증류로 새 KV를 생성하는 반면, 이 기법은 기존 캐시 KV를 참조로 활용한다.
- **예상 효과**: 메모리 −70% (DeltaKV 실증 원본의 29% 유지), accuracy delta ±0.5% 이내, 처리량 +1.5~2× (Sparse-vLLM 결합 시 DeltaKV 실증 기준)
- **accuracy-preserving 근거**: DeltaKV(arXiv 2602.08005)가 KV 메모리를 원본의 29%(71% 감소)로 압축하면서 LongBench, SCBench, AIME에서 근무손실 정확도를 실증. 훈련 목적함수(MSE + NTP 하이브리드)로 의미 충실도 유지. 잔차 인코딩의 정확도 보존 메커니즘: 참조 토큰과의 잔차가 작을수록 압축 오류가 작고, 참조 검색 시 코사인 유사도 임계값으로 유사도 낮은 토큰은 원본 저장. 검증: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + SCBench.
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-3: GVoteSegmentBudgetAllocator — 몬테카를로 미래 쿼리 투표 기반 세그먼트별 자동 압축 예산 결정
- **가설**: 서로 다른 비연속 세그먼트가 미래 쿼리에 필요한 정도가 다르므로 단일 고정 압축 비율을 전체 세그먼트에 적용하면 중요한 세그먼트가 과압축되는 문제를, 몬테카를로 방식으로 잠재 미래 쿼리를 샘플링하고 각 세그먼트 내 키에 대한 투표를 집계해 세그먼트별 최적 압축 예산을 자동으로 결정하면(GVote 원리를 세그먼트 단위로 확장), 이전 사이클의 모든 C 기법(TailorKVManager, AdaptiveKVCompressibilityCodec 포함)이 레이어별 또는 언어/도메인별 적응에 머문 것과 달리 세그먼트별 미래 접근 가치에 기반한 압축 예산 자동 결정이 가능하다.
- **접근 방법**:
  1. `GVoteSegmentBudgetAllocator` 클래스 (`src/cache/gvote_segment_budget.py`)
  2. 잠재 쿼리 샘플링: 현재 입력 쿼리 임베딩 근방에서 N=32~64개 잠재 쿼리 벡터를 몬테카를로 샘플링 (가우시안 노이즈 추가).
  3. 투표 집계: 각 잠재 쿼리 q_i에 대해 Top-K 키 선택 → 세그먼트 S_j 내 키가 Top-K에 포함된 횟수를 투표로 집계. 투표수 높은 세그먼트 = 미래 중요 세그먼트.
  4. 예산 결정: 투표수 기반 세그먼트 중요도 점수 → `compress_budget(S_j) = budget_min + (budget_max - budget_min) × normalized_vote_score(S_j)`. 예산 범위: [4비트, 16비트].
  5. 다운스트림 코덱 연동: `GVoteSegmentBudgetAllocator`가 결정한 예산을 eOptShrinkQCodec(C-1) 또는 DeltaKVCrossSegmentCodec(C-2)의 압축 파라미터로 주입.
  6. 이전 C 기법과의 차별점: AdaptiveKVCompressibilityCodec(이전 C 기법)이 언어/도메인 감지 기반 고정 프로파일로 압축률을 결정하는 반면, GVoteSegmentBudgetAllocator는 각 요청의 미래 쿼리 패턴을 몬테카를로로 추정해 세그먼트별 압축 예산을 동적으로 결정한다.
- **예상 효과**: 동일 압축 예산에서 accuracy +5~15%p (GVote ICLR 2026 실증), 메모리 효율 최적화 (중요 세그먼트 보존 + 비중요 세그먼트 공격적 압축), accuracy delta ±0.5% 이내
- **accuracy-preserving 근거**: GVote(ICLR 2026)가 Llama, Qwen 모델에서 약 2× 메모리 감소 달성, 정확도 유지 또는 향상을 실증. Multi-Doc QA에서 10% 메모리로 0.35 정확도(20%+ 메모리 사용 기준선 초과). 미래 쿼리 투표 집계가 중요도 낮은 세그먼트를 더 공격적으로 압축하고 중요도 높은 세그먼트를 보존하므로 전체적 정확도 손실이 최소화된다. 검증: Multi-Doc QA + LongBench + WikiText-2 perplexity ±1%.
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 C-4: ManifoldKVWindowedEviction — 슬라이딩 윈도우 유클리드 아웃라이어 탐지 기반 장문 컨텍스트 퇴거
- **가설**: 기존 코사인 유사도 기반 KV 퇴거가 스케일(크기) 정보를 무시해 의미론적으로 중요한 토큰을 놓치는 문제와, 64K+ 컨텍스트에서 전역 중심이 의미 없어지는 "중심 희석 문제"를, 로컬 슬라이딩 윈도우 중심을 기준으로 유클리드 거리로 순위화하는 훈련-무료 스코어러를 적용하면(WindowedManifoldKV 원리), 이전 사이클의 모든 퇴거 기법(RedundancyAwareEvictionPolicy, HybridEvictionPolicy, LookaheadQCacheEviction 등)이 코사인/어텐션 스코어 기반으로 머문 것과 달리 장문 컨텍스트(64K+)에서 +49점 추가 정확도 개선이 가능하다.
- **접근 방법**:
  1. `ManifoldKVWindowedEviction` 클래스 (`src/cache/manifoldkv_windowed.py`): `CacheStore` 인터페이스 내 eviction policy로 구현
  2. 윈도우 중심 계산: 슬라이딩 윈도우(기본 크기 4096 토큰)를 컨텍스트 위로 이동하며 각 윈도우 내 키 벡터의 중심(centroid) 계산. 전역 중심 대신 로컬 중심 사용.
  3. 유클리드 아웃라이어 스코어: `outlier_score(k_i, window) = ||k_i - centroid(window)||_2`. 스코어가 높을수록 윈도우 내 두드러진(의미론적으로 중요한) 토큰.
  4. 퇴거 결정: 전체 퇴거 후보 중 `outlier_score` 하위 토큰 우선 퇴거. 방향(코사인) + 크기(유클리드)를 모두 반영.
  5. 3줄 구현 가능: ManifoldKV 논문의 핵심 기여처럼 기존 퇴거 파이프라인에 `outlier_score` 계산만 추가. 이전 사이클의 `CompressedSegmentCache.evict()` 정책으로 drop-in 교체.
  6. 이전 C 기법과의 차별점: RedundancyAwareEvictionPolicy(이전 C)가 코사인 유사도 기반 중복성을 측정하는 반면, ManifoldKVWindowedEviction은 유클리드 거리 기반 아웃라이어 탐지로 "스케일 정보 포함 의미론적 두드러짐"을 측정한다. 장문 컨텍스트에서의 "중심 희석 문제"를 슬라이딩 윈도우로 해결하는 설계가 완전히 새롭다.
- **예상 효과**: 퇴거 정확도 향상 (3-키 NIAH 50% 압축에서 +15.4점 개선, ManifoldKV 실증), 장문 64K+ 컨텍스트에서 +49점 (WindowedManifoldKV 실증), accuracy delta ±0.5% 이내, 구현 비용 최소 (3줄 추가)
- **accuracy-preserving 근거**: ManifoldKV(arXiv 2602.08343)가 RULER 벤치마크에서 20% 압축 시 95.7% 정확도 달성, 3-키 NIAH 50% 압축에서 KeyDiff(77.0%) 대비 92.4%(+15.4점) 실증. WindowedManifoldKV가 64K+ 컨텍스트에서 +49점 개선으로 장문 압축 SOTA 달성. 4개 아키텍처에서 튜닝 없이 동작해 범용성 입증. 유클리드 아웃라이어 탐지가 스케일 정보를 포함해 코사인 기반 대비 false negative(중요 토큰 퇴거) 감소. 검증: RULER 128K + WikiText-2 perplexity ±1% + NIAH 다중 키 시나리오.
- **구현 난이도**: low
- **우선순위**: 4

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: A+C — PreemptiveKVOffloadScheduler + eOptShrinkQCodec (선점-압축 파이프라인)
- **조합 근거**: PreemptiveKVOffloadScheduler(A-1)가 요청 선점 시 선점된 요청의 KV를 CPU로 전송할 때, eOptShrinkQCodec(C-1)의 자동 저랭크 압축을 GPU에서 전송 전 인라인으로 적용하면 전송해야 할 KV 크기가 2.2비트 수준으로 줄어 선점 오버헤드가 추가로 감소한다. 트렌드 리포트의 A+C 시너지 포인트(TokenFlow + eOptShrinkQ 결합)를 직접 구현한다.
- **구체 설계**:
  1. `CompressedPreemptionPipeline` 모듈 (`src/scheduler/compressed_preemption.py`)
  2. 선점 결정 시: `eOptShrinkQCodec.encode(kv_to_offload)` → 압축된 KV를 CPU 전송 큐에 적재
  3. GPU→CPU 전송 단위: 압축된 KV 블록 (원본의 ~14% = 2.2비트 / 16비트). PCIe 대역폭 요구 1/7 감소.
  4. 재개 시 복원: CPU→GPU 전송 후 `eOptShrinkQCodec.decode(compressed_kv)` → 원본 근사 복원 → 어텐션 계산
  5. 압축-전송 파이프라인 오버랩: CUDA 스트림 분리로 eOptShrinkQ 압축(compute stream)과 PCIe 전송(memory stream)을 오버랩
  6. 정확도 영향: 복원된 KV가 원본과 근사이므로 재개 후 첫 토큰의 어텐션 품질이 소폭 저하 가능 → 중요 요청(SLA 우선순위 상위)은 선점 대상에서 제외하는 필터 추가
- **예상 복합 효과**: 처리량 +40~60%, P99 TTFT −80% (선점 효과) + 선점 오버헤드 −30~40% (압축 전송 효과), 메모리 −75~80% (eOptShrinkQ 압축), accuracy delta ±0.6% 이내
- **구현 난이도**: medium
- **우선순위**: 1

### 아이디어 Cross-2: B+C — StaticDynamicSegmentCache + DeltaKVCrossSegmentCodec (정적 세그먼트 잔차 압축)
- **조합 근거**: StaticDynamicSegmentCache(B-1)의 정적 세그먼트는 갱신 없이 반복 재사용되므로 DeltaKVCrossSegmentCodec(C-2)의 잔차 인코딩을 적용하기에 최적이다. 정적 세그먼트들 간의 유사 토큰을 전역 참조로 활용하면 "정적 기저 + 세그먼트별 잔차" 구조로 메모리를 원본의 29% 수준으로 줄이면서 정적 세그먼트의 높은 재사용률을 유지할 수 있다. 트렌드 리포트의 B+C 시너지("공통 세그먼트를 기저로, 차이를 잔차로")를 직접 구현한다.
- **구체 설계**:
  1. 정적 세그먼트 등록 시: `DeltaKVCrossSegmentCodec.encode_against_reference(segment, reference_pool)` 호출 → 기존 정적 세그먼트들을 참조풀(reference_pool)로 잔차 인코딩
  2. 참조풀 관리: 가장 자주 접근된 Top-K 정적 세그먼트를 FP16 원본으로 참조풀에 보존 (K=5~10). 나머지 정적 세그먼트는 잔차 인코딩 저장.
  3. 동적 세그먼트: 잔차 인코딩 없이 원본 또는 eOptShrinkQCodec으로 압축 저장 (갱신 빈도 높아 참조로 활용하기 부적절)
  4. 재사용 경로: 정적 세그먼트 히트 시 `decode(delta + ref_kv)` → 빠른 복원. 참조 세그먼트가 캐시에 있으면 복원 비용 최소.
- **예상 복합 효과**: 처리량 +30~45%, 메모리 −70~80% (정적 세그먼트 잔차 압축 + 동적 세그먼트 eOptShrinkQ), 비연속 히트율 +25~35%p, accuracy delta ±0.6% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-3: A+B — CrossRegionSegmentRouter + StaticDynamicSegmentCache (정적 세그먼트 크로스-리전 공유)
- **조합 근거**: StaticDynamicSegmentCache(B-1)의 정적 세그먼트는 여러 사용자·에이전트가 공유하는 공통 컨텍스트(시스템 프롬프트, 공통 문서)이므로 크로스-리전으로 공유할 가치가 높다. CrossRegionSegmentRouter(A-2)에 "정적 세그먼트 우선 크로스-리전 공유" 정책을 추가하면, 정적 세그먼트 하나를 한 리전에서 계산해 다른 리전들이 잔차 압축된 형태로 재사용하는 "정적 세그먼트 크로스-리전 캐시 풀"이 형성된다.
- **구체 설계**:
  1. `StaticSegmentGlobalPool` 추상화: 정적 세그먼트를 리전 경계 없이 전역 풀에서 관리
  2. `CrossRegionSegmentRouter`가 요청에서 정적 세그먼트 검색 시 전역 풀 우선 조회 → 다른 리전의 정적 세그먼트 히트 시 DeltaKVCrossSegmentCodec 압축 형태로 전송
  3. 전송 비용 절감: 압축된 정적 세그먼트(원본의 29%) 전송으로 크로스-리전 전송 임계값 대폭 하향
  4. 동적 세그먼트: 로컬 리전에만 보존, 크로스-리전 공유 없음
- **예상 복합 효과**: 처리량 +35~50%, 비연속 히트율 +30~40%p, 크로스-리전 KV 전송 비용 −70% (잔차 압축), accuracy delta ±0.6% 이내
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 Cross-4: A+B+C — PreemptiveKVOffloadScheduler + StaticDynamicSegmentCache + eOptShrinkQCodec (에이전트 메모리 인식 선점-압축 완전 통합)
- **조합 근거**: 트렌드 리포트의 A+B+C 통합 파이프라인(KEEP Static-Dynamic 분리 → DeltaKV 잔차 압축 → GORGO 라우팅 → TokenFlow 선점)을 이번 사이클의 세 핵심 아이디어로 직접 구현한다. StaticDynamicSegmentCache(B-1)가 에이전트 메모리를 정적/동적으로 분리하고, eOptShrinkQCodec(C-1)이 정적 세그먼트를 2.2비트로 자동 압축하며, PreemptiveKVOffloadScheduler(A-1)가 선점 시 압축된 KV를 CPU로 능동 전송하는 완전 통합 파이프라인.
- **구체 설계**:
  1. 에이전트 요청 수신 → StaticDynamicSegmentCache가 세그먼트 분류
  2. 정적 세그먼트 → eOptShrinkQCodec으로 BBP 자동 저랭크 압축 후 캐시 (고압축 + 재사용 효율)
  3. 동적 세그먼트 → 원본 FP16 캐시 (갱신 빈도 높아 압축 오버헤드 피함)
  4. 요청 폭주 시 → PreemptiveKVOffloadScheduler가 선점 결정 → 선점된 요청의 동적 세그먼트 KV를 eOptShrinkQCodec으로 인라인 압축 후 CPU 전송
  5. 선점 요청 재개 시 → CPU에서 압축 해제 후 GPU로 전송 → 동적 세그먼트 갱신 이벤트가 있었으면 Multi-hop 최소 재계산만 실행
  6. GVoteSegmentBudgetAllocator(C-3)를 메타-압축 정책으로 활용: 각 세그먼트의 미래 접근 가치에 따라 eOptShrinkQ 압축 예산 동적 조정
- **예상 복합 효과**: 처리량 +50~70%, 메모리 −80~85%, 비연속 히트율 +35~45%p (전체 히트의 65%+ 목표), P99 TTFT −80%, accuracy delta ±0.6% 이내
- **구현 난이도**: high
- **우선순위**: 2

## 이번 사이클 최우선 구현 타겟

**Cross-1(A+C): PreemptiveKVOffloadScheduler(A-1) + eOptShrinkQCodec(C-1) + CompressedPreemptionPipeline 통합**, 보조로 StaticDynamicSegmentCache(B-1), ManifoldKVWindowedEviction(C-4), DeltaKVCrossSegmentCodec(C-2) 구현

구현 순서:
1. **A-1** `PreemptiveKVOffloadScheduler`: 토큰 버퍼 점유율/소비율 모니터링 + 선점 결정 + 비동기 GPU→CPU KV 전송. `BaseScheduler` 상속. TTFT p99/p50 측정 스크립트 포함.
2. **C-1** `eOptShrinkQCodec`: BBP 위상전이 자동 랭크 선택 + TurboQuantCodec(기구현) 잔차 파이프라인. K/V 비대칭(Key 2비트 / Value 3비트) 기본 설정. **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + LongBench 8개 서브태스크.
3. **Cross-1** `CompressedPreemptionPipeline`: A-1 선점 시 C-1 인라인 압축 + PCIe 전송 오버랩. CUDA 스트림 분리 구현.
4. **B-1** `StaticDynamicSegmentCache`: 세그먼트 정적/동적 분류 API + Multi-hop 무효화 전파 깊이 제어. `CacheStore` 인터페이스 구현.
5. **C-4** `ManifoldKVWindowedEviction`: 슬라이딩 윈도우 유클리드 아웃라이어 스코어. 기존 `CompressedSegmentCache.evict()` drop-in 교체. (low 난이도, 빠른 배포)

이유:
- eOptShrinkQCodec은 이전 사이클의 TurboQuantCodec(기구현)을 백엔드로 활용하므로 구현 복잡도가 낮으면서도 2.2비트라는 새로운 압축 수준을 달성한다. Activity C 메모리 목표 −30%를 5× 이상 초과하는 가장 이론적으로 탄탄한 단일 기법이다.
- PreemptiveKVOffloadScheduler는 요청 폭주 시 P99 TTFT 80% 감소라는 EuroSys 2026 최고 수준 실증 데이터를 가진 신규 설계 차원을 도입한다. TTFT 오버헤드 +5% 이내 목표와 처리량 +20% 목표를 동시에 달성하는 가장 강력한 A 기법이다.
- Cross-1 통합(A+C)이 두 기법의 상승 효과로 선점 오버헤드를 추가 30~40% 감소시켜 연구 목표 지표를 폭넓게 커버한다.

## planner에게 전달할 요청사항
- **Activity A-1(PreemptiveKVOffloadScheduler) 구현 시**: `src/scheduler/preemptive_kv_offload.py` 신규 생성. 선점 트리거 파라미터: `threshold_preempt`(기본 0.85), `consumption_rate_window`(기본 32 토큰). 비동기 KV 전송은 `torch.cuda.CUDAGraph` 또는 `asyncio.Queue` 패턴으로 구현. TTFT p50/p99 측정 스크립트(`experiments/run_preemptive_ttft.py`) 작성 필수. vLLM v0.9 비동기 KV Connector와의 통합 경로 주석으로 명시.
- **Activity C-1(eOptShrinkQCodec) 구현 시**: `src/cache/eopt_shrinkq_codec.py` 신규 생성. BBP 위상전이 자동 랭크 선택 구현: `torch.linalg.svd` 후 특이값 분포에서 BBP 임계값(`sigma_plus_1 = (1 + sqrt(aspect_ratio))^2 × noise_level`) 이상의 특이값 수를 자동 랭크로 선택. TurboQuantCodec(기구현)을 잔차 양자화 백엔드로 임포트해 재활용. **accuracy-preserving 검증 필수**: `experiments/run_eopt_accuracy.py` 스크립트 작성, WikiText-2 perplexity ±1% + LongBench 8개 서브태스크 + 멀티-니들 NIAH 검증. K/V 비대칭 압축 파라미터(`key_bits=2, value_bits=3`) 설정 YAML로 외부화.
- **Cross-1(CompressedPreemptionPipeline) 구현 시**: `src/scheduler/compressed_preemption.py` 신규 생성. eOptShrinkQCodec 압축을 CUDA compute stream에서, PCIe 전송을 memory stream에서 동시 실행하는 이중 스트림 파이프라인 구현. 압축-전송 오버랩 효과 측정(`overlap_efficiency` 메트릭) 포함. 정확도 중요 요청(SLA tier-A)에 대한 선점 제외 필터 파라미터 노출.
- **Activity B-1(StaticDynamicSegmentCache) 구현 시**: `src/cache/static_dynamic_segment.py` 신규 생성. `mark_static(segment_id)` / `mark_dynamic(segment_id)` / `update_segment(segment_id, new_tokens)` API 포함. 어텐션 의존성 그래프 기반 영향 범위 추정 대신 초기 구현에서는 단순화된 "변경 토큰 이후 최대 N개 세그먼트만 무효화" 근사 사용(`max_invalidation_range=2` 파라미터). `CacheStore` 인터페이스 구현. Multi-hop 재계산 깊이 제한 파라미터(`max_recompute_hops=2`) 노출.
- **Activity C-4(ManifoldKVWindowedEviction) 구현 시**: `src/cache/manifoldkv_windowed.py` 신규 생성. 슬라이딩 윈도우 크기 파라미터(`window_size=4096`). 유클리드 거리 계산은 `torch.cdist` 활용(배치 연산 지원). 기존 `CompressedSegmentCache.evict()` 메서드에 drop-in 교체 인터페이스 제공. RULER 128K + NIAH 다중 키 정확도 검증 스크립트 포함.
- **측정 지표 (이번 사이클 필수)**:
  1. **선점 효율**: 선점 횟수 / 총 요청 수, 선점 당 평균 KV 전송 크기(압축 전/후), GPU→CPU 전송 오버랩 효율
  2. **압축 정확도**: eOptShrinkQCodec 자동 랭크 선택 결과(레이어별 랭크 r), WikiText-2 perplexity ±1%, LongBench 8개 서브태스크
  3. **TTFT**: P50/P99 (선점형 vs 비선점형 비교), 요청 폭주 시나리오 포함
  4. **비연속 히트율**: 정적 세그먼트 히트 / 동적 세그먼트 히트 / 기존 해시 히트 분리 측정
  5. **처리량**: tokens/sec 베이스라인 대비 (목표 +20%)
- **테스트 필수 항목**: `tests/unit/test_preemptive_scheduler.py` (선점 트리거 + KV 전송 비동기 테스트), `tests/unit/test_eopt_shrinkq_accuracy.py` (BBP 랭크 선택 + perplexity ±1% 포함), `tests/unit/test_static_dynamic_segment.py` (정적/동적 분류 + Multi-hop 무효화 전파), `tests/unit/test_manifoldkv_eviction.py` (유클리드 아웃라이어 스코어 + 64K 컨텍스트), `tests/integration/test_cross_ac_preempt_compress.py` (A+C 통합 파이프라인)
