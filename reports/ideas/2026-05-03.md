## 변화 요약

## Activity A 아이디어

### 아이디어 A-1: DualMap 이중 해시 + 의미 히트율 가중 라우팅
- **가설**: 각 요청을 두 개의 독립 해시 함수로 두 후보 인스턴스에 매핑하되, 후보 선택 기준을 단순 KV 접두사 히트가 아닌 KVShare DHD 스타일의 "의미 유사도 기반 예상 세그먼트 히트율"로 대체하면, 접두사 불일치 요청에서도 캐시 친화성을 유지하면서 부하 균형을 달성할 수 있다.
- **접근 방법**:
  1. `DualMapSemanticRouter` 클래스 (`src/scheduler/dual_map_semantic_router.py`)
  2. 두 독립 해시 함수 h1, h2로 각 요청을 두 후보 노드 (n1, n2)에 매핑
  3. 각 후보 노드에서 "의미 히트율" 계산: 요청 임베딩과 노드 캐시 세그먼트 임베딩 간 코사인 유사도 상위 k개 평균
  4. 라우팅 스코어 = semantic_hit_rate × (1 - load_ratio) + SLO_위반_시_부하_우선_전환 플래그
  5. 핫스팟-인식 재균형: semantic hit rate가 양쪽 후보 모두 임계값 이하면 부하 기준으로만 선택
  6. 경량 임베딩: 세그먼트 임베딩은 평균 Key 벡터 d_head 차원 → 추가 모델 불필요
- **대상 환경**: 멀티 노드
- **멀티 노드 고려사항**: 노드별 세그먼트 임베딩 인덱스를 주기적으로 가십(gossip) 동기화. 의미 유사도 계산 비용은 O(N_segments × d_head) — 노드당 세그먼트 수 제한으로 오버헤드 관리. SLO 위반 감지 시 즉시 부하 우선 전환(DualMap 논문 실증 패턴 그대로 적용).
- **예상 효과**: 처리량 +20~25%, 캐시 히트율 +15~20%p (의미 기반), Scheduling overhead TTFT +4% 이내
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 A-2: Dynamo 1.0 KVBM 이벤트 스트림 기반 압축-상태 인식 라우팅
- **가설**: NVIDIA Dynamo 1.0의 전역 KV 이벤트 스트림(KVBM 방출)에 TurboQuant 압축 상태 메타데이터(비트폭, 레이어별 압축률)를 태그로 추가하면, 스케줄러가 "3비트 압축 KV를 가진 노드 vs 원본 FP16 KV를 가진 노드" 중 요청의 정확도 요구사항에 따라 라우팅 결정을 내릴 수 있다.
- **접근 방법**:
  1. `CompressionStateRouter` 클래스 (`src/scheduler/compression_state_router.py`)
  2. 각 KV 블록 이벤트에 메타데이터 태그 추가: `{block_id, node_id, bitwidth, layer_idx, accuracy_tier}`
  3. accuracy_tier 분류: tier-A (FP16, 정밀도 최우선), tier-B (3비트 TurboQuant, 일반), tier-C (1비트 sign-only, 속도 우선)
  4. 요청 SLA 기반 라우팅: 정밀도 우선 요청 → tier-A 노드, 처리량 우선 요청 → tier-B/C 노드
  5. KVBM 이벤트 수신 인터페이스: 실제 Dynamo 없을 때 시뮬레이션 이벤트 스트림으로 대체
  6. 기존 `MultiNodeScheduler` (2026-04-30)의 라우팅 레이어 위에 압축 상태 필터로 구현 가능
- **대상 환경**: 멀티 노드
- **멀티 노드 고려사항**: 전역 KV 이벤트 스트림의 이벤트 빈도와 라우터 결정 빈도를 분리(비동기 업데이트). 이벤트 지연이 라우팅 결정 품질에 미치는 영향 모델링 필요.
- **예상 효과**: 처리량 +10~15% (tier별 라우팅 최적화), 정확도-처리량 트레이드오프 명시적 제어 가능
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 A-3: TraCT CXL 기반 NIC-없는 KV 전송 스케줄러
- **가설**: CXL 공유 메모리를 KV 전송 기판으로 활용하면 NIC 홉을 완전 제거할 수 있으며, 이를 전제로 "CXL 접근 가능 범위 내 KV 우선 재사용" 정책을 스케줄러에 통합하면 RDMA/DRAM 기반 KV 전달 대비 TTFT를 최대 9.8× 줄일 수 있다.
- **접근 방법**:
  1. `CXLAwareScheduler` 클래스 (`src/scheduler/cxl_aware_scheduler.py`)
  2. 노드 토폴로지 모델: CXL 접근 가능 노드 집합(rack-local)과 RDMA 접근 노드 집합(remote)을 분리
  3. KV 배치 정책: 빈번히 공유되는 세그먼트는 CXL 공유 메모리에 우선 배치 (랙 내 모든 노드 즉시 접근)
  4. 라우팅 우선순위: CXL 히트 > HBM 히트 > RDMA 히트 순으로 노드 선택
  5. 시뮬레이션: CXL latency (100~300ns) / RDMA latency (~1ms) / HBM (0) 파라미터로 cost 모델 구성
  6. `BaseScheduler` 상속, `get_kv_transfer_cost(src, dst, protocol)` 추상 메서드 구현
- **대상 환경**: 멀티 노드 (랙 스케일)
- **멀티 노드 고려사항**: CXL 공유 메모리 용량 제약(랙 단위 풀) → 중요 세그먼트 선택 정책 필요. CXL 풀이 없는 환경에서는 RDMA 모드로 자동 전환(fallback).
- **예상 효과**: 처리량 +15~20%, TTFT p50 −20~40% (CXL 히트 시), Scheduling overhead +2% 이내
- **구현 난이도**: medium
- **우선순위**: 3

## Activity B 아이디어

### 아이디어 B-1: KVShare DHD 스타일 의미 유사도 기반 비연속 KV 공유
- **가설**: 정확 토큰 해시 매칭(현재 구현) 대신, Dual-Stage High Deviation(DHD) 알고리즘처럼 프리필 단계에서 세그먼트 간 의미 유사도를 계산하고 편차가 큰 KV만 선택적으로 재계산하면, 접두사가 전혀 일치하지 않더라도 멀티테넌트 환경에서 KV를 공유해 히트율을 60% 이상 향상시킬 수 있다.
- **접근 방법**:
  1. `SemanticSegmentCache` 클래스 (`src/cache/semantic_segment_cache.py`): `CacheStore` 인터페이스 상속
  2. 프리필 단계: 세그먼트 임베딩(평균 K 벡터) 계산 → FAISS(또는 경량 brute-force) 유사도 검색으로 top-k 유사 세그먼트 검색
  3. DHD 기준: 후보 KV와 쿼리 세그먼트 간 head-wise L2 편차 계산 → 편차 > threshold인 head/token만 재계산
  4. 재사용 경로: 편차 낮은 KV 세그먼트 직접 재사용, 편차 높은 부분만 새로 계산 후 혼합
  5. 캐시 저장: 재계산 결과를 새 세그먼트로 캐시에 추가 (기존 세그먼트는 LRU 보존)
  6. 기존 `SegmentedHashCache`와 병렬 운용 가능: 정확 해시 히트 우선, 미스 시 의미 유사도 검색 폴백
- **예상 효과**: 비연속 히트율 +25~35%p (정확 해시 미스지만 의미 유사 세그먼트 존재 시), TTFT −30~50%
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 B-2: FreeKV 투기적 KV 검색 — 크리티컬 패스 외 비동기 세그먼트 검색
- **가설**: 비연속 세그먼트 검색(현재 구현에서 동기적으로 수행)을 FreeKV 투기적 검색 패턴으로 크리티컬 패스 밖으로 이동시키고, CPU-GPU 하이브리드 KV 레이아웃과 이중 버퍼 스트리밍으로 단편화 전송을 제거하면, 비연속 히트율을 유지하면서 TTFT 오버헤드를 +5% 이내로 제어할 수 있다.
- **접근 방법**:
  1. `SpeculativeSegmentFetcher` 클래스 (`src/cache/speculative_fetcher.py`)
  2. 투기적 검색: 이전 배치 처리 중에 다음 배치 요청의 세그먼트 검색을 비동기로 미리 실행
  3. CPU-GPU 하이브리드 레이아웃: 자주 히트되는 세그먼트는 GPU HBM에, 덜 히트되는 세그먼트는 CPU DRAM에 배치. 전송 시 contiguous 버퍼로 재배치하여 단편화 제거
  4. 이중 버퍼(double-buffering): 현재 배치가 GPU에서 처리되는 동안 다음 배치의 KV 세그먼트를 DRAM → HBM 스트리밍
  5. 보정(correction): 투기적 검색이 실제 요청과 불일치 시 세밀한 보정(fine-grained correction) — 오불일치 세그먼트만 동기 재계산
  6. `CompressedSegmentCache`의 `get()` 메서드 위에 투기적 레이어로 추가 (기존 구현 비침습적 확장)
- **예상 효과**: 비연속 세그먼트 검색 오버헤드 −90% (크리티컬 패스 이탈), TTFT 오버헤드 +2~3% 이내, 처리량 +10~15%
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 B-3: StepCache 스타일 출력-레벨 스텝 재사용 (KV 내부 재사용 보완)
- **가설**: 기존 Activity B가 KV 내부(프리필 시점) 세그먼트를 재사용하는 것과 달리, 동일·유사 구조의 출력(코드 생성, JSON, 반복 에이전트 루프)을 스텝 단위로 분할해 가장 매칭되는 캐시 요청을 찾고 실패 구간만 연속 블록 패칭으로 재생성하면, 전체 요청의 79% 이상에서 재사용 전용 패스트 경로를 실현할 수 있다.
- **접근 방법**:
  1. `StepLevelCache` 클래스 (`src/cache/step_level_cache.py`): KV-내부 재사용과 직교하는 출력-레벨 캐시
  2. 스텝 분할: 이전 출력을 고정 스텝(32~128 토큰) 단위로 분할
  3. 태스크-인식 검증기(task-aware verifier): 스텝 콘텐츠를 경량 룰 기반 검증 (JSON 구조 일치, 코드 구문 일치 등)
  4. 최고-매칭 검색: 기존 `SegmentedHashCache`의 청크 해시를 스텝 해시로 재활용 → 유사도 상위 1개 캐시 요청 선택
  5. 연속 블록 패칭: 검증 실패 스텝 구간만 재생성 후 캐시 엔트리 업데이트
  6. KV 내부 재사용(B-1 SemanticSegmentCache)과 스택: KV 히트 → 스텝 재사용 순으로 계층적 재사용
- **예상 효과**: 재사용 전용 패스트 경로 비율 +60~75% (반복 구조 요청 기준), 평균 TTFT −60~80% (패스트 경로 적용 시)
- **구현 난이도**: medium
- **우선순위**: 3

## Activity C 아이디어

### 아이디어 C-1: TurboQuant PolarQuant+QJL 2단계 벡터 양자화 (훈련-무료 3비트)
- **가설**: KV 벡터에 랜덤 회전 행렬(PolarQuant)로 분산을 균일하게 재분배한 후 스칼라 양자화하고, 잔차 오류를 1비트 QJL(Quantized Johnson-Lindenstrauss) 변환으로 보정하는 2단계 파이프라인을 적용하면, 훈련·캘리브레이션 없이 3비트에서 FP16 수준 정확도를 유지하면서 6× 메모리 절감이 가능하다.
- **접근 방법**:
  1. `TurboQuantCodec` 클래스 (`src/cache/turbo_quant_codec.py`): `CompressionCodec` 인터페이스 구현
  2. PolarQuant: `encode_kv(kv, layer_idx)` → 랜덤 회전 행렬 R 적용 (R은 레이어별 고정 시드로 생성) → 3비트 스칼라 양자화
  3. QJL 잔차 보정: 양자화 잔차 = kv_rotated - dequant(quant(kv_rotated)) → 1비트 JL 변환(랜덤 ±1 행렬)으로 부호 벡터 저장
  4. `decode_kv(compressed, layer_idx)` → 3비트 역양자화 + 1비트 QJL 잔차 복원 → 역회전 행렬 R^T 적용
  5. 메모리 레이아웃: 3비트 본체 + 1비트 QJL 잔차 = 실효 4비트 이하 (기존 INT4 동등 저장 공간에서 정확도 우위)
  6. 기존 `CompressedSegmentCache`의 codec을 `TurboQuantCodec`으로 교체(drop-in 대체)
  7. 레이어별 적용: DepthKV 스타일 민감도 프로파일링으로 민감 레이어는 4비트, 나머지는 3비트 적용
- **예상 효과**: 메모리 −60~75% (3비트 기준 6× 절감), accuracy delta ±0.5% 이내 (정보이론 한계 ~2.7배 내)
- **accuracy-preserving 근거**: TurboQuant(ICLR 2026, Google Research/NYU/KAIST)가 Gemma·Mistral에서 3비트 정확도 완전 보존을 실증. PolarQuant의 랜덤 회전이 outlier 분산을 균일화해 스칼라 양자화 오류를 최소화. QJL 잔차 보정이 체계적 오류를 1비트 수준에서 교정. 기존 INT4(SAW-INT4)보다 낮은 비트폭에서도 더 높은 정확도: outlier 처리가 근본적으로 우수. 검증: WikiText-2 perplexity ±1% + HellaSwag + LongBench 3개 서브태스크.
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 구현 타겟)

### 아이디어 C-2: DepthKV + FastKV 2D 차등 압축 (레이어 축 × 단계 축)
- **가설**: 기존 압축 기법이 레이어 차원(DepthKV)이나 단계 차원(FastKV 프리필/디코드 분리) 중 하나만 고려하는 것과 달리, 레이어별 민감도(DepthKV) × 처리 단계(FastKV TSP) 2차원 예산 행렬을 구성하면, 동일 메모리 예산에서 더 정밀한 압축 할당이 가능해 정확도 보존이 강화된다.
- **접근 방법**:
  1. `TwoDimBudgetScheduler` 클래스 (`src/cache/two_dim_budget.py`)
  2. 레이어 축: DepthKV 스타일 민감도 프로파일링 — 초기 N/3 레이어(고민감도), 중간 레이어, 후반 레이어별 예산 비율 결정
  3. 단계 축: FastKV TSP 스타일 — 초기 레이어는 전체 컨텍스트(풀 프리필), 안정화 시점 이후 레이어는 핵심 토큰만 전파(디코드 KV 보존)
  4. 예산 행렬 B[layer_idx][phase] = {budget_ratio, bitwidth, evict_threshold}
  5. 총 메모리 예산 M에서 행렬 합이 M을 만족하도록 탐욕 최적화(greedy allocation)
  6. `TurboQuantCodec`(C-1)과 통합: 비트폭 결정을 예산 행렬에서 조회하여 레이어·단계별 TurboQuant 파라미터 설정
- **예상 효과**: 메모리 −50~65% (균일 압축 대비 추가 −10~15%p), accuracy delta ±0.6% 이내 (LongBench 기준 균일 압축 대비 일관 향상)
- **accuracy-preserving 근거**: DepthKV가 동일 예산에서 LongBench·RULER 균일 압축 일관 초과함을 실증. FastKV가 프리필 1.82×·디코드 2.87× 가속하면서 디코드-전용 가속 방법과 동등 정확도 달성. 2D 할당은 각 축 단독 할당의 교집합이므로 각 축의 accuracy-preserving 보장이 2D에서도 유지. 검증: LongBench + RULER + WikiText-2 perplexity.
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 C-3: CAOTE 어텐션 출력 오류 기반 퇴거 + StructKV 글로벌 in-degree 중심성
- **가설**: CAOTE의 "어텐션 출력에 대한 토큰 기여도(attention score × Value 벡터)" 기준으로 퇴거 후보를 1차 필터링하고, StructKV의 "글로벌 in-degree centrality(레이어 교차 어텐션 패턴 집계)"로 2차 정렬하면, 단순 attention score 기반 퇴거(H2O, SnapKV) 대비 글로벌 정보 허브 토큰을 보존하면서 정확도를 추가로 개선할 수 있다.
- **접근 방법**:
  1. `HybridEvictionPolicy` 클래스 (`src/cache/hybrid_eviction.py`)
  2. CAOTE 기준: `attn_output_contribution(token_i)` = Σ_j [softmax(QK/√d)_ij × V_i] — 닫힌 형태 계산
  3. StructKV in-degree centrality: 레이어 L1 ~ L_final까지 각 토큰이 어텐션 수신하는 횟수 집계 → global_hub_score[token_i]
  4. 결합 퇴거 스코어 = α × attn_output_contribution + (1-α) × global_hub_score (α=0.6 기본값)
  5. Dynamic Pivot Detection(StructKV): 정보이론 메트릭으로 최적 압축 레이어 탐색 → 해당 레이어 이후 퇴거 결정
  6. 기존 `CompressedSegmentCache.evict()` 또는 `CapKVEvictionPolicy` 대체 가능 (독립 policy 클래스)
- **예상 효과**: 메모리 −35~50%, accuracy delta ±0.4% 이내 (기존 attention score 기반 대비 정확도 개선)
- **accuracy-preserving 근거**: CAOTE가 Value 기여도를 포함한 어텐션 출력 오류 최소화로 기존 방법 대비 일관 개선 실증(LLAMA3, QWEN2.5). StructKV가 글로벌 정보 허브 토큰을 보존해 레이어 교차 정보 손실 방지. 두 기준의 결합은 "로컬 출력 오류 최소화(CAOTE) + 글로벌 구조 보존(StructKV)"으로 상호 보완적 — 각 단독 대비 퇴거 결정의 False Negative 감소. WikiText-2 perplexity ±1% + LongBench QA/요약 정확도.
- **구현 난이도**: medium
- **우선순위**: 3

### 아이디어 C-4: FreqKV 주파수 도메인 점진적 압축 (유효 컨텍스트 2× 이상 확장)
- **가설**: KV 상태를 주파수 도메인으로 변환해 에너지가 집중된 저주파 성분만 보존하고 고주파 성분을 필터링하면, 파라미터 추가 없이 최대 32× 컨텍스트 확장이 가능하며, 오래된 컨텍스트의 압축률을 높이고 최신 토큰은 덜 압축하는 점진적 방식으로 perplexity를 안정적으로 유지할 수 있다.
- **접근 방법**:
  1. `FreqDomainCodec` 클래스 (`src/cache/freq_domain_codec.py`): `CompressionCodec` 인터페이스 구현
  2. 변환: `encode_kv(kv, position_age)` → DCT(Discrete Cosine Transform) 적용 → 저주파 계수 k개만 보존 (k = dim × compress_ratio)
  3. 점진적 압축: `position_age` (현재 디코드 위치 - 생성 위치)에 따라 compress_ratio 조정 — age 0~128: ratio=0.9(거의 무압축), 128~512: ratio=0.5, 512+: ratio=0.2
  4. `decode_kv(compressed, position_age)` → IDCT로 근사 복원
  5. `CompressedSegmentCache`에서 세그먼트 나이(age)를 메타데이터로 추적, `FreqDomainCodec`에 전달
  6. TurboQuantCodec(C-1)과 직교 결합 가능: 주파수 압축 후 잔여 계수를 TurboQuant 3비트 양자화
- **예상 효과**: 유효 컨텍스트 길이 4~8× 확장 (목표 2× 초과 달성), 메모리 −60~80% (점진적 압축), perplexity delta ±0.8% 이내
- **accuracy-preserving 근거**: FreqKV가 8K 훈련 후 LLaMA-2-7B의 컨텍스트 윈도우를 256K로 확장하면서 perplexity 안정 유지를 실증. 에너지 보존 원리: KV의 저주파 성분에 의미 정보가 집중되어 고주파 필터링 시 perplexity 변화 최소. 점진적 압축: 최신 토큰(attention에 더 많이 참조)은 고품질 유지, 오래된 토큰(참조 빈도 낮음)을 공격적 압축 — 어텐션 패턴과 일치. WikiText-2 perplexity + RULER long-context 벤치마크(128K 이상) + LLaMA-2-7B, Llama-3 검증.
- **구현 난이도**: low
- **우선순위**: 4

## 크로스 Activity 시너지 아이디어

### 아이디어 Cross-1: A+B — DualMap 의미 히트율 라우팅 + FreeKV 투기적 검색 (비동기 파이프라인 통합)
- **조합 근거**: DualMapSemanticRouter(A-1)가 의미 유사도 기반으로 최적 노드에 요청을 배분하고, SpeculativeSegmentFetcher(B-2)가 해당 노드에서 세그먼트 검색을 비동기로 크리티컬 패스 밖에서 완료한다. 두 기법을 파이프라인으로 결합하면 라우팅 결정 → 비동기 세그먼트 프리패칭 → 어텐션 계산의 세 단계가 중첩(overlap)되어 TTFT 오버헤드를 최소화한다.
- **구체 설계**:
  1. 라우팅 결정 시점에 `SpeculativeSegmentFetcher.prefetch_async(request, target_node)` 호출
  2. 배치 구성 완료 시점에 prefetch 결과를 동기화(최대 5ms 대기 후 miss 처리)
  3. DualMap 라우팅 스코어에 "prefetch 예상 완료 시간" 항목 추가 — prefetch가 빠를 노드를 우선 선택
  4. 멀티노드: CPU-GPU 이중 버퍼를 노드 간 RDMA 버퍼로 확장
- **예상 복합 효과**: 처리량 +25~30%, 비연속 히트율 +20~25%p, TTFT 오버헤드 +2% 이내
- **구현 난이도**: medium
- **우선순위**: 2

### 아이디어 Cross-2: B+C — SemanticSegmentCache + TurboQuant 3비트 (의미 히트 세그먼트 압축 저장)
- **조합 근거**: SemanticSegmentCache(B-1)의 의미 유사도 기반 KV 공유는 더 많은 세그먼트를 캐시에 저장할수록 히트율이 높아진다. TurboQuantCodec(C-1)으로 세그먼트를 3비트로 압축하면 동일 메모리에 6× 더 많은 세그먼트를 보존할 수 있어 의미 히트율을 추가로 향상시킨다. 또한 DHD 재계산 시 TurboQuant 압축 KV를 출발점으로 사용해 재계산 오버헤드를 줄인다.
- **구체 설계**:
  1. `SemanticSegmentCache.put(kv, embedding)` → `TurboQuantCodec.encode(kv)` → 3비트 압축 세그먼트 저장 + 임베딩 인덱스 갱신
  2. `SemanticSegmentCache.get(query_embedding)` → 유사 세그먼트 검색 → `TurboQuantCodec.decode(compressed)` → DHD 편차 체크 → 재계산 또는 직접 사용
  3. 메모리 계층: TurboQuant 3비트 세그먼트 → GPU HBM(핫 세그먼트), CPU DRAM(콜드 세그먼트)
  4. 재계산 예산: 전체 세그먼트의 20% 이하만 재계산 (DHD 편차 임계값으로 조정)
- **예상 복합 효과**: 처리량 +20~30%, 메모리 −65~75%, 비연속 히트율 +30~40%p, 정확도 ±0.5% 이내
- **구현 난이도**: medium
- **우선순위**: 1 (이번 사이클 최우선 타겟)

### 아이디어 Cross-3: A+C — Dynamo KVBM 이벤트 스트림 + DepthKV 레이어별 예산 공동 최적화
- **조합 근거**: CompressionStateRouter(A-2)가 전역 KV 이벤트 스트림에서 압축 상태를 추적하고, DepthKV 스타일 레이어별 민감도 정보를 이벤트 태그에 포함시키면, 스케줄러가 "레이어별 압축률이 높아지는 시점"을 예측해 선제적 KV 전송·퇴거를 결정할 수 있다. 즉, 레이어 민감도 정보가 스케줄링 신호로 작동하는 새로운 A+C 폐루프를 형성한다.
- **구체 설계**:
  1. KVBM 이벤트에 `layer_sensitivity_score`, `current_bitwidth`, `eviction_pressure` 필드 추가
  2. `CompressionStateRouter`가 이벤트 스트림을 구독 → 압축 압력(eviction_pressure)이 임계값 초과 노드 회피
  3. TwoDimBudgetScheduler(C-2)가 레이어 민감도를 이벤트로 방출 → CompressionStateRouter가 수신하여 라우팅 조정
  4. 폐루프: 라우팅 결정 → KV 배치 → 레이어 압축 예산 소비 → 이벤트 방출 → 다음 라우팅 결정 반영
- **예상 복합 효과**: 처리량 +15~20%, 메모리 −55~65%, 정확도 ±0.6% 이내, 라우팅 오버헤드 +3% 이내
- **구현 난이도**: high
- **우선순위**: 3

### 아이디어 Cross-4: A+B+C — DualMap 의미 라우팅 + DHD KV 공유 + TurboQuant 3비트 완전 통합
- **조합 근거**: SemanticSegmentCache(B-1)의 의미 기반 비연속 KV 공유 + TurboQuantCodec(C-1)의 3비트 압축 저장 + DualMapSemanticRouter(A-1)의 의미 히트율 가중 이중 해시 라우팅을 완전 통합한다. KVShare 논문의 "캐시-인식 스케줄러가 KV 히트율 기반으로 요청 우선순위를 결정하여 연속 배칭을 오케스트레이션"하는 구조를 A+B+C 전체 파이프라인으로 구현.
- **구체 설계**:
  1. 요청 수신 → DualMapSemanticRouter가 두 후보 노드에서 의미 히트율 계산 (B-1 세그먼트 임베딩 인덱스 조회)
  2. 최적 노드 선택 → SpeculativeSegmentFetcher가 해당 노드에서 비동기 프리패칭 시작 (B-2)
  3. 세그먼트 캐시 조회 → SemanticSegmentCache DHD 편차 체크 → 재계산 필요 토큰만 계산
  4. 신규 세그먼트 저장 → TurboQuantCodec 3비트 압축 → 임베딩 인덱스 갱신
  5. 배치 구성 → 캐시-인식 연속 배칭(KVShare 스타일: 히트율 높은 요청 우선)
  6. KVBM 이벤트 방출 → 다른 노드의 CompressionStateRouter에 압축 상태 전파
- **예상 복합 효과**: 처리량 +30~40%, 메모리 −65~75%, 비연속 히트율 +30~40%p(전체 히트의 50%+ 목표), 정확도 ±0.5% 이내
- **구현 난이도**: high
- **우선순위**: 2 (중기 통합 타겟)

## 이번 사이클 최우선 구현 타겟

**Cross-2(B+C): SemanticSegmentCache(B-1) + TurboQuantCodec(C-1) + TwoDimBudgetScheduler(C-2)**

구현 순서:
1. **C-1** `TurboQuantCodec`: PolarQuant 랜덤 회전 + 3비트 스칼라 양자화 + QJL 잔차 1비트 보정. `CompressionCodec` 인터페이스 drop-in 교체. (독립 모듈, 기존 codec 인터페이스 호환)
2. **B-1** `SemanticSegmentCache`: 세그먼트 임베딩(평균 K 벡터) + FAISS/brute-force 유사도 검색 + DHD 편차 기반 선택적 재계산. TurboQuantCodec으로 압축 저장.
3. **C-2** `TwoDimBudgetScheduler`: DepthKV 레이어 민감도 프로파일링 + FastKV TSP 단계 분리 → 2D 예산 행렬. TurboQuantCodec 비트폭 파라미터 주입.
4. **Cross-2 통합**: SemanticSegmentCache와 TurboQuantCodec 파이프라인 연결. DHD 재계산 시 3비트 압축 KV에서 출발하는 경로 추가.

이유:
- Cross-2는 이전 사이클(2026-05-02)의 B+C(RelayCaching + CapKV + Sign VQ)와 모든 알고리즘·자료구조가 다르다(의미 유사도 검색 vs 해시 매칭, PolarQuant+QJL vs 레버리지 스코어, 2D 예산 행렬 vs 단일 퇴거 정책).
- TurboQuantCodec이 메모리 목표 −30%를 6× 초과 달성하며, 더 많은 세그먼트 보존 → 의미 히트율 향상의 연쇄 효과.
- SemanticSegmentCache DHD가 이전 사이클의 정확 해시 매칭 한계를 극복해 비연속 히트율 30% 이상 목표를 안정적으로 달성.

## planner에게 전달할 요청사항

- **Activity B-1(SemanticSegmentCache) 구현 시**: `src/cache/semantic_segment_cache.py` 신규 생성. 세그먼트 임베딩은 평균 Key 벡터(추가 모델 불필요). FAISS 의존성이 부담스러우면 brute-force cosine 유사도(N_segments ≤ 10K 가정)로 대체 가능. DHD 편차 임계값(deviation_threshold)을 설정 파라미터로 노출.
- **Activity C-1(TurboQuantCodec) 구현 시**: `src/cache/turbo_quant_codec.py` 신규 생성. 랜덤 회전 행렬은 레이어별 고정 시드로 생성(재현성 보장). **accuracy-preserving 검증 필수**: WikiText-2 perplexity ±1% + HellaSwag 정확도 + LongBench 서브태스크 3개. 기존 `CompressionCodec` 인터페이스 drop-in 교체로 기존 `CompressedSegmentCache` 테스트 재활용.
- **Activity C-2(TwoDimBudgetScheduler) 구현 시**: `src/cache/two_dim_budget.py` 신규 생성. 레이어 민감도 프로파일링은 소수 배치(≥10 요청) 오프라인 실행. 예산 행렬 시각화 기능 포함(디버깅 용이).
- **Activity A-1(DualMapSemanticRouter) 구현 시**: `src/scheduler/dual_map_semantic_router.py` 신규 생성. 단일노드 시뮬레이션 가능(노드 수 파라미터 = 1로 설정). SLO 위반 임계값(slo_ttft_ms) 설정 파라미터 필수.
- **Cross-2 통합 시**: 의미 기반 히트(semantic hit)와 정확 해시 히트(exact hit)를 분리 측정. DHD 재계산 비율(recompute_ratio)을 별도 메트릭으로 추적.
- **측정 지표 (이번 사이클)**: (1) 비연속 히트율 — 의미 히트 vs 정확 히트 분리, (2) KV 메모리 감소율 — TurboQuant 3비트 기준, (3) Compression Accuracy Delta — WikiText-2 perplexity + LongBench, (4) 처리량 — tokens/sec 베이스라인 대비, (5) TTFT p50/p99 — 의미 검색 오버헤드 포함.
- **테스트 필수 항목**: `tests/unit/test_turbo_quant_accuracy.py`, `tests/unit/test_semantic_segment_cache.py`, `tests/integration/test_cross_bc_pipeline.py`
